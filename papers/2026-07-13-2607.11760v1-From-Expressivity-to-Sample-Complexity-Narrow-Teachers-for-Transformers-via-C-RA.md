# From Expressivity to Sample Complexity: Narrow Teachers for Transformers via C-RASP

**Authors:** Michael Rizvi-Martel, Satwik Bhattamishra, Guillaume Rabusseau, Michael Hahn
**Published:** 2026-07-13 | **arXiv:** [2607.11760v1](http://arxiv.org/abs/2607.11760v1)

## Summary
This paper bridges a gap in transformer theory: while prior work showed *what* tasks transformers can express, this work tackles *how efficiently* they learn those tasks. The authors propose sample complexity bounds for learning C-RASP constructions (a formal language for describing transformer computations), moving from expressivity questions toward practical learnability guarantees.

## Theoretical Background
Transformers are neural networks using attention mechanisms to process sequences. Prior work (e.g., [[Attention Is All You Need]]) established that transformers can express various algorithms, but mostly through hand-crafted weight constructions — not by studying whether gradient descent can *find* them. C-RASP (Causal Restricted Access Sequential Processing) is a programming language designed to describe what transformers can compute; it sits between informal intuitions and formal complexity theory. This work applies loss landscape analysis — a technique examining the geometry of optimization surfaces — to understand how sample complexity (the number of training examples needed) scales with task complexity in transformers.

## Core Contributions
1. **First sample complexity bounds for C-RASP**: Quantifies how many examples are needed to learn constructions expressible in C-RASP, moving beyond prior expressivity-only results.
2. **Connection between expressivity and learnability**: Formalizes the gap between "can express" and "can learn," showing that expressivity alone doesn't guarantee efficient learning.
3. **Loss landscape perspective**: Uses loss landscape analysis to derive preliminary bounds, suggesting that geometric properties of the optimization surface constrain sample complexity.

## Connections to Other Work
Relates to broader theoretical frameworks for understanding [[Transformers]]: [[mechanistic-interpretability]] work tries to reverse-engineer trained models; this paper addresses the forward problem (learning). Complements [[RASP]] (the non-causal predecessor to C-RASP) and prior work on transformer expressivity (e.g., work showing transformers can simulate Turing machines). Similar in spirit to recent work on implicit bias and loss landscape geometry in deep learning, but specialized to the attention-based setting.

## Practical Applications
- **Curriculum learning design**: Understanding sample complexity guides how to structure training data for complex transformer tasks.
- **Model scaling laws**: Informs predictions of how many examples language models need for new tasks.
- **Interpretability tools**: Knowing what samples are "necessary" helps identify which training examples contributed most to learning algorithmic behavior.
- **Architecture search**: Sample complexity bounds could guide decisions about model depth/width for target tasks.

## Product & Startup Ideas
1. **Synthetic data generation service**: Use sample complexity theory to optimize synthetic data for fine-tuning transformers on specialized tasks (law, medicine, code).
2. **Model efficiency platform**: Predict how many examples a custom transformer needs for a task before expensive training; optimize data collection budgets.
3. **Interpretability debugger**: For practitioners, identify which training examples or algorithmic steps are "hard to learn," highlighting potential model failures.
4. **Curriculum learning engine**: Automatically structure training data by predicted learnability; apply to language model instruction tuning.
5. **Cost optimizer for ML ops**: Estimate minimum dataset size needed for task-specific transformers; reduce data labeling costs by 30–50%.

## Technical Limitations
- **Preliminary bounds**: Authors note these are "preliminary" — likely loose or specific to simplified settings; gap between theory and practice remains large.
- **C-RASP idealization**: Real transformers don't directly implement C-RASP programs; the mapping between learned weights and formal constructions is fuzzy.
- **Loss landscape assumptions**: Bounds may assume benign loss geometry (e.g., no spurious minima) that may not hold in practice for realistic tasks.
- **Scalability of analysis**: Deriving bounds requires explicit construction; unclear how to extend to more complex, real-world algorithms.
- **Training dynamics gap**: Bounds on sample complexity don't explain *which* algorithmic solution gradient descent converges to or *how long* it takes.

## Open Questions
1. How tight are the bounds empirically? Do actual training curves match the theoretical predictions?
2. Can bounds be extended to non-causal RASP or to more expressive formal languages?
3. How do architectural choices (depth, width, attention heads) affect sample complexity constants?
4. Can this framework predict which real NLP tasks require more data to learn than others?
5. Do the bounds generalize to learning *multiple* C-RASP constructions sequentially (e.g., in multi-task settings)?

## Key Takeaway
Transformers need provably bounded data to learn formal algorithms; sample complexity now connects expressivity to learnability theory.
