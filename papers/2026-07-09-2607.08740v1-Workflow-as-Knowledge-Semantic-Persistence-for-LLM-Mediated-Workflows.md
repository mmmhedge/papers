# Workflow as Knowledge: Semantic Persistence for LLM-Mediated Workflows

**Authors:** Emanuele Quinto, Carlo Andrea Rozzi, Francesco Zanitti
**Published:** 2026-07-09 | **arXiv:** [2607.08740v1](http://arxiv.org/abs/2607.08740v1)

## Summary
This paper proposes a conceptual framework for representing LLM-mediated workflows as persistent, inspectable knowledge objects rather than ephemeral execution traces. The key insight is treating workflow definitions, instances, inference records, and dependencies as first-class entities in a shared knowledge substrate—enabling workflows to be resumed, reviewed, and reasoned about as structured data. This matters because current workflow systems treat LLM calls as black boxes; making workflows themselves transparent and persistent enables better debugging, auditability, and human oversight of LLM applications.

## Theoretical Background
The paper draws on several conceptual traditions. It adopts Lisp-inspired metaphors—symbolic forms and live-image thinking—as explanatory tools for reasoning about computation and state. The distinction between *derive* (deterministic computation) and *infer* (LLM-mediated judgment under declared constraints) echoes formal logic systems where different inference modes have different semantics. The emphasis on persistent knowledge objects parallels ideas from knowledge representation and semantic web traditions, where facts and rules are treated as reusable, queryable entities rather than buried in code execution. The paper positions itself against purely imperative workflow engines (like common DAG schedulers) by proposing that workflows themselves become data.

## Core Contributions
1. **Semantic distinction between derive and infer**: A formal conceptual separation (not yet formalized in transition semantics) that clarifies when computation is deterministic versus when it requires LLM judgment with declared context and capability policies.
2. **Workflow-as-knowledge model**: A framework where workflow definitions, instances, inference records, context snapshots, and dependency relations are all represented as persistent, inspectable objects in a shared knowledge substrate—not just execution logs.
3. **Language-independent conceptual account**: Rather than proposing a specific implementation, the paper offers an explanatory model using symbolic computation and object identity that could apply across multiple workflow systems and programming languages.
4. **Preliminary semantic persistence framework**: The beginnings of a theory (admittedly incomplete) in which workflows are resumable, reviewable, and embeddable within larger knowledge systems.

## Connections to Other Work
This work sits at the intersection of [[agents]] (autonomous systems using tools and reasoning), [[workflow orchestration]] (systems like Airflow, Temporal), and [[knowledge representation]] systems. It implicitly responds to limitations in current tool-use frameworks for LLMs—where tool calls and reasoning traces are often lost or opaque. The emphasis on capability policies and executor-controlled constraints relates to [[alignment]] concerns (controlling what an LLM agent can do). The symbolic-form / object-identity framing echoes older AI traditions (frames, description logics) but applies them to a new problem: making LLM application state legible and persistent.

## Practical Applications
1. **Auditable AI workflows in regulated domains**: Healthcare, finance, and legal firms can log and inspect every step of an LLM-assisted decision, proving compliance and enabling human review before deployment.
2. **Debuggable multi-step LLM pipelines**: Instead of retrying an entire chain-of-thought, developers can inspect intermediate inference records, identify where an LLM went wrong, and resume from that checkpoint.
3. **Knowledge base construction from workflows**: Running the same workflow many times yields a structured knowledge graph of inference decisions, patterns, and edge cases—enabling continuous learning.
4. **Resumable long-running tasks**: A workflow paused mid-execution (e.g., waiting for human approval) becomes a queryable, inspectable object; context can be efficiently recovered and decisions audited before resumption.
5. **Collaborative LLM applications**: Multiple agents or humans can inspect a shared workflow knowledge substrate, reducing miscommunication and enabling transparent delegation.

## Product & Startup Ideas
1. **LLM Workflow Debugger**: A hosted IDE for building and debugging LLM workflows, with persistent checkpointing, step-by-step replay, and a queryable knowledge graph of all inferences—targeting ML engineers and prompt engineers.
2. **Compliance & Audit Layer for LLM Apps**: A middleware that automatically captures and structures inference records, context, and decisions from any LLM workflow, generating audit trails for regulated industries (healthcare, finance, legal).
3. **Knowledge Base Auto-Builder**: A service that runs LLM workflows repeatedly on a corpus, persists all inference decisions, and builds a queryable semantic knowledge base that improves with each run—useful for research, documentation, or domain expertise extraction.
4. **LLM Agent Orchestration Platform**: A SaaS workflow engine specifically designed for multi-LLM, multi-agent systems with built-in resumability, capability policies, and transparent inter-agent communication logs.
5. **Persistent Agentic Memory System**: A database that treats agent state, decisions, and learned patterns as persistent, queryable objects—enabling agents to reason over their own history and improve over time.

## Technical Limitations
1. **No formal semantics yet**: The paper explicitly defers formal transition semantics to future work; without it, the model is conceptual but unproven in edge cases (e.g., what happens if context changes mid-workflow, or if an LLM hallucinates new dependencies?).
2. **Scalability of persistence unclear**: Storing every inference record, context snapshot, and dependency relation could become prohibitively expensive for high-throughput or long-running workflows; the paper doesn't address storage, indexing, or garbage collection.
3. **Capability policies not fully specified**: The executor-controlled capability policy is mentioned but not concretely defined—it's unclear how to express "what an LLM can do" in a way that's both secure and flexible.
4. **Integration with existing systems**: The paper is language-independent conceptually but gives little guidance on how to retrofit this model onto existing workflow engines, LLM frameworks, or databases.
5. **Ambiguity in "live-image thinking"**: The Lisp metaphor is evocative but vague; it's unclear when this would differ from a traditional immutable-log approach or how it would be implemented.

## Open Questions
1. **Formal semantics**: What are the transition rules for derive and infer? How do they interact? What properties should be preserved (consistency, idempotence)?
2. **Context management**: How should context snapshots be versioned and reconciled if a workflow branches or is resumed after external state changes?
3. **Query language**: What would users ask of the knowledge substrate? ("Show me all inferences that relied on this premise." "Find workflows where the LLM contradicted itself.") What query language makes sense?
4. **Scalability & caching**: How do you efficiently persist and index massive inference records? Can you approximate or cache knowledge objects without losing auditability?
5. **Capability policy formalism**: How should policies be expressed, enforced, and verified? Can you prove an LLM will respect declared constraints?
6. **Comparison with immutable logs**: How does semantic persistence differ operationally from a well-designed audit log or event store? What does the knowledge-object view enable that logs don't?

## Key Takeaway
By treating LLM workflows as persistent, inspectable knowledge objects—not just execution traces—you can build auditable, resumable, debuggable intelligent systems that humans and machines can reason about together.
