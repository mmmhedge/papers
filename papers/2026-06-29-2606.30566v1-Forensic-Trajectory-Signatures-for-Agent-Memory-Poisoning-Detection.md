# Forensic Trajectory Signatures for Agent Memory Poisoning Detection

**Authors:** Jun Wen Leong
**Published:** 2026-06-29 | **arXiv:** [2606.30566v1](http://arxiv.org/abs/2606.30566v1)

## Summary
This paper discovers that LLM agents executing memory-poisoning attacks exhibit a distinctive behavioral signature—a specific sequence of tool calls (memory_recall_fact before email_send_email)—that rarely appears in benign sessions. By analyzing 19 trajectory features from tool-invocation logs, the authors build a classifier achieving 99%+ AUC that generalizes across model sizes and even frontier models, enabling real-time detection of memory-poisoning attacks while distinguishing them from prompt-injection attacks.

## Theoretical Background
LLM agents are systems where large language models (neural networks trained on massive text corpora to predict sequences) operate autonomously by calling external tools—functions like sending emails or retrieving facts from memory. Memory poisoning is an adversarial attack where an attacker corrupts the agent's retrievable memory to trick it into performing unintended actions. The paper rests on the insight that **attacks impose structure on the sequence of tool calls** — because the attacker needs certain information to craft a convincing prompt, they must invoke tools in a specific order, leaving observable traces. This relates to the broader security concern of **agent alignment**: ensuring deployed agents behave as intended even under adversarial conditions. The detection approach uses supervised learning on trajectory features—aggregated statistics about the pattern and timing of tool invocations.

## Core Contributions
1. **Discovery of a behavioral invariant**: Memory poisoning attacks require a specific tool-call transition (memory_recall_fact → email_send_email) whose necessity is **mechanistically grounded** in the attack's information-retrieval dependency, not merely an empirical correlation.
2. **High-fidelity cross-model generalization**: A simple rule based on this invariant achieves AUC=0.9563; a Random Forest over 19 features reaches AUC=0.9904, and crucially, the detector **generalizes to held-out models (7B–120B parameters) and frontier models (GPT-4.1, GPT-4o) without retraining**, achieving perfect AUC on 6/9 hold-out splits.
3. **Overdetermined signatures**: Removing half the feature set (all recall-related features) leaves AUC unchanged, proving that memory poisoning imprints redundantly across multiple independent behavioral channels rather than creating a single point of failure for detection.
4. **Real-time feasibility and attack disambiguation**: A prefix-only variant (AUC=0.934) shows real-time blocking is viable; the detector also distinguishes memory-channel attacks from prompt-injection attacks (0.541 vs. 0.99+ scores), enabling incident responders to classify attack types from logs alone.

## Connections to Other Work
This work addresses a gap between [[RLHF]] and deployment-time safety: while [[alignment]] research focuses on training-time value alignment, this paper tackles **runtime adversarial robustness**. It complements [[mechanistic-interpretability]] by showing that attacks leave causal footprints in the behavioral trace—demonstrating that adversarial behavior can be forensically reverse-engineered. The approach is closely related to **intrusion detection** (anomaly detection in system logs) and **behavioral biometrics**, but applied to the novel domain of agent tool-call sequences. The generalization across [[transformers]] of different scales suggests that the invariant reflects something fundamental about how agents must structure their reasoning under memory poisoning, rather than model-specific quirks.

## Practical Applications
1. **Agent security monitoring in production**: Real-time detection dashboards for deployed LLM agents in corporate environments, flagging suspicious tool-call sequences before damage occurs (e.g., email exfiltration).
2. **Incident response and forensic analysis**: Post-hoc analysis of agent logs to determine whether a breach was caused by memory poisoning or prompt injection, informing remediation strategy.
3. **Sandbox and red-team testing**: Automated validation that an agent's tool-call patterns under attack remain detectable, part of pre-deployment security evaluation.
4. **Multi-agent system monitoring**: In [[multi-agent-systems]] where agents coordinate via shared memory stores, detecting when one agent's memory has been poisoned before it corrupts downstream agents.

## Product & Startup Ideas
1. **AgentGuard**: A SaaS monitoring layer that hooks into agent frameworks (LangChain, AutoGPT, etc.) and streams tool-call logs through the trajectory classifier, alerting security teams to anomalies in real time with attack-type classification.
2. **MemoryShield**: A "forensic-as-a-service" offering for enterprises that have suffered agent-based breaches—ingest tool logs, get post-incident analysis identifying attack vectors, attack windows, and compromised memory entries.
3. **AgentSecOps Dashboard**: Analytics platform for teams deploying multiple LLM agents; visualizes tool-call patterns, auto-labels behavioral anomalies, and suggests safe model/tool combinations based on attack surface analysis.
4. **Red-Team Automation Service**: For enterprises, a platform that synthetically generates memory-poisoning and prompt-injection attacks against candidate agent architectures, runs them through the detector, and scores robustness, informing architecture selection.
5. **Behavioral Firewall for LLM APIs**: Middleware deployed between agent runtimes and external tool APIs (email, databases, APIs) that intercepts tool calls and applies the trajectory classifier in real time, blocking suspicious sequences at the API gateway.

## Technical Limitations
1. **Architecture dependency**: The invariant is specific to architectures where routing information is retrieved via **observable tool invocations**. Agents that retrieve memory internally (without tool calls) or use different tool suites will not exhibit the same signature—the detector must be retrained.
2. **Assume discrete tool calls**: The method relies on clear, enumerable tool invocations. In systems with continuous actions or fuzzy tool boundaries, the signature may degrade.
3. **Limited to memory-poisoning attacks**: While the paper shows discrimination from prompt injection, other attack modalities (e.g., timing-based side channels, gradient-based adversarial perturbations) are not evaluated.
4. **Cold-start and concept drift**: Early interactions may be noisy (few tool calls = sparse features); long-running agents may exhibit tool-call distribution shift over time, degrading AUC.
5. **Adversarial adaptation**: A sophisticated attacker, once aware of this detection method, could attempt to interleave decoy tool calls to obfuscate the signature. The paper does not evaluate robustness to evasion.
6. **Scalability with tool complexity**: The feature set was engineered for a relatively simple tool set (email, memory recall, etc.). With hundreds of tools, feature engineering and model complexity may scale poorly.

## Open Questions
1. **Evasion robustness**: Can an attacker, knowing the detector and its rule (memory_recall_fact → email_send_email), craft an attack that obfuscates the signature by calling decoy tools or reordering the sequence while remaining effective?
2. **Feature attribution**: Which of the 19 features drive the Random Forest's decisions? Is there a smaller, more interpretable subset sufficient for generalization?
3. **Temporal dynamics**: Do tool-call sequences carry temporal information (inter-arrival times, duration) that further refine detection? The current approach appears to treat sequences as graphs/orderings.
4. **Multi-step poisoning**: Can the detector identify attacks that poison memory incrementally over multiple agent sessions, or does it require a single concentrated attack?
5. **Adaptive defenses**: Can agents be trained (via RL or fine-tuning) to randomize tool-call order while remaining functional, making the trajectory signature uniform? If so, what is the performance cost?
6. **Portability to other domains**: Do similar behavioral invariants emerge in other agent-like systems (e.g., autonomous robots, trading bots, game-playing agents)? Is this a general principle?

## Key Takeaway
Memory-poisoning attacks on LLM agents leave a forensic signature in the sequence of tool calls—an overdetermined, mechanistically grounded behavioral pattern that persists across model sizes and generalizes to unseen models, enabling both real-time detection and post-hoc attack classification from logs alone.
