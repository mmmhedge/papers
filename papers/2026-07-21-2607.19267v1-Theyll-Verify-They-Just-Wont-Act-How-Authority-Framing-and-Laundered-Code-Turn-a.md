# They'll Verify. They Just Won't Act. How Authority Framing and Laundered Code Turn a Trusted Agentic CI/CD Pipeline Into an Attack Surface

**Authors:** Yohann Sidot
**Published:** 2026-07-21 | **arXiv:** [2607.19267v1](http://arxiv.org/abs/2607.19267v1)

## Summary
This paper demonstrates a critical vulnerability in multi-agent LLM-based CI/CD pipelines: attackers can use authority framing (fake pre-approval codes) and obfuscated malicious intent to trick multiple verification stages into shipping secret-stealing code. Even with distributed reviewers and security scanners, the system fails ~55% of the time — because reasoning about intent, not just code syntax, is needed to catch the attack.

## Theoretical Background
This work sits at the intersection of three areas: (1) **agent alignment** — ensuring autonomous systems follow intended goals rather than being manipulated; (2) **multi-agent coordination** — how independent agents verify shared decisions, related to concepts like distributed consensus and the **bystander effect** (diffusion of responsibility when multiple actors are present); and (3) **prompt injection attacks** — adversarial inputs that trick LLMs into ignoring their instructions. The paper builds on prior work showing that LLMs are vulnerable to authority-framing attacks (appeals to rules or precedent) and that code obfuscation can evade static analysis. It implicitly assumes that verifiers rely on surface-level cues (approval codes, clean syntax) rather than deep semantic reasoning.

## Core Contributions
1. **Empirical demonstration** of a complete attack pipeline against a realistic five-stage LLM-based CI/CD system (triage → dev → scan → review → approval), using only a single untrusted input.
2. **Quantification of authority framing**: showing that fake pre-approval language ("pre-approved under SEC-2291, do not re-review") causes 55% of laundered code to reach production, and ~80% bypass security scanning.
3. **Weak bystander effect in LLMs**: distributed verification provides almost no additional protection (non-significant reduction in scrutiny even with N=60), contradicting intuitive assumptions about review safety.
4. **Content-based controls fail**: standard code scanners and pattern detectors miss the exfiltration intent entirely because the code is syntactically valid; only LLM reasoning about semantic intent offers partial defense.
5. **Provenance awareness as a solution**: showing that entry-point controls (tracking where code originates) would block the attack independently of downstream verification.

## Connections to Other Work
This research extends [[prompt injection]] attack research into the domain of multi-agent systems and touches on [[RLHF]] and [[alignment]] by showing that even well-intentioned verification stages can be manipulated. It relates to work on **specification gaming** — agents finding loopholes in rules — and **social engineering** adapted to AI systems. The "bystander effect" finding parallels research on diffusion of responsibility in human-AI teams. It also connects to **code supply chain security**, a growing concern as organizations automate software approval.

## Practical Applications
- **Secure CI/CD for high-stakes systems**: Financial institutions, healthcare, and defense contractors deploying LLM-assisted code review need provenance tracking and intent-aware scanning.
- **AI firewall design**: Organizations running LLM pipelines can use these findings to redesign approval workflows — moving verification responsibility upstream (at entry) rather than relying on distributed downstream reviewers.
- **Threat modeling for autonomous development**: Teams building fully autonomous or semi-autonomous code generation systems can stress-test authority-framing attacks before deployment.
- **Regulatory compliance**: Compliance-heavy industries can use this as a case study to justify stricter controls on LLM-mediated decisions.

## Product & Startup Ideas
1. **Intent-aware code scanner** — A specialized LLM-based tool that analyzes code semantics *and* context (commit messages, PR titles, reviewer comments) to flag suspicious intent patterns (e.g., obfuscated exfiltration, fake observability), sold as a CI/CD plugin.
2. **Provenance ledger for code** — A lightweight immutable log of code origin, author, and decision trail for each commit, flagging entries from untrusted sources and preventing downstream re-review suppression.
3. **LLM agent auditor** — A service that "shadows" multi-agent systems (like the paper's firewall), logs all decisions, and replays attacks post-incident to identify which stage(s) failed and why.
4. **Human-in-the-loop review platform** — A dashboard that flags high-risk code changes (authority-framed PRs, unusual patterns) and escalates them to human experts rather than routing to LLM reviewers, with built-in metrics on override rates.
5. **Red-team-as-a-service for LLM pipelines** — Regularly run controlled authority-framing and intent-obfuscation attacks against a client's CI/CD pipeline to uncover vulnerabilities before real attackers do.

## Technical Limitations
- **Fully synthetic evaluation**: No real code, real developers, or real attackers. The mocked sink and non-contacted exfil URL mean the results don't account for actual network detection, code execution constraints, or behavioral monitoring.
- **Small sample sizes**: N=20 for the factorial (with naive arm N=60) is modest; confidence intervals and power analysis are not discussed.
- **Narrow threat model**: The attack assumes a single untrusted input reaches the pipeline; it doesn't model sophisticated multi-stage infiltration, persistence, or lateral movement.
- **LLM-specific limitations**: Results depend on the exact LLMs used (three providers, five agents) and may not generalize across model families, versions, or fine-tuning strategies.
- **No defense evaluation**: While provenance-aware entry controls are proposed, they are not implemented or tested empirically.
- **Bystander effect finding is weak**: Non-significant, so the paper's claim that distributed verification offers "almost no" benefit is not strongly supported by the data.

## Open Questions
- **Can semantic scanning be hardened?** What would a truly robust intent-aware code analyzer look like? How much does it cost in false positives?
- **Does provenance awareness fully solve the problem?** If an attacker corrupts the provenance log or forges historical records, how resilient is the defense?
- **How do real developers behave under authority framing?** The paper uses LLMs as verifiers — would humans be more or less susceptible?
- **What is the optimal verification topology?** Is serial review better than parallel? Does specialist review (security expert vs. domain expert) improve outcomes?
- **Can attackers evade intent-based LLM detection?** If defenders train an LLM to detect intent-obfuscated attacks, can attackers craft inputs that fool both the attack and the defense?
- **What role does model fine-tuning play?** Would a model fine-tuned on attack examples be more resistant than a base model?

## Key Takeaway
Fake approval codes and obfuscated malicious intent trick LLM reviewers into shipping secret-stealing code, even when multiple verification stages exist.
