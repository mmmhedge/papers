# QueenBee Planner: Skill-Evolving Communication Topologies for Token-Efficient LLM Multi-Agent Systems

**Authors:** Congjia Tian, Yuhang Yao, Jiaming Cui
**Published:** 2026-06-25 | **arXiv:** [2606.27492v1](http://arxiv.org/abs/2606.27492v1)

## Summary
QueenBee Planner is a framework that enables LLM-based multi-agent systems to autonomously learn and improve their inter-agent communication topologies—the directed graphs specifying who communicates with whom and when. Rather than fixing communication patterns or using generic generation, the framework learns reusable architectural design skills through execution traces, achieving significant improvements in token efficiency and task performance on aggregation and coordination benchmarks.

## Theoretical Background
Multi-agent systems coordinate through information exchange, and the structure of this communication—the topology—critically affects efficiency and solution quality. Prior work often uses fixed topologies (e.g., hierarchies, rings) or naive generation without learning. This paper builds on:

- **Graph-based communication**: Representing agent interactions as directed acyclic graphs (DAGs) where edges encode message passing.
- **Skill distillation**: Learning generalizable rules from execution traces rather than memorizing individual task solutions.
- **Credit assignment and attribution**: Distinguishing genuinely helpful design choices from spurious correlations or luck—using variance-aware credit and motif-level attribution to isolate causal design patterns.
- **Self-play improvement with verification**: Learning from experience while guarding against overfitting false explanations, using held-out validation gates and falsification checks.

The core insight is treating communication topology design as a learnable, transferable skill—similar to how humans acquire architectural intuition—rather than a task-specific hyperparameter.

## Core Contributions

1. **Skill-evolving planner architecture**: An outer LLM planner that generates temporal communication DAGs while worker agents and task logic remain frozen. This decouples topology learning from task performance, making learned topologies potentially transferable.

2. **Evidence-backed design rules with three actions**: Distillation yields interpretable rules (Preserve, Modify, Avoid) grounded in execution traces, not just black-box optimization.

3. **Rigorous safeguards against self-deception**:
   - Held-out acceptance gates: validate rules on held-out tasks before adoption.
   - Variance-aware credit: account for noise and luck in single runs.
   - Motif-level attribution: identify graph substructures that causally drive improvements.
   - Transfer trust: measure whether rules generalize across task variations.
   - Insight falsification: explicitly test whether proposed rules are actually causal.
   - Structural deduplication: eliminate redundant rules.

4. **Empirical validation**: On Count-Frequency and Silo-Bench tasks, self-evolved topologies outperform fixed topologies and cold LLM generation—reducing RMSE from 12.53 to 7.87 while cutting messages, model calls, and token cost.

## Connections to Other Work
This work sits at the intersection of:

- **Multi-agent communication design**: Relates to prior work on optimal communication graphs and emergent protocols, but adds systematic learning and falsification.
- **Prompt engineering and in-context learning**: Similar to [[in-context few-shot learning]], but applied to architectural rather than task-level design.
- **Program synthesis and [[reinforcement-learning]]**: Uses execution traces to refine learned structures, but with explicit causal verification to avoid [[reward hacking]] and spurious correlations.
- **Interpretability in learned systems**: Echoes [[mechanistic-interpretability]] in demanding evidence-backed explanations rather than accepting plausible-sounding rules.

Differs from naive topology search (which has exponential complexity) and from fixed hierarchies (which ignore task structure). Closest cousins are [[curriculum learning]] and [[meta-learning]], though focused on architectural rather than parameter adaptation.

## Practical Applications

1. **Enterprise knowledge coordination**: Large organizations route inquiries through teams. QueenBee can learn optimal routing topologies (who briefs whom, in what order) for different problem classes, reducing redundant meetings and accelerating resolution.

2. **Research collaboration networks**: Academic or R&D teams can optimize who communicates with whom at each stage of a project—design phase, execution, review—improving cycle time and preventing bottlenecks.

3. **Distributed data analytics**: In federated learning or privacy-preserving analytics, learned topologies specify which data silos exchange insights with which aggregators, reducing communication overhead and latency.

4. **Customer support systems**: Route customer inquiries through specialized agent pools (billing, technical, account) by learning which agent combinations resolve issues fastest, maximizing first-contact resolution.

5. **Supply chain coordination**: Dynamically optimize communication between suppliers, warehouses, and distribution centers based on demand patterns and bottleneck history.

## Product & Startup Ideas

1. **AgentTopology.ai**: A SaaS platform where enterprises upload their current multi-agent workflows (chatbot networks, internal knowledge systems) and QueenBee automatically optimizes communication patterns, billing by tokens saved and latency reduced.

2. **SkillVault for LLM Ops**: A marketplace where teams publish learned communication topologies (e.g., "optimal routing for customer support with 12 agents," "efficient aggregation for distributed sensors") as reusable, versioned design skills. Enterprises license and adapt them.

3. **Adaptive Swarms**: Embedded team-coordination software for remote-first companies. Learns and evolves communication topologies for Slack/Teams-integrated AI agents that help teams self-organize—predicting who should speak to whom and when.

4. **FederatedLearning.Pro**: Specializes in optimizing communication in federated ML pipelines. QueenBee learns which data nodes aggregate with which, minimizing bandwidth, rounds, and model calls while maintaining privacy.

5. **MultiAgent Debugger**: Developer tool that visualizes and optimizes communication topologies in multi-agent codebases. Shows bottlenecks, learns improvements from execution logs, and suggests refactors with confidence scores.

## Technical Limitations

1. **Scalability of DAG learning**: Generating temporal DAGs scales combinatorially with agent count and rounds. No explicit complexity analysis or scaling study is provided; unclear how the method performs with >50 agents or deep temporal horizons.

2. **Transferability assumptions**: The paper claims topologies are "reusable" but only tests within narrow task families (Count-Frequency, Silo-Bench). Whether a learned topology for one task class transfers to substantially different problems is open.

3. **Falsification rigor**: While the paper proposes insight falsification, the actual implementation and threshold for "passing" falsification are underspecified. A false positive rate analysis would strengthen claims.

4. **Cold-start and burn-in**: No discussion of how many executions/samples are needed before the planner generates competitive topologies. If burn-in is large, real-world deployment may be expensive.

5. **Agent heterogeneity**: Workers are assumed interchangeable (frozen, identical). Real teams have specialists. Unclear how the method handles agents with different capabilities or reliability.

6. **Non-LLM coordination**: Relies on an outer LLM planner to generate DAGs. For time-critical applications, LLM latency and cost may dominate any message-passing savings.

## Open Questions

1. **Minimum population for generalization**: What is the smallest set of task instances needed to learn rules that reliably transfer to unseen tasks in the same family? How does this scale with agent count?

2. **Motif semantics**: Which graph motifs (e.g., fan-in, staged aggregation, hierarchy) emerge as "design primitives" across domains? Can we build a interpretable grammar of reusable topological patterns?

3. **Adversarial robustness**: Do learned topologies degrade gracefully if an agent becomes slow or unreliable? How fragile are they to agent failures?

4. **Online adaptation**: Can the planner refine topologies in real-time as it executes tasks, or only offline after batch collection? What is the latency/quality tradeoff?

5. **Comparison to gradient-based search**: How does evidence-backed rule distillation compare to direct topology optimization (e.g., differentiable architecture search or evolutionary algorithms)? Is interpretability worth the performance gap?

6. **Heterogeneous agents and skills**: Can the framework extend to teams where agents have different capabilities? Would learned rules specify both topology and agent assignment?

## Key Takeaway
QueenBee demonstrates that multi-agent systems can learn reusable, interpretable communication architectures through careful safeguards against false explanations—trading some generality in topology search for credible, transferable design knowledge that reduces both token cost and latency.
