# CodeRescue: Budget-Calibrated Recovery Routing for Coding Agents

**Authors:** Qijia He, Jiayi Cheng, Chenqian Le, Rui Wang, Xunmei Liu et al.
**Published:** 2026-07-21 | **arXiv:** [2607.19338v1](http://arxiv.org/abs/2607.19338v1)

## Summary
CodeRescue solves a key problem in AI coding agents: after a failed attempt, should the system try cheaper recovery or escalate to a stronger (more expensive) model? The paper trains a smart router that learns when recovery is worthwhile versus when escalation is better, and adds a budget-calibration layer so the same router works under different cost constraints without retraining. This matters because execution feedback in coding makes cheap recovery often viable—throwing away that signal by always escalating wastes resources.

## Theoretical Background
This work builds on two foundations. First, cost-aware model cascading (e.g., try a small model first, escalate if it fails) has become standard in large-language-model deployment. Second, the paper leverages [[Conformal Risk Control]] (CRC)—a statistical framework that adjusts decision thresholds at deployment time to guarantee cost bounds without retraining, using only exchangeability assumptions on data. In coding, execution feedback is unique: a failed run produces a detailed error trace that a cheap model can often learn from, creating a recovery signal that pure classification-based cascading ignores.

## Core Contributions
1. **Recovery routing formulation**: Frames post-failure decisions not as binary (escalate or fail) but as choosing among heterogeneous actions—retry with cheaper model, escalate, use different strategy—using supervised learning on execution rollouts.
2. **CRC deployment layer**: Introduces Conformal Risk Control to dynamically adjust cost penalties at inference time, enabling a single trained router to operate under arbitrary budgets (e.g., 35% or 50% of escalation cost) without retraining.
3. **Empirical evidence of complementarity**: Shows across five coding benchmarks that cheap recovery and escalation succeed on different failure types—validating that a router beats fixed policies.
4. **Practical frontier**: Demonstrates a calibrated point that matches always-escalate solve rates while using 35% of its mean cost, quantifying the efficiency gain.

## Connections to Other Work
This extends [[cost-aware model cascading]] by treating execution feedback seriously—most cascade work ignores or underutilizes error signals. It relates to [[reinforcement learning]] for agent control by framing recovery routing as a sequential decision problem, though the paper uses supervised learning on rollouts rather than RL. The use of [[Conformal Risk Control]] parallels emerging work on post-hoc recalibration for budgeted inference in large-language-models. Also connects to [[test-time compute scaling]]—allocating compute after seeing intermediate failures—and [[chain-of-thought]] reasoning, where feedback can guide recovery strategies.

## Practical Applications
- **LLM coding assistants** (GitHub Copilot, Claude Code, etc.): Route repair attempts intelligently under varying API costs.
- **Enterprise code generation** where per-token billing is measurable: maximize solve rate within a cost budget for bulk code generation tasks.
- **Educational coding platforms**: Offer students cheap retry-with-hints, escalate to teacher intervention only when necessary.
- **Automated program repair**: Route between lightweight transformations and expensive proof-based synthesis based on feedback.
- **Continuous integration pipelines**: Decide whether to retry a failing test with a cheaper fix or escalate to human review, calibrated to pipeline time/cost budgets.

## Product & Startup Ideas
1. **CodeRescue SaaS**: A middleware layer that wraps existing LLM coding APIs, intelligently routes recovery vs. escalation per-request, and exposes a cost-per-solve budget knob to enterprise customers.
2. **Coding agent orchestration platform**: Build a control plane for heterogeneous coding models (GPT, Claude, Llama, local) that learns recovery routing per client and auto-calibrates under their compute budget.
3. **LLM debugging-as-a-service**: Offer cheap "hint generation" and error re-analysis as a recovery tier before escalating to premium expert models; charge per recovery success.
4. **AI pair programmer with transparent cost controls**: IDE plugin that shows users the router's decision (retry/escalate) with predicted cost, letting them approve or override in real time.
5. **Automated code review routing**: Route code review feedback through cheap LLM re-generation, escalate to human reviewers only when the model's recovery confidence is low—reduce review bottlenecks.

## Technical Limitations
- **Supervised router trained offline**: Router is trained on past rollouts; if the execution environment, model behavior, or failure distribution shifts significantly, the router may not generalize well.
- **Exchangeability assumption in CRC**: CRC's guarantees rely on data exchangeability; if future failures differ systematically from training failures (e.g., new code patterns), calibration may not hold.
- **Limited to iterative feedback loops**: Primarily applies to domains with executable feedback (code, queries, structured outputs); less useful for one-shot generation tasks.
- **Evaluation on five benchmarks**: All are coding tasks; unclear how well the approach transfers to other sequential-feedback domains (math, reasoning, retrieval).
- **Computational cost of routing**: Training and running the router itself has overhead; only worth it if recovery/escalation costs are significant relative to routing cost.
- **No exploration of mixed strategies**: Router chooses discrete actions; doesn't explore blended approaches (e.g., partial escalation, parallel attempts).

## Open Questions
- How does router performance degrade with distribution shift in failure types (e.g., deploying to a new programming language or codebase)?
- Can the router learn *what kind of recovery* to attempt (e.g., different prompts, search strategies, decompositions) rather than just escalate/retry?
- What happens if the cost structure changes mid-deployment (e.g., model prices drop or batch size constraints shift)? Can CRC adapt without retraining?
- Does the approach scale to more than two model tiers? Can it route among 3–5 models of varying cost/capability?
- How does execution feedback quality affect routing (e.g., informative vs. noisy error messages)? Is there a minimum feedback richness required?
- Can this be combined with [[reinforcement learning]] to optimize the recovery strategy itself (e.g., which fixes to try first)?

## Key Takeaway
Train a budget-agnostic router to decide whether to retry cheap recovery or escalate—then recalibrate it at deployment time using statistics, not retraining.
