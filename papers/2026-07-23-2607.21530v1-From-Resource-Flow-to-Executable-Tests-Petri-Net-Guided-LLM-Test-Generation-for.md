# From Resource Flow to Executable Tests: Petri-Net-Guided LLM Test Generation for Concurrent Stateful Rust APIs

**Authors:** Kaiwen Zhang, Guanjun Liu
**Published:** 2026-07-23 | **arXiv:** [2607.21530v1](http://arxiv.org/abs/2607.21530v1)

## Summary
This paper combines Petri nets (a formal model for concurrent systems) with large language models to automatically generate rigorous test cases for concurrent Rust APIs. The key insight is that LLMs alone produce shallow or incorrect tests for complex concurrent code, while formal methods alone require too much manual work—so the authors use Petri nets as a structured blueprint to guide LLM test generation, catching bugs that respect both API semantics and concurrency rules.

## Theoretical Background
Testing concurrent, stateful code is hard because the number of possible execution interleavings grows exponentially. Petri nets are a mathematical formalism for modeling concurrent systems: they represent resources as tokens and allowed operations as transitions, capturing both sequential constraints and parallel execution paths. Large language models can generate code from natural language or examples but struggle with formal constraints and complex preconditions. The paper bridges these two worlds—using Petri nets to encode what valid concurrent behavior looks like, then using LLMs to convert those formal scenarios into runnable Rust code.

## Core Contributions
1. A methodology to represent concurrent stateful Rust APIs as colored Petri nets, encoding resource ownership, lifecycle states, and causal dependencies.
2. Techniques to derive legal, near-legal, and partial-order concurrent test scenarios from the Petri net, creating structured intermediate representations.
3. A local-faithfulness contract and structural repair loop that enforce the Petri net's constraints during LLM-based code synthesis, preventing the model from drifting into invalid states.
4. Petri-guided schedule shaping to prioritize high-conflict concurrency patterns for more systematic exploration.
5. A layered semantic oracle to distinguish whether test failures come from synthesis mistakes or genuine API violations.

## Connections to Other Work
This work sits at the intersection of [[formal methods]], [[testing]], and [[generative models]]. It relates to model-based testing (which uses formal specifications to drive test generation) but avoids their manual burden by leveraging LLMs. It extends recent work using LLMs for code synthesis by adding formal constraints; unlike naive LLM test generation, it preserves semantic intent through repair loops. The approach echoes ideas from [[RLHF]] and constrained generation—using external structure to guide neural models toward valid outputs.

## Practical Applications
- **Library development**: Rust library authors can auto-generate comprehensive concurrent test suites for complex APIs (e.g., async runtimes, thread-safe data structures) without hand-writing hundreds of edge cases.
- **Regression testing**: When APIs evolve, re-generate tests automatically while preserving coverage of concurrency scenarios.
- **Safety-critical systems**: For domains like embedded or systems programming where concurrency bugs are costly, generate tests that explore many interleavings systematically.
- **Continuous integration**: Integrate Petri-net-guided test generation into CI/CD pipelines to catch race conditions and state violations early.

## Product & Startup Ideas
1. **Concurrent API Test Generator (SaaS)**: Upload a Rust library + Petri net spec, auto-generate and run a comprehensive test suite in the cloud; report coverage and concurrency bugs.
2. **IDE Plugin**: Embed Petri-net-guided test generation directly in VS Code or JetBrains IDEs; developers right-click an API and generate tests without leaving the editor.
3. **Contract Verification Service**: Combine Petri nets with runtime monitoring to verify that deployed Rust services respect their concurrency contracts; alert on violations.
4. **Rust Library Quality Dashboard**: Analyze popular crates, auto-generate tests via this method, publish test coverage and concurrency-bug reports to help developers choose safe libraries.
5. **Safety Certification Tool**: For regulated industries, provide a turnkey tool to generate, run, and formally certify concurrent test suites for Rust code, producing audit-ready reports.

## Technical Limitations
- **Petri net specification burden**: Someone must manually model the API as a Petri net; this is non-trivial for complex APIs and shifts work rather than eliminating it.
- **Scalability**: As APIs grow in state space and transitions, the Petri net explodes; deriving all legal scenarios may become infeasible.
- **LLM reliability**: Even with Petri-guided constraints, LLMs can still make mistakes during code synthesis; the repair loop mitigates but doesn't eliminate errors.
- **Oracle design**: Determining whether a test failure is a real bug requires a semantic oracle, which is domain-specific and must be carefully designed.
- **Rust-specific**: The method is tailored to Rust's ownership model and async ecosystem; generalization to other concurrent languages (Java, Go, Python) is unclear.
- **Incomplete concurrency exploration**: Even with schedule shaping, exploring all meaningful interleavings is computationally hard; the method is heuristic.

## Open Questions
- How much manual Petri net modeling is truly required for real-world Rust APIs? Can the net be inferred automatically from code?
- Does the method scale to APIs with hundreds of states and transitions? Where does it break?
- How does test quality (bug-detection rate, coverage) compare to human-written tests or other automated methods like fuzzing?
- Can the approach handle dynamically created resources (e.g., thread spawning at runtime) that aren't fixed at spec time?
- How well does the layered semantic oracle generalize across different API domains and failure modes?
- Can this integrate with existing property-based testing frameworks (QuickCheck, Proptest) to combine concurrency scenarios with property checking?

## Key Takeaway
Petri nets guide LLMs to generate rigorous concurrent tests, bridging formal specification and practical code synthesis.
