# GaP: A Graph-as-Policy Multi-Agent Self-Learning Harness For Variational Automation Tasks

**Authors:** Kaiyuan Chen, Shuangyu Xie, Letian Fu, Justin Yu, William Pacini et al.
**Published:** 2026-07-06 | **arXiv:** [2607.05369v1](http://arxiv.org/abs/2607.05369v1)

## Summary
This paper presents GaP (Graph-as-Policy), a system that uses AI agents to automatically design robotic task programs by representing them as computation graphs — networks of perception, planning, and control operations. The key insight is that rather than learning a black-box neural policy or hand-coding rigid procedures, GaP generates interpretable graph structures and iteratively refines them through simulation, achieving high reliability on real-world robotic tasks with geometric variation that typically defeat standard learning approaches.

## Theoretical Background
The work builds on two established robotics paradigms. **Task and Motion Planning (TAMP)** decomposes robot problems into discrete symbolic planning (what sequence of actions?) and continuous motion planning (how to execute smoothly?). **Robot Operating System (ROS)** is a middleware framework that organizes robot software as modular, reusable components. The authors combine these with recent advances in agentic coding systems — LLM-based agents that can write and refine software — to automate the design of task graphs. The core challenge they address is "Variational Automation" (VA): tasks where object shapes, sizes, and poses vary significantly, breaking fixed automation but still needing high reliability for commercial deployment.

## Core Contributions
1. **Graph-as-Policy framework**: A structured representation where robot task solutions are directed acyclic graphs of modular skills (perception→planning→control), rather than monolithic policies or static programs.
2. **Multi-agent code generation loop**: Uses LLM agents to generate graph candidates and iteratively refine them based on simulation success metrics, treating the search space as a code optimization problem rather than pure learning.
3. **Internal simulation harness**: Enables parallel evaluation of candidate graphs on multiple task instances without real-world interaction, dramatically accelerating refinement.
4. **Modular Open Robot Skill Library (MORSL)**: A curated library of reusable perception, planning, and control modules that ground the generated graphs in real capability.
5. **Empirical validation**: 8 new VA benchmarks (4 simulation, 4 real-world) showing substantial improvements over model-free baselines and hand-coded policies.

## Connections to Other Work
GaP sits at the intersection of several research threads:
- **[[Reinforcement Learning]]** vs. programmatic control: Unlike end-to-end model-free RL (which struggles with reliability for rare failure modes), GaP maintains interpretability by generating explicit task graphs.
- **[[TAMP]] and planning-based robotics**: Unlike classical TAMP (which requires hand-engineered domain models), GaP uses agents to discover task decompositions automatically.
- **Agentic AI systems and [[LLM]]-based code generation**: Related to recent work on using LLMs to write and refine software (e.g., code agents for scientific discovery), but applied to embodied robotics rather than pure software.
- **Modular and compositional learning**: Echoes the compositional generalization literature, but with a focus on discovering modular graph structures rather than learning compositional embeddings.

## Practical Applications
1. **Pick-and-place automation**: Handling variable-geometry parts (castings, forgings) without per-variant engineering — directly applicable to electronics assembly, automotive parts handling, logistics.
2. **Bin-picking at scale**: Grasping random objects from bins with shape variation; critical for e-commerce fulfillment centers and recycling facilities.
3. **Deformable object manipulation**: Folding, inserting, or arranging flexible materials (cables, fabrics, food) where geometry varies but success criteria are well-defined.
4. **Collaborative human-robot tasks**: Environments where the robot must adapt to human-placed or partially-visible objects (e.g., assembly line support, handoff scenarios).
5. **Rapid re-tasking in job-shop manufacturing**: Reconfiguring automation for new part variants without weeks of programming and commissioning.

## Product & Startup Ideas
1. **Robotic task compiler service**: A cloud platform where factories upload task videos/specs, GaP automatically generates and tests graph programs, and deploys them to robot fleets. Subscription model based on task complexity or robot hours.
2. **Plug-and-play skill library marketplace**: Monetize MORSL as a curated, versioned registry of robot skills (perception modules, planning algorithms, control primitives) that integrators and manufacturers license and compose.
3. **Simulation-as-a-service for robot automation**: Host industrial robot simulators optimized for parallel graph evaluation; sell compute credits to manufacturers for rapid task prototyping before deployment.
4. **Robotics consulting + GaP licensing**: Partner with system integrators to reduce commissioning time for variable automation; offer the GaP harness as a tool-suite with professional services.
5. **Labor arbitrage through remote task design**: Enable manufacturers in high-wage regions to remotely design tasks (via GaP) executed by cheaper robots in lower-cost facilities, collapsing the geography of automation.

## Technical Limitations
1. **Skill library completeness**: GaP is only as good as MORSL; if a task requires a skill not in the library, the framework cannot synthesize it. Expanding MORSL requires manual engineering and validation.
2. **Simulation-to-reality gap**: Parallel simulation evaluation assumes sim models are faithful. Real-world physics (contact dynamics, friction, wear) may deviate, leading to false positives in simulation.
3. **Graph search complexity**: As MORSL grows, the space of possible graphs explodes combinatorially. No clear guidance on scalability to hundreds or thousands of modules.
4. **Structured task assumption**: Requires tasks to decompose naturally into perception→planning→control pipelines. Open-ended, non-hierarchical tasks may resist this representation.
5. **Agent-driven code generation quality**: Depends on LLM capabilities; generation may favor familiar patterns or fail on novel task structures that don't match training distribution.
6. **Real-world validation scope**: Evaluated on 4 real-world tasks (relatively small). Unclear how well the approach generalizes to diverse industries, sensor modalities, or robot morphologies.

## Open Questions
1. How does GaP scale when MORSL grows beyond the current size? Is there an active learning strategy to identify missing skills?
2. Can the framework handle adversarial or worst-case task instances, or does iterative refinement overfit to the training distribution of variation?
3. What is the lower bound on simulation fidelity required to predict real-world success? Can we quantify sim-to-real transfer reliably?
4. How sensitive is the approach to the choice of LLM and prompting strategy? Could a smaller, cheaper model work, or do you need frontier LLMs?
5. Can GaP handle continuous task evolution — e.g., incrementally refining graphs as new failure modes emerge in deployment, without requiring full re-training?
6. How does performance degrade when the task requires coordination across multiple robots or continuous adaptation (not just a fixed sequence)?

## Key Takeaway
GaP shows that combining agent-based code generation with interpretable graph-structured task programs and parallel simulation can achieve the reliability needed for real-world variable automation without sacrificing adaptability or interpretability.
