# Generative Skill Composition for LLM Agents

**Authors:** Xinyu Zhao, Zhen Tan, Vaishnav Tadiparthi, Nakul Agarwal, Kwonjoon Lee et al.
**Published:** 2026-06-30 | **arXiv:** [2606.32025v1](http://arxiv.org/abs/2606.32025v1)

## Summary
This paper tackles a central problem for LLM-based agents: how to select and order a sequence of reusable skills (modular procedural knowledge packages) to solve complex tasks. The authors propose SkillComposer, a constrained decoder that jointly predicts which skills to use, how many times, and in what order—treating this as a structured sequence prediction problem rather than independent skill retrieval. On real coding tasks, the method outperforms simpler retrieval baselines and nearly matches the performance ceiling of oracle skill selection.

## Theoretical Background
LLM agents operate by orchestrating multiple tools or skills to solve multi-step problems. Prior work either: (1) shows the agent the full skill library and lets it reason over all options, which is expensive and prone to distraction, or (2) uses embedding-based or LLM-based ranking to retrieve a few relevant skills independently. This paper observes that skill composition has *structure*: the choice of skill *i* constrains which skills can follow it (dependencies), and the total number and sequence matter jointly. The paper formalizes this as a constrained sequence modeling problem, akin to decoding under constraints in machine translation or code generation.

## Core Contributions
1. **Formalization of structured skill composition**: Reframes the problem as a joint three-dimensional decision (subset, count, order) rather than independent retrieval steps—a conceptual shift.
2. **SkillComposer architecture**: Uses a constrained autoregressive decoder that predicts skill sequences over discrete skill identifiers, naturally capturing inter-skill dependencies and avoiding the combinatorial explosion of enumerating all subsets.
3. **Real-world training and evaluation**: Builds a training dataset from a human-curated production skill library and evaluates on SkillsBench with two state-of-the-art coding agents (GPT-5.2-Codex, Gemini-3-Pro-Preview), showing +18–23 percentage-point improvements over no-skill baselines.

## Practical Applications
1. **Coding assistants and IDEs**: Automatically compose sequences of refactoring, testing, and debugging skills for multi-file code transformations.
2. **Scientific workflow automation**: Chain analysis, visualization, and reporting skills for data science pipelines.
3. **Enterprise RPA (Robotic Process Automation)**: Orchestrate sequences of business process skills (filing, approval routing, data entry) without hard-coding workflows.
4. **Educational tutoring systems**: Compose pedagogical skills (explain concept → worked example → practice problem → feedback) in optimal order for student goals.
5. **API-heavy service orchestration**: Dynamically chain microservices or third-party APIs based on task requirements, reducing human configuration burden.

## Product & Startup Ideas
1. **Skill Composer as a Service (SCoaS)**: A SaaS platform where enterprises upload their domain-specific skill library (internal tools, APIs, data sources), and the system automatically optimizes skill composition for new tasks. Monetize per-composition or per-agent deployment.
2. **IDE plugin for multi-step code tasks**: Integrate SkillComposer into VS Code or JetBrains to automatically suggest and execute skill sequences for refactoring, testing, and documentation—reducing boilerplate work for developers.
3. **Workflow automation no-code builder**: A low-code platform that lets non-technical users define domain skills (templates), then AI automatically composes them for new workflows, competing with Zapier/Make.
4. **Skill library marketplace**: A community platform where developers publish reusable skills (e.g., "run unit tests," "generate API docs"), and other teams can compose them into agents. Charging for premium skills or curation.
5. **Fine-tuning service for domain-specific agents**: Offer a managed service that trains custom SkillComposer models on clients' skill libraries and internal tasks, handling the data labeling and retraining pipeline.

## Technical Limitations
1. **Skill library dependency**: Performance critically depends on having a well-curated, semantically diverse skill library; the method does not help if skills are poorly designed or redundant.
2. **Training data scarcity**: Requires human-annotated (task, skill-composition) pairs; scaling to new domains or agents requires expensive labeling effort.
3. **Generalization across domains**: The paper evaluates on coding tasks; unclear how well SkillComposer transfers to other domains (e.g., scientific workflows, business processes) where skill dependencies may differ fundamentally.
4. **Skill granularity trade-off**: Too fine-grained skills create long sequences (harder to decode); too coarse-grained skills reduce flexibility. The paper doesn't systematically study this choice.
5. **Error propagation**: If the model predicts a skill that fails, there's no mechanism for recovery or backtracking; later skills may fail due to broken preconditions.
6. **Limited interpretability**: It's unclear *why* the model chose a particular skill sequence; debugging failures or understanding model behavior requires post-hoc analysis.

## Open Questions
1. **Hierarchical composition**: Can SkillComposer be extended to hierarchical skill trees, where high-level goals compose mid-level skills, which in turn compose primitives?
2. **Few-shot adaptation**: Given a small number of examples in a new domain, can the model quickly adapt its composition strategy without full retraining?
3. **Counterfactual reasoning**: How can the agent learn from failed compositions to avoid them in the future? Is there a learning signal from task failure?
4. **Skill discovery**: The paper assumes a fixed skill library. Can the agent or training procedure help *identify* missing skills or propose new compositions not seen in training?
5. **Interactive refinement**: Can a user interactively guide or override skill choices, and does the model learn from these corrections to improve on similar future tasks?
6. **Skill dependency graphs**: Can explicit modeling of skill dependencies (e.g., as a DAG of prerequisites) improve composition quality and reduce errors?

## Key Takeaway
Structured skill composition—jointly predicting subset, count, and order via constrained sequence decoding—is a more effective way to orchestrate LLM agent skills than treating each skill selection independently, delivering substantial practical improvements on real coding tasks.
