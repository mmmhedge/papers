# Bridge Evidence: Static Retrieval Utility Does Not Predict Causal Utility in Multi-Step Agentic Search

**Authors:** Debayan Mukhopadhyay, Utshab Kumar Ghosh, Shubham Chatterjee
**Published:** 2026-07-16 | **arXiv:** [2607.15253v1](http://arxiv.org/abs/2607.15253v1)

## Summary

This paper reveals a fundamental gap between how retrieval systems are currently trained and how they actually perform when integrated into multi-step AI agents. Traditional retrieval systems are evaluated on whether a single document directly answers a question; but when a language model acts as a search agent making multiple queries over time, a document's true value lies in enabling better future searches, not in answering the current question. The authors measure this gap systematically and find that about one-third of documents agents actually use appear useless by traditional metrics—these "bridge documents" matter only for redirecting the agent's search trajectory.

## Theoretical Background

This work sits at the intersection of information retrieval (IR) and agentic AI systems. Traditional retrieval-augmented generation (RAG) evaluates documents in isolation: given a question and a candidate document, does a reader model extract the correct answer? This Static RAG Utility (SRU) framework assumes documents are consumed independently. However, modern language models now act as agents—systems that reason across multiple steps, issue sequential queries, and maintain a trajectory of reasoning (like ReAct-style architectures). The paper contrasts SRU against Counterfactual Trajectory Utility (CTU), which measures a document's causal impact on an agent's entire multi-turn search process by comparing what happens when that document is removed.

## Core Contributions

1. **Empirical quantification of the static-causal gap**: The paper demonstrates that Static RAG Utility and Counterfactual Trajectory Utility are nearly statistically independent (Spearman rho = -0.026) across 23,322 document observations. This is not intuitive—high static relevance should predict causal usefulness, but doesn't.

2. **Discovery and naming of "bridge documents"**: ~33% of documents the agent reads are causally load-bearing yet static-irrelevant. These documents earn their utility by providing discriminative entities that redirect the agent's search trajectory.

3. **Mechanistic explanation**: Using Observable Entity Relevance (OER), the authors show that discriminative entities (entities distinguishing relevant from non-relevant candidates) appear 4.02× more frequently in the agent's next query when they came from bridge documents, explaining *why* these documents matter.

4. **Robustness across metrics**: The pattern holds even when Static RAG Utility is replaced with BM25 + cross-encoder proxies, suggesting the finding is not an artifact of a specific relevance model.

## Connections to Other Work

This paper directly challenges the assumptions underlying [[Retrieval-Augmented Generation (RAG)]] and traditional IR evaluation metrics. It aligns with emerging work on [[agents]] and multi-step reasoning that questions whether standard benchmarks (like HotpotQA) capture real agentic behavior. The work connects to mechanistic understanding of how language models use retrieved information, and to broader concerns in AI safety about optimizing the wrong proxy metric—here, static relevance instead of actual task performance in sequential decision-making.

## Practical Applications

1. **Search engine re-ranking for agentic systems**: Web search providers serving AI agents could optimize ranking not for direct answer quality but for "search trajectory improvement," potentially ordering results that redirect search differently rather than directly answering queries.

2. **Fact-checking and knowledge graph navigation**: For agents reasoning through multi-hop questions or fact verification, retrieval systems could prioritize documents that bridge knowledge gaps and enable the next hop, not documents that directly contradict or verify a single claim.

3. **Research paper discovery for AI scientists**: Tools helping researchers navigate literature could optimize for papers that open new research directions (bridge documents) rather than papers directly answering current queries.

4. **Customer support chatbots**: Routing systems could surface knowledge base articles that guide the agent to ask better follow-up questions, not just documents answering the current customer query.

## Product & Startup Ideas

1. **"AgentRAG" optimization platform**: A tool that profiles and re-ranks retrieval results specifically for multi-turn agentic workflows, measuring trajectory utility instead of static relevance. Sells to companies building AI agent products.

2. **Bridge document detection service**: An API that ingests a document corpus and agent interaction logs, identifies bridge documents, and retrains retrieval models to surface them. Charges per corpus refresh.

3. **Agentic search analytics dashboard**: Monitor which documents are actually *causally* helping your deployed search agent across turns. Identify blind spots where static-good documents fail and causally-useful documents are missed.

4. **Curriculum learning for retrieval fine-tuning**: A training service that uses counterfactual trajectory replay to fine-tune retrieval models on causally relevant examples rather than static relevance judgments. Improves downstream agent performance on reasoning tasks.

5. **Knowledge graph construction from agent logs**: Automatically extract and index the "discriminative entities" that agents use to pivot search direction, building task-aware entity indices optimized for agentic workflows.

## Technical Limitations

1. **Single task and agent architecture**: Experiments use only HotpotQA and ReAct-style agents. The generalization to other reasoning frameworks, tasks, or domains (open-domain QA, math reasoning, code search) is unknown.

2. **Counterfactual assumptions**: CTU relies on replaying trajectories after document deletion. This assumes the rest of the agent's behavior is deterministic or stable—in practice, stochasticity in language models may introduce noise.

3. **Corpus and scale**: Testing on HotpotQA's curated Wikipedia documents may not reflect behavior on noisier, larger web-scale corpora where document quality and redundancy vary dramatically.

4. **No solution provided**: The paper diagnoses the problem but offers no concrete method to optimize for CTU instead of SRU. How to train a retriever on counterfactual utility at scale remains an open challenge.

5. **Entity-centric explanation**: The OER mechanism explanation focuses on entity overlap. Documents may matter for other reasons—rhetorical clarity, argument structure, subtle disambiguation—not captured by entity discrimination alone.

## Open Questions

1. Can we train a retriever end-to-end optimizing for CTU rather than SRU, and does it outperform standard RAG on agentic tasks?

2. How does bridge document utility vary across different types of agents (chain-of-thought vs. ReAct vs. tree search) and reasoning domains (math, code, open-domain QA)?

3. Are bridge documents task-specific or corpus-specific? Would the same document be a bridge for different agents or different questions?

4. Can we predict bridge documents *before* running the full trajectory, or do we always need counterfactual replay?

5. How does document redundancy in larger corpora affect the static-causal gap? Does Wikipedia's structured nature make this problem more severe than on noisier web data?

## Key Takeaway

Documents that help AI agents search better across turns look useless by traditional relevance metrics—agent systems need causal evaluation, not static ranking.
