# Token-Flow Firewall: Semantic Runtime Auditing for Persistent AI Agents

**Authors:** Puji Wang, Yingchen Zhang, Ruqing Zhang, Jiafeng Guo, Xueqi Cheng
**Published:** 2026-07-09 | **arXiv:** [2607.08395v1](http://arxiv.org/abs/2607.08395v1)

## Summary
This paper addresses security risks in persistent AI agents—long-lived systems that maintain memory and interact with tools over time—by proposing TokenWall, a runtime defense framework that monitors natural-language token flows (memory updates, tool arguments, inter-component communications) to intercept unsafe behavior before execution. The work matters because traditional LLM safety approaches assume single-turn interactions, but persistent agents create larger attack surfaces where unsafe content can propagate through reusable skills and stateful components.

## Theoretical Background
The paper builds on two foundational concepts: (1) **LLM safety and alignment**, which traditionally focuses on preventing harmful outputs during inference, typically through training methods like RLHF (reinforcement learning from human feedback) or post-hoc filtering; and (2) **runtime security**, a classical computer security paradigm where monitors intercept program execution at critical boundaries (source-sink analysis) to prevent privilege escalation or unauthorized data flows. The key insight is that persistent agents violate assumptions of both domains—they maintain state across turns and mediate interactions through natural language, creating new semantic attack vectors. The paper frames this as a **semantic flow control problem**: instead of tracking data at the binary/bytecode level (traditional runtime security), TokenWall tracks semantic meaning through token sequences, applying boundary-aware inspection at agent component interfaces.

## Core Contributions
1. **Problem formulation**: Identifies that persistent AI agents create a substantially larger semantic attack surface than single-turn chatbots, with threat vectors through memory, skills, and tool mediation that prior defenses don't address.
2. **TokenWall framework**: A practical runtime firewall that performs full-coverage pre-execution semantic auditing without relying on expensive remote LLM arbitration for every decision, using lightweight local inspection plus selective escalation.
3. **Structured audit records**: Introduces the concept of source-sink audit records that capture semantic flows through agent boundaries (e.g., what data flows from retrieval → memory → tool invocation).
4. **Empirical validation**: Demonstrates 87.5% attack mitigation (12.5% success rate) while maintaining 97.4% benign throughput and adding only 0.69s latency—a concrete security-utility trade-off that is practically deployable.

## Connections to Other Work
This work connects to:
- **[[RLHF]]** and post-training safety methods: TokenWall is complementary—it's a runtime control layer, not a training-time fix; it catches failures that training-based alignment misses.
- **Multi-step reasoning and chain-of-thought**: Persistent agents often use internal reasoning chains stored in memory; TokenWall monitors these as semantic flows.
- **Tool use and function calling**: Prior work on tool-use safety (e.g., verifying tool arguments) is one component of TokenWall's boundary-aware auditing.
- **Classical runtime security** (taint analysis, sandboxing): TokenWall adapts source-sink analysis to the semantic/NLP domain rather than binary programs.
- **Prompt injection and jailbreak defenses**: These are specific threat patterns that TokenWall's broader semantic flow monitoring is designed to catch.

## Practical Applications
1. **Enterprise AI assistants** managing sensitive internal tools (HR systems, financial databases)—TokenWall prevents agents from exfiltrating or corrupting data through tool misuse.
2. **Autonomous task execution systems** (e.g., code-generation agents that can run scripts or modify files)—runtime mediation of tool calls before execution prevents unintended destructive actions.
3. **Multi-turn customer support or compliance agents** that maintain conversation history and customer data—prevents leakage or misuse of persistent state.
4. **Robotics and embodied agent frameworks** where tool-mediated interactions have physical consequences—semantic auditing of action commands before execution.
5. **Research/scientific automation agents** that query databases and generate experiments—ensures that retrieved or inferred data flows safely through subsequent tools.

## Product & Startup Ideas
1. **TokenWall-as-a-Service**: A hosted semantic firewall API that wraps existing agent frameworks (LangChain, AutoGen, etc.), offering drop-in runtime auditing without re-architecting agents; target: enterprises deploying LLM agents on sensitive data.
2. **Agent Security Audit Dashboard**: A monitoring & alerting SaaS that visualizes token flows in production agents, flags ambiguous high-risk cases, and provides forensic replay of security incidents.
3. **Secure Agent SDK**: A developer toolkit bundling TokenWall with safe-by-default agent templates for common use cases (customer support, task automation, research tools), reducing security burden on teams.
4. **Semantic Containment for Model Serving**: Extend TokenWall to multi-agent orchestration platforms, where multiple agents interact—enforce semantic isolation policies between agents at token-flow boundaries.
5. **Compliance & Audit Automation**: A product for regulated industries (financial services, healthcare) that uses TokenWall to auto-generate security audit trails and compliance reports for persistent agent deployments.

## Technical Limitations
1. **Boundary definition**: The framework assumes clear source-sink boundaries in agent architectures; loosely coupled or emergent interaction patterns may be harder to audit systematically.
2. **Semantic richness and evasion**: The lightweight local inspection module's decision logic (what constitutes "risky") may be rule-based or heuristic; an adversary with knowledge of TokenWall's decision criteria could craft semantically benign but harmful inputs ("obfuscated attacks").
3. **Latency trade-offs at scale**: The paper shows 0.69s overhead on benign cases; this may become unacceptable for real-time interactive agents or high-frequency trading/control loops.
4. **Escalation to arbitration**: When TokenWall is uncertain, it delegates to a "stronger arbitration module"—the paper doesn't specify whether this is a larger model, human review, or something else; if it's human-in-the-loop, scalability is limited.
5. **Memory and persistence**: The framework assumes token flows are the primary attack vector; but persistent agent state (embeddings, learned patterns, skill libraries) might be manipulated in ways not visible as token flows.
6. **Generalization to new agent architectures**: As agent designs evolve (e.g., world models, embodied reasoning), the audit record structure may need redesign.

## Open Questions
1. **Can adversaries evade semantic auditing through paraphrasing or indirect reference?** Does TokenWall remain robust against sophisticated prompt injections that preserve harmful intent while changing surface form?
2. **How does TokenWall scale to very large context windows and multi-turn histories?** Does full-coverage auditing become prohibitive with 100k+ token contexts?
3. **What is the false-positive rate on novel or creative agent behaviors that are benign but unusual?** Does the lightweight inspector's precision/recall degrade in practice, forcing excessive escalation?
4. **Can the framework be extended to distributed or federated multi-agent systems** where token flows cross trust boundaries (e.g., APIs between independent agencies)?
5. **What is the minimum specification for "source" and "sink" boundaries in different agent frameworks** (AutoGen, LangChain, custom)? How much customization is needed per deployment?
6. **Does pre-execution mediation miss dynamic/runtime attacks** where unsafe behavior emerges only after tool execution (e.g., an agent observes a tool output and updates memory unsafely)?

## Key Takeaway
TokenWall demonstrates that persistent AI agents require a new class of defenses—semantic runtime firewalls that mediate token flows at component boundaries—rather than relying solely on training-time alignment or post-hoc output filtering.
