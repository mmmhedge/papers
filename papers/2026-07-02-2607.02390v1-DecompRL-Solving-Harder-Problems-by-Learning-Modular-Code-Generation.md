# DecompRL: Solving Harder Problems by Learning Modular Code Generation

**Authors:** Juliette Decugis, Fabian Gloeckle, Francis Bach, Taco Cohen, Gabriel Synnaeve
**Published:** 2026-07-02 | **arXiv:** [2607.02390v1](http://arxiv.org/abs/2607.02390v1)

## Summary
This paper shows how to make LLMs solve harder coding problems by teaching them to break tasks into modular sub-functions rather than trying to generate complete solutions directly. The key insight: instead of sampling more (expensive GPU time) or using RL naively (which fails when the base model is too far from correct answers), the authors use RL to train models to decompose problems hierarchically, then recombine implementations of different modules to explore a vastly larger solution space cheaply on CPU. On competitive programming benchmarks, this approach cuts GPU costs by ~50× while solving problems standard methods cannot reach.

## Theoretical Background
The paper builds on several established ideas:

**Large language models for code generation** — recent work shows LLMs can write code, but their accuracy on hard algorithmic problems is limited by the size of the solution space they must search.

**Reinforcement learning with verifiable rewards** — RL can improve single-attempt accuracy when you can verify correctness (e.g., test cases pass), but it requires many samples and doesn't help if the base policy assigns near-zero probability to any correct solution.

**Program synthesis via decomposition** — classical work in program synthesis uses hierarchical problem decomposition, but modern LLMs aren't explicitly trained for modular generation.

**Test-time scaling** — prior work shows sampling multiple candidates and picking the best (via verifiable rewards) improves accuracy, but linearly increases compute cost. The paper's core motivation is that this strategy fails on genuinely hard problems where most samples are wrong.

## Core Contributions
1. **DecompRL algorithm**: A novel RL approach that explicitly learns to decompose problems into $n$ independent sub-functions and generate multiple implementations ($k$ variants) for each. This yields up to $k^n$ candidate solutions without $k^n$ separate forward passes—only $k$ per module.

2. **Cheap recombination strategy**: After RL training generates implementations, the evaluation loop shifts to CPU, where you can test all $k^n$ combinations against test cases for negligible cost. This exploits the fact that verifying correctness is cheap; generating code is expensive.

3. **Empirical validation**: On LiveCodeBench and CodeContests (tested with Qwen 2.5 7B and Code World Model 32B), DecompRL solves problems that standard generation, diversity-optimized RL, and repeated sampling cannot solve, while reducing GPU token cost by ~50×.

4. **Scaling analysis**: The paper demonstrates that modular decomposition becomes more valuable as problem difficulty increases (beyond 10^5 tokens per problem), suggesting a fundamental shift in which strategy dominates at different difficulty regimes.

## Connections to Other Work
- **[[Reinforcement Learning from Human Feedback (RLHF)]]**: DecompRL uses RL with verifiable rewards (test case correctness), which is a form of automated reward signal rather than human feedback.
- **Program synthesis and inductive programming**: Classical approaches (e.g., neural module networks, program induction) also use modular decomposition; DecompRL applies this to modern LLMs and code.
- **Test-time scaling and inference compute**: Related to work on [[speculative decoding]], chain-of-thought prompting, and other test-time strategies, but uniquely addresses the problem that sampling fails when the base policy is inadequate.
- **Hierarchical RL and option frameworks**: DecompRL resembles hierarchical RL in learning to decompose, but applied to code generation rather than continuous control.
- **Neural architecture search and meta-learning**: The idea of learning to generate modular solutions relates to learning reusable components, though this work is task-specific rather than architecture-level.

## Practical Applications
1. **Competitive programming assistance**: AutoComplete or hint-generation tools for platforms like Codeforces or LeetCode, where problems are hard enough to defeat standard LLMs but modular decomposition helps.

2. **Enterprise code generation for complex algorithms**: Internal tools for generating business logic (e.g., financial calculations, ETL pipelines) where correctness is verifiable and problems can be naturally decomposed (e.g., data cleaning, transformation, aggregation).

3. **Hardware-constrained code generation**: Embedded systems or edge devices that cannot afford many forward passes; DecompRL's approach shifts cost to CPU verification, reducing GPU/accelerator load.

4. **Coding interview preparation platforms**: Tools that help candidates solve hard algorithmic problems by guiding modular thinking and generating partial solutions that can be combined.

5. **Legacy code modernization**: Breaking down large functions into smaller, independently testable modules during refactoring—an RL agent trained via DecompRL could suggest modular decompositions.

## Product & Startup Ideas
1. **Modular Code Copilot for Interviews**: A coding assistant (VS Code extension) that teaches decomposition by example, generates modular solutions in real-time, and ranks them by correctness. Monetize via subscription for interview prep platforms or enterprise hiring tools.

2. **Smart Bug-Fix Engine**: A service that decomposes buggy code into modules, generates alternative implementations, and suggests fixes. Target: enterprises with large codebases (e.g., financial software, safety-critical systems).

3. **Competitive Programming Coach SaaS**: A platform that uses DecompRL to generate hints and partial modular solutions for programming contest problems, guiding users through decomposition. Freemium model (free basic hints, premium for detailed solutions and analytics).

4. **Automated Test-Case Generation + Verification Toolkit**: Complement to DecompRL—a tool that generates test cases for modules, then lets DecompRL recombine implementations safely. B2B SaaS for companies automating code quality.

5. **Modular Code-Search Engine**: A search system that indexes code by decomposable modules (functions, classes), then uses DecompRL-trained models to find and recombine existing modules to solve new problems. Internal tool for large tech companies, or SaaS for code reuse platforms.

## Technical Limitations
1. **Decomposition overhead**: The RL training cost is not fully characterized. If learning to decompose requires many trajectories, the upfront cost may be high; only hard problems justify the amortized gain.

2. **Modularity assumption**: Not all problems decompose naturally. If a problem is inherently sequential or requires tight coupling between sub-functions, modular recombination may not help. The paper doesn't discuss detection of non-decomposable problems.

3. **Verification requirement**: The approach requires access to correct test cases or a verifiable reward signal. On open-ended tasks (e.g., creative writing, summarization) or tasks with fuzzy correctness (e.g., code style, maintainability), the method breaks down.

4. **Search space explosion**: While $k^n$ is cheap to evaluate on CPU, the number of candidate solutions grows exponentially. If $n$ is large (e.g., 10+ modules), even CPU verification becomes slow; the paper tests up to moderate problem sizes.

5. **Generalization of learned decompositions**: It's unclear how well decomposition policies trained on one problem distribution (e.g., competitive programming) transfer to different domains (e.g., web development, data science).

6. **Training data and RL stability**: RL training for code generation is notoriously sample-inefficient. The paper doesn't discuss how many RL steps or trajectories are needed per problem or problem class.

## Open Questions
1. **What makes a decomposition learnable?** Can you characterize problem properties (graph structure, dependency depth, solution branching factor) that predict whether DecompRL will outperform baselines? Is there a way to detect non-decomposable problems early?

2. **How does $n$ (number of modules) scale with problem difficulty?** As problems get harder, do models tend to decompose into more modules? Is there an optimal $n$, and can it be learned or inferred?

3. **Transfer learning for decompositions**: Can a model trained to decompose problems in one domain (e.g., graph algorithms) decompose problems in another (e.g., dynamic programming)? How much domain-specific RL is needed?

4. **Partial correctness and soft rewards**: What if some modules are correct and others are not? Can the method be extended with soft reward signals or program repair to fix broken implementations?

5. **Comparison to other hierarchical strategies**: How does DecompRL compare to other hierarchical search methods (e.g., best-first search over decomposition trees, A* search with heuristics from intermediate modules)?

6. **Scalability to larger models and problem sets**: The paper uses models up to 32B parameters. How does the method scale to 70B+ models? Does the $\sim$50× GPU cost reduction hold across model
