# SkillFuzz: Fuzzing Skill Composition for Implicit Intents Discovery in Open Skill Marketplaces

**Authors:** Jinwei Hu, Yi Dong, Youcheng Sun, Xiaowei Huang
**Published:** 2026-07-02 | **arXiv:** [2607.02345v1](http://arxiv.org/abs/2607.02345v1)

## Summary
This paper addresses a critical safety gap in LLM-based skill marketplaces: individually benign skills can combine in unexpected ways to cause harmful behavior. The authors propose SkillFuzz, a fuzzing approach that automatically discovers these "implicit intents" by testing skill compositions without needing to execute them, using planning artifacts as a window into agent intent. This matters because open marketplaces with hundreds of community-contributed skills create an exponentially large space of dangerous combinations that traditional auditing methods miss.

## Theoretical Background
The paper builds on several foundational ideas. First, LLM-based agents (systems that use language models to plan and execute tasks) increasingly rely on modular skills — reusable instruction documents that guide behavior. Second, it draws from software testing methodology, particularly fuzzing, which automatically generates and tests inputs to find bugs. Third, it uses differential testing, a technique where an oracle (reference system) identifies anomalies by comparing behavior against a baseline. The key insight is reframing skill-composition safety as a testing problem rather than a formal verification one, making it tractable for large, dynamic marketplaces.

## Core Contributions
1. **Formalization**: First to frame implicit-intent discovery as a fuzzing problem over skill compositions, treating compositions (not individual skills) as the unit under analysis.
2. **Execution-free approach**: Extracts structured skill contracts and uses planning artifacts to detect intent deviation *before* execution, avoiding the need for expensive runtime testing environments.
3. **Scalable search**: Introduces contract-guided Monte Carlo Tree Search to intelligently prioritize potentially conflicting skill combinations, solving the exponential search space problem.
4. **Empirical validation**: Discovers >1,000 distinct implicit intents and confirms >80% of high-risk compositions in execution-time validation.

## Connections to Other Work
This work sits at the intersection of [[alignment]] (ensuring AI systems do what users intend), [[agents]] (autonomous LLM-based systems), and software security. It relates to [[RLHF]] and broader alignment research by addressing a compositional form of misalignment that emerges from interaction, not individual components. It's complementary to mechanistic interpretability work — rather than understanding *why* a model makes decisions, it detects *when* compositions lead to wrong decisions. The fuzzing methodology echoes adversarial testing in ML but applies it to agent behavior rather than model robustness. This also connects to work on agent specification and reward misalignment, but at the marketplace/composition level.

## Practical Applications
1. **Skill marketplace moderation**: Automated pre-admission testing for platforms like Slack App Directory, AWS Lambda marketplace, or OpenAI plugins, catching dangerous combinations before users access them.
2. **Agent safety validation**: Before deploying autonomous agents in enterprise software (e.g., GitHub Copilot-like tools that compose multiple skills), verify no implicit intents emerge.
3. **Regulatory compliance**: For industries requiring governance (finance, healthcare), automated auditing to ensure skill compositions don't create unintended side-effects or regulatory violations.
4. **Bug detection in multi-tool systems**: Testing AI systems that compose multiple external tools (search, calculation, file access) to find emergent privilege escalation or information leakage risks.

## Product & Startup Ideas
1. **SkillGuard SaaS**: A managed testing-as-a-service for skill/plugin marketplaces — operators upload skills, get automated implicit-intent reports before publication, with a dashboard showing high-risk pairs and recommendations.
2. **LLM Agent Compliance Scanner**: B2B tool for enterprises deploying multi-skill agents; integrates into CI/CD, scans new skill additions or versions against existing ones, flags combinations that deviate from intended behavior.
3. **Marketplace Security Insurance**: Underwrite skill marketplace operators by providing automated auditing reports; charge based on marketplace size and frequency of skill updates; resell safety certification to users.
4. **AI Safety Testing Framework**: Open-source/freemium tool for researchers and small teams, with paid tiers offering hosted fuzzing, integration with popular agent frameworks, and explainability reports for flagged combinations.
5. **Skill Composition Analytics**: Observability platform that logs implicit-intent risks in production agent deployments (for mature systems), letting teams monitor and respond to emergent behaviors over time.

## Technical Limitations
1. **Scalability bottleneck**: Even with Monte Carlo Tree Search, as marketplaces grow beyond thousands of skills, pairwise + higher-order interactions become intractable; the exponential problem is reduced but not solved.
2. **Planning artifact assumption**: Relies on agents exposing planning decisions (reasoning traces, action plans) before execution; some LLM architectures or deployment modes may not provide this, making the approach inapplicable.
3. **Baseline dependency**: The "differential oracle" compares against a skill-free baseline, but if the baseline itself is misspecified or unrepresentative, false negatives will emerge.
4. **Skill contract extraction**: Quality depends on how well contracts capture intent; poorly documented or obfuscated skills may yield incomplete contracts, missing subtle interactions.
5. **Limited to composition-level bugs**: Cannot detect single-skill misalignment, only emergent effects; a deeply misaligned skill that harms users alone won't be caught.
6. **Evaluation scope**: Tested on "representative workloads" but unclear how well results generalize to real-world marketplaces with domain-specific skills (e.g., financial trading, medical advice).

## Open Questions
1. **Higher-order interactions**: Can the approach scale to n-way interactions (3+ skills)? The paper focuses on pairwise; do many real implicit intents emerge only from triples or larger coalitions?
2. **Adversarial skill design**: If marketplace users know about SkillFuzz, could they craft skills that evade detection (e.g., by hiding intent until late execution stages)?
3. **Intent specification**: How to formally specify "intended" agent behavior for domains where intent is ambiguous or context-dependent? The baseline approach works but may be too narrow.
4. **Adaptive testing**: Should the fuzzer learn from detected intents to improve future search prioritization within a single marketplace session, or across marketplace versions?
5. **Skill versioning**: When a skill is updated, how much of the prior fuzzing results remain valid? Do implicit intents persist or disappear?
6. **User-driven implicit intents**: Can implicit intents reflect *user* intent (e.g., a user activates skills knowing they'll combine to break ToS)? How to distinguish intentional from unintentional misuse?

## Key Takeaway
Skill composition in LLM agent marketplaces creates an exponential safety problem that traditional per-skill auditing misses, and execution-free fuzzing over composition space is a practical, scalable way to detect these "implicit intents" before users are harmed.
