# ProjAgent: Procedural Similarity Retrieval for Repository-Level Code Generation

**Authors:** QiHong Chen, Aaron Imani, Iftekhar Ahmed
**Published:** 2026-07-09 | **arXiv:** [2607.08691v1](http://arxiv.org/abs/2607.08691v1)

## Summary
This paper presents ProjAgent, a system for generating code at the repository level by retrieving similar functions based on procedural logic rather than just lexical or semantic similarity. The key innovation is using an "agentic workflow" to decompose target functions into reasoning steps, then finding existing repository functions that solve similar procedural problems—even if they use different variable names or operate in different domains. The system combines this procedural retrieval with conventional semantic search and iterative code repair, achieving strong results on a standard benchmark.

## Theoretical Background
Repository-level code generation extends traditional code completion by requiring systems to understand and implement functions while respecting cross-file dependencies and project conventions. Prior retrieval-based approaches use three main signals: lexical similarity (matching keywords), structural similarity (matching code syntax trees), and semantic similarity (matching meaning via embeddings). The core insight in this work is that procedures—the *sequence of logical operations* a function performs—can be similar even when identifiers and application domains differ. The paper builds on recent advances in using large language models (LLMs) as reasoning engines and retrieval-augmented generation (RAG), where relevant context is fetched and provided to the model alongside the generation task. An "agentic workflow" refers to an iterative process where an LLM agent decomposes a problem into steps and acts on each step sequentially.

## Core Contributions
1. **Procedural similarity as a retrieval signal**: Formalizes the notion that functions can be similar in their logical flow and algorithmic approach even when surface features (names, domains) differ. This is the first explicit use of procedural similarity for code retrieval.
2. **Agentic decomposition**: Uses an LLM agent to break down the target function into intermediate reasoning steps, enabling step-by-step procedural similarity matching.
3. **Hybrid retrieval strategy**: Combines procedural similarity retrieval with semantic retrieval to construct richer context for code generation.
4. **Iterative repair with static analysis**: Incorporates compiler and static-analysis feedback in a conservative loop to fix generated code, improving correctness.
5. **Empirical validation**: Demonstrates 41.14% Pass@1 on REPOCOD, outperforming existing retrieval baselines and validating that procedural similarity is a previously unexplored but effective signal.

## Connections to Other Work
ProjAgent advances the field of [[Retrieval-Augmented Generation]] (RAG) for code. It relates to broader work in:
- **Code search and retrieval**: Traditional approaches like BM25 (lexical) or embedding-based methods miss procedural patterns.
- **[[Large Language Models]] for code generation**: Similar to [[Codex]] and [[GitHub Copilot]] but adds explicit procedural reasoning.
- **Program synthesis and repair**: The static-analysis feedback loop connects to work on [[neural program repair]] and feedback-driven generation.
- **Multi-step reasoning in LLMs**: Aligns with chain-of-thought and agentic decomposition paradigms (e.g., [[ReAct]]).
- **Repository-level understanding**: Extends single-function completion to whole-project context, closer to [[Neural Code Summarization]] and [[Cross-File Code Understanding]].

## Practical Applications
1. **IDE plugin for developers**: Real-time code completion that retrieves similar algorithmic patterns from the codebase, accelerating implementation of functions that solve analogous problems.
2. **Automated code migration and refactoring**: When porting code to a new language or framework, find repository functions with equivalent procedural logic to adapt rather than rewrite.
3. **Onboarding tool for large codebases**: Help new engineers understand unfamiliar projects by identifying functions with similar procedural patterns to those they're studying.
4. **Bug detection and pattern standardization**: Identify functions that implement similar logic but with inconsistent patterns, flags potential bugs or opportunities for consolidation.
5. **Legacy code modernization**: Automatically generate updated implementations of old functions by finding procedurally similar modern patterns in the same repository.

## Product & Startup Ideas
1. **ProcedureMatch Pro**: A VS Code/IDE extension that intelligently surfaces similar algorithmic patterns from a codebase in real-time, letting developers copy/adapt proven implementations instead of writing from scratch. Freemium model with per-org pricing.

3. **Cross-Domain Code Transfer Platform**: A service that lets developers query "show me how other projects solve this algorithm" by procedural similarity, unlocking knowledge from open-source or internal codebases. Monetized through API access.

## Technical Limitations
1. **Scalability to very large repos**: The agentic decomposition and retrieval pipeline may become slow on massive codebases with millions of functions; computational complexity not explicitly analyzed.
2. **Procedural similarity definition**: The paper doesn't formally define what constitutes "similar procedures"—relies on LLM reasoning, which may be inconsistent or miss subtle algorithmic patterns.
3. **Language specificity**: Evaluated only on likely a single programming language (exact language not stated in excerpt); unclear how well procedural similarity transfers across languages with different idioms.
4. **Static analysis limitations**: The repair loop uses compiler/static-analysis feedback, which cannot catch logical errors (e.g., off-by-one bugs, incorrect algorithm), only syntax/type errors.
5. **Dependency resolution**: Complex cross-file dependencies and circular imports may not be fully handled; the paper doesn't detail how procedural decomposition accounts for function calls with external effects.
6. **Benchmark generalization**: REPOCOD may not represent real-world repository diversity; results may not generalize to projects with very different structures or domains.

## Open Questions
1. How does procedural similarity scale to repositories with millions of functions? Is there a way to prune the search space without losing recall?
2. Can procedural similarity be formalized mathematically, or is LLM-based decomposition the only practical approach? Would explicit intermediate representations (e.g., control-flow graphs, dataflow) outperform agentic reasoning?
3. How does this approach handle functions with similar procedures but fundamentally different correctness semantics (e.g., both implement "loop and accumulate" but one computes sum, one computes product)?
4. Could this combine with [[RLHF]] or [[Reinforcement Learning]] to reward the agent for finding truly procedurally similar matches during retrieval?
5. Does procedural similarity work across different programming paradigms (functional vs. imperative) or languages, or is it language-specific?
6. How much does the iterative static-analysis repair actually improve correctness vs. just handling trivial syntax fixes? Are logical errors detected and fixed?
7. Would an explicit procedural embedding model (learned from code) outperform agentic decomposition for retrieval at scale?

## Key Takeaway
Procedural logic—the algorithmic strategy and flow of a function—is a powerful but previously unexplored retrieval signal for code generation that can find relevant patterns even when identifiers and domains differ.
