# WattGPU: Predicting Inference Power and Latency on Unseen GPUs and LLMs

**Authors:** Mauricio Fadel Argerich, Jonathan Fürst, Marta Patiño-Martínez
**Published:** 2026-07-02 | **arXiv:** [2607.02391v1](http://arxiv.org/abs/2607.02391v1)

## Summary
WattGPU predicts the power consumption and inference latency of large language models running on unseen GPUs without requiring any hardware profiling. Using only publicly available model specifications and GPU details, it achieves 3–4× better accuracy than physical baselines (like thermal design power) for predicting how much energy an LLM will draw and how long it takes to generate each token, which matters because data centers need to match specific models to efficient hardware without testing every combination.

## Theoretical Background
The paper builds on two established ideas: (1) **predictive modeling for hardware performance**, which has used machine learning to estimate latency or power from system specifications, and (2) **LLM deployment optimization**, where practitioners currently rely on exhaustive profiling (running every model-GPU pair to measure real consumption) or simple physics-based estimates like Load-Scaled Thermal Design Power (TDP — the maximum heat a chip can dissipate under load) and roofline models (theoretical peak compute throughput). The core challenge is that LLM inference involves complex interactions between model size, batch size, memory bandwidth, and GPU architecture that simple formulas miss, yet collecting empirical data for every new GPU-model pair is prohibitively expensive.

## Core Contributions
1. **Two standalone predictive models**: One for mean GPU power draw and one for Inter-Token Latency (ITL — time to generate each subsequent token), both trained only on metadata (LLM parameter count, architecture, etc.) and GPU specs (memory, clock speeds, architecture) with no profiling data required.

2. **Strong generalization**: Demonstrates via rigorous leave-one-GPU-out and leave-one-LLM-out cross-validation that models trained on 7 GPUs can predict unseen 8th GPU power within ≤3.4% median error (offline) and ≤13.5% (server mode), and latency within ≤8.5% in server mode.

3. **GPU ranking correlation**: Models preserve Kendall τ ≥0.76 correlation with actual GPU rankings in server scenarios, meaning operators can reliably identify the most efficient GPU even if absolute error is higher.

4. **Public dataset and reproducibility**: Release of 42 open-source LLMs (0.1B–27B) profiled across 8 NVIDIA server-grade GPUs with code and data.

## Connections to Other Work
This work sits at the intersection of [[deep-learning]] system performance prediction and [[optimization]] for inference deployments. It contrasts with prior work that either (a) requires profiling data from target hardware, limiting generalization, or (b) relies on physics-first models like roofline and TDP that ignore the complexity of LLM computation patterns. The framing aligns with broader research on **hardware-aware model selection** and **energy-efficient inference**, but uses a data-driven regression approach rather than simulation or analytical modeling. The leave-one-out evaluation strategy echoes cross-validation practices in [[large-language-models]] evaluation, adapted here to the hardware domain.

## Practical Applications
1. **Data center GPU procurement**: Operators can predict power and latency costs of deploying new LLMs on candidate GPUs before purchasing, enabling data-driven hardware budgeting.

2. **Model-to-GPU assignment**: Dynamically route inference requests to the most efficient GPU for a given model based on predicted latency and power, minimizing total energy and cost.

3. **Capacity planning**: Estimate how many GPUs of each type are needed to serve a mix of LLMs with latency and power constraints, without prototyping each combination.

4. **LLM provider cost estimation**: Companies offering LLM-as-a-Service can predict operational costs (power, cooling, hardware amortization) for serving new models on different infrastructure.

5. **Sustainability reporting**: Data centers can estimate their carbon footprint from LLM inference before deployment, supporting climate commitments.

## Product & Startup Ideas
1. **WattGPU SaaS dashboard**: Hosted tool where data center operators input their LLMs, candidate GPUs, and constraints (latency, power budget) and receive cost-optimized hardware recommendations and deployment plans.

2. **GPU efficiency benchmarking platform**: Public marketplace/leaderboard comparing GPUs and LLMs by predicted power-per-inference and latency, helping developers and operators make purchasing decisions.

3. **LLM inference cost forecasting for cloud providers**: Integrate WattGPU into a cloud cost estimator (Azure/AWS/GCP style) to auto-quote LLM inference pricing without running test workloads.

4. **Energy-aware model distillation service**: Use WattGPU predictions to recommend which larger LLMs should be distilled (compressed) into smaller versions for efficiency, ranked by power savings.

5. **Hardware-aware fine-tuning optimizer**: SaaS tool that recommends which GPUs to use for fine-tuning specific LLMs to minimize wall-clock time and energy cost, using WattGPU latency/power predictions.

## Technical Limitations
1. **Limited GPU diversity**: Evaluated on only 8 NVIDIA server-grade GPUs; unclear how well the approach generalizes to AMD, Intel, or consumer-grade hardware, or to older/newer architectures not in the training set.

2. **Server scenario error is higher**: 13.5% median error for power in server mode (batching) vs. 3.4% offline, suggesting the model may struggle with complex batching dynamics and GPU resource contention.

3. **Metadata-only training is a constraint and strength**: Using only specs limits applicability (you need accurate GPU documentation and model parameter counts) but also the model's ability to capture undocumented architectural quirks or firmware changes.

4. **LLM size range**: Trained on 0.1B–27B parameter models; behavior of very large models (70B+) or mixture-of-experts architectures may differ substantially.

5. **Latency model assumes steady-state**: Inter-token latency may not capture cold-start overhead, batching scheduling variability, or memory fragmentation effects in production.

6. **No multi-modal or specialized model coverage**: Evaluated on text-only LLMs; unclear how predictions apply to vision-language models or sparse/quantized variants.

## Open Questions
1. **Why does server-mode power error jump to 13.5%?** Is it due to batch scheduling variance, memory bandwidth contention, or GPU power management state transitions? Ablation on batch size and concurrency would help.

2. **How do predictions scale to 50B+ parameter models?** Collecting data on larger models would test whether the relationships discovered in 0.1–27B still hold.

3. **Can the model be extended to multi-GPU configurations?** Most production deployments use tensor parallelism across multiple GPUs; predicting cross-GPU communication overhead is an open problem.

4. **How sensitive is the model to quantization and sparsity?** If models are INT8-quantized or pruned, do the same predictive relationships hold?

5. **Can you predict power and latency under dynamic workloads?** The paper assumes static batching; real data centers have variable request arrival rates and model switching.

6. **Generalization to other GPU vendors**: What would it take to extend WattGPU to AMD Instinct or Intel Gaudi GPUs? Are the relationships fundamentally different?

## Key Takeaway
WattGPU shows that LLM inference power and latency on unseen GPUs can be predicted with high accuracy using only public model and hardware specs—eliminating the need for expensive profiling—which could democratize efficient LLM deployment across data centers.
