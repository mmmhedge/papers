# Efficient Clustering with Provable Guardrails for LLM Inference at Scale

**Authors:** Longshaokan Wang, Wai Tsang Keung, Punit Ghodasara, Roman Wang, Ali Dashti et al.
**Published:** 2026-07-22 | **arXiv:** [2607.19704v1](http://arxiv.org/abs/2607.19704v1)

## Summary
This paper tackles a critical bottleneck in scaling LLM applications: the computational cost of running inference on millions of requests. The authors propose a clustering algorithm that groups similar inputs and reuses LLM outputs across cluster members, with mathematical guarantees that each member stays sufficiently close to its representative. The method is deployed at scale (38 million customers) and achieves 50× cost and latency reduction.

## Theoretical Background
LLM inference cost scales linearly with the number of requests, making naive deployment to millions of users economically infeasible. The paper builds on clustering theory — specifically Mini-batch K-Means for initial cluster formation — and then frames representative selection as a Set Cover problem over alpha-balls in embedding space (a geometric region defined by distance threshold). The approach combines computational geometry with greedy optimization: after clustering, it greedily selects representatives that guarantee all cluster members stay within a bounded distance while maintaining exact categorical attribute matching. This connects to classical approximation algorithms like Johnson-Chvatal heuristics, adapted for the constraint that outputs must be safely reusable across similar inputs.

## Core Contributions
1. **Two-stage algorithm** combining Mini-batch K-Means initialization with greedy representative selection that enforces three hard constraints simultaneously: (i) minimal within-cluster similarity, (ii) exact categorical attribute matching, (iii) scalability to tens of millions of samples.
2. **Theoretical guarantees** that per-sample quality is enforced exactly by construction—not probabilistically or approximately.
3. **Time and memory complexity analysis**: O(nd + n²d/K) time and O(nd + n²/K²) space, achieving linear scaling in n when K grows proportionally with n.
4. **Production deployment results**: 10–1000× speedup vs. standard clustering methods; 50× cost/latency reduction on 38 million customers while preserving personalization.

## Connections to Other Work
This work sits at the intersection of efficient inference optimization and classical clustering. It relates to:
- [[Transformer]] scaling challenges and inference optimization literature
- Standard clustering baselines (K-Means, hierarchical methods) which lack per-sample quality guarantees
- [[RLHF]] and persona-based recommendation systems, where output reuse must preserve downstream task quality
- Approximate nearest-neighbor search and embedding-based retrieval systems
- Set Cover approximation algorithms from combinatorial optimization

The paper's novelty is in *jointly* solving three constraints (similarity, categoricals, scale) that prior work addressed separately or not at all.

## Practical Applications
1. **Persona-based recommendation systems**: Route millions of users through fewer LLM calls by clustering on user attributes; deployed production example shows 50× improvement.
2. **Chatbot platforms at scale**: Cluster user queries by intent/topic; reuse LLM generations for equivalent queries, reducing API costs for high-traffic applications.
3. **Multi-lingual or multi-region content generation**: Cluster requests by language/region; generate content once per cluster, localize outputs.
4. **Customer support automation**: Group support tickets by category/issue type; generate initial responses via LLM once per cluster, customize per ticket.
5. **Real-time ad targeting**: Cluster user contexts; reuse LLM-generated ad recommendations within tight similarity bounds to meet latency SLAs.

## Product & Startup Ideas
1. **LLM Cost Optimizer SaaS**: Plug-and-play middleware that clusters incoming requests before hitting any LLM API (OpenAI, Anthropic, etc.), with dashboards showing cost/latency savings. Charge per billion tokens saved.
2. **Persona Engine for E-commerce**: Pre-cluster customers into personas with LLM-generated profiles and recommendations; sell as managed service to Shopify/commerce platforms.
3. **Enterprise Query Deduplication**: Internal tool for enterprises running LLM workloads (customer service, content generation); automatically cluster and deduplicate queries across teams, with audit trails.
4. **Multi-tenant LLM Platform**: Host LLM inference with built-in clustering; bill customers per unique cluster query rather than per request, massively reducing their costs.
5. **Real-time Recommendation API**: Fast, cheap recommendation layer for mobile/web apps; cluster users in real-time, serve persona-based recommendations at sub-100ms latency.

## Technical Limitations
1. **Embedding quality dependency**: The method assumes high-quality embeddings; poor embeddings or domain mismatch will degrade clustering and break similarity guarantees.
2. **Initial cluster count tuning**: Requires setting K (number of initial clusters); no principled guidance provided. Too few K defeats the purpose; too many K wastes computation.
3. **Categorical constraints rigidity**: Exact attribute matching is enforced, but some use cases may want soft matching or weighted attributes.
4. **Cold-start for new data distributions**: If deployment distribution shifts after initial clustering, guardrails may no longer hold; requires re-clustering.
5. **Reuse safety is application-dependent**: The paper assumes output reusability is safe within similarity bounds, but this depends entirely on the downstream task. For safety-critical applications (medical advice, legal guidance), the guardrails may be insufficient.
6. **Memory scaling for very large K**: When K grows proportionally with n, the O(n²/K²) memory term still requires careful tuning; very large n may still hit memory limits.

## Open Questions
1. **Adaptive K selection**: Can we derive a principled, data-dependent rule for choosing K that optimizes cost/latency without manual tuning?
2. **Hierarchical clustering extension**: Would hierarchical variants (e.g., multi-level clustering) improve both speed and quality for nested persona structures?
3. **Online clustering**: How would the algorithm adapt if new data arrives continuously and must be clustered without recomputing from scratch?
4. **Similarity metric learning**: Can the embedding space be learned or adapted to better align with downstream task similarity?
5. **Formal reusability conditions**: What conditions on the LLM task, outputs, and downstream application formally guarantee that reuse is safe?
6. **Comparison to learned routing**: How does this greedy approach compare to learned routing policies (e.g., neural networks predicting cluster membership)?

## Key Takeaway
Clustering LLM requests with provable safety bounds cuts inference cost 50× by reusing outputs across similar customers.
