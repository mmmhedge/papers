# Reasoning effort, not tool access, buys first-try reliability in agentic code generation: an observational study

**Authors:** Achint Mehta
**Published:** 2026-07-02 | **arXiv:** [2607.02436v1](http://arxiv.org/abs/2607.02436v1)

## Summary
This paper measures what actually improves code generation when AI agents build software independently. Rather than assuming more tools and features help, the authors ran 90 identical agent runs building the same application and measured what mattered: surprisingly, better reasoning effort (asking the model to think harder) lifted first-try success from 28% to 89%, while a testing tool wasted money without improving outcomes. The lesson is simple—match your fix to your actual failure mode.

## Theoretical Background
This work sits at the intersection of [[large-language-models|LLM]] capabilities assessment and agentic AI systems (autonomous agents that take actions to solve problems). Prior research assumes that giving agents more tools and capabilities improves outcomes, but few papers directly measure this assumption in realistic software engineering tasks. The study builds on observational methodology from empirical software engineering, applying it to measure what drives agent success: model capability tier (frontier vs. local models), reasoning effort (how much computational budget the model spends thinking), tool access (testing capabilities), and prompt design (framing the task). The underlying theoretical question is whether agent failures come from missing information (what a tool would fix) or from weak reasoning (what more compute or a better model would fix).

## Core Contributions
1. **First direct measurement** of tool utility in agentic code generation—showing a 42–68% cost increase with zero functional improvement, contradicting the assumption that tools help.
2. **Quantification of reasoning effort's actual impact**—raising reasoning from High to xHigh lifts perfect first-try runs from 28% to 89% and cuts corrective prompts by 5×, establishing reasoning as the dominant lever.
3. **Criterion-level failure analysis**—revealing that container deployment failures (44% failure rate) are model-generation-sensitive, while overall totals hide this variability; most failures stem from weak reasoning, not visible design flaws.
4. **Validation that prompt framing matters but is cheap**—a one-paragraph paraphrase reproduces full visual quality gains of longer design-oriented prompts, separating signal from overhead.

## Connections to Other Work
This relates to broader questions in [[agents|agentic systems]] around sample efficiency and cost-effectiveness. It challenges the implicit assumptions in tool-augmented LLM systems (e.g., [[Agents that reason by tool use]]), showing empirically that reasoning investment often outpaces tool provision. The work also connects to [[RLHF|alignment and reliability concerns]]—first-try reliability is a practical proxy for safety and trustworthiness in deployed systems. It echoes findings in [[mechanistic-interpretability]] research showing that model capacity and compute matter more than we assume, and resonates with recent work questioning whether bigger prompt engineering or more external tools are the right scaling direction for agentic AI.

## Practical Applications
1. **Software development tool prioritization**—teams deploying agentic coders should invest in better models and higher reasoning budgets before buying testing/validation tools.
2. **Cost optimization for AI-assisted coding**—CI/CD pipelines can reduce spend by removing redundant tool integrations and instead using higher reasoning effort on fewer runs.
3. **Deployment reliability improvement**—since container deployment was the dominant failure mode, teams can focus hardening on that specific layer rather than general robustness.
4. **Prompt engineering ROI decisions**—organizations can use short, focused design directives instead of lengthy architectural prompts, cutting overhead while maintaining quality.
5. **Model selection for code generation**—frontier models justify their cost in code generation workflows where first-try reliability is critical; local models are unsuitable without significant reasoning overhead.

## Product & Startup Ideas
1. **Reasoning-as-a-service for code generation**—a wrapper that routes coding tasks to variable reasoning effort tiers (High/xHigh/Extreme) based on task difficulty and cost tolerance, optimizing Pareto frontier of cost vs. reliability.
2. **Agentic code agent benchmarking platform**—a SaaS that runs user specs against multiple models, reasoning budgets, and configurations, then reports cost-adjusted reliability scores to help teams pick configurations without guesswork.
3. **Domain-specific prompt compiler**—a tool that automatically distills verbose design prompts into minimal one-paragraph directives for code generation, capturing the visual quality gains at 10% of the text.
4. **Failure-mode-aware agentic orchestration**—a system that logs where agents fail (reasoning vs. tooling), then adaptively allocates compute (more reasoning budget vs. tool invocation) on retry based on learned failure modes per task type.
5. **Lightweight deployment validator**—a focused tool specifically for catching container/deployment errors in generated code (the dominant failure), rather than general testing, reducing cost overhead from 50%+ to <10%.

## Technical Limitations
1. **Single application scope**—one spec (retrospective board) may not generalize; different application types (data pipelines, ML systems, distributed services) may have different failure mode distributions.
2. **Reasoning effort is opaque**—"xHigh" is a black box; the paper doesn't reveal what xHigh reasoning entails (chain-of-thought length, search budget, ensemble count), limiting reproducibility and mechanistic insight.
3. **Testing tool was generic**—the testing tool used may have been poorly designed or integrated; a better testing harness might show different ROI.
4. **Fixed rubric bias**—the 14-criterion rubric was predetermined; it may miss failure modes that matter in production (security, performance, maintainability).
5. **No user study**—reliability and quality are measured objectively; subjective factors (developer satisfaction, ease of debugging generated code) are unstudied.
6. **Frontier model cost not contextualized**—the paper doesn't compare total cost (model API cost + reasoning effort cost) across tiers, only reasoning effort overhead, leaving the true cost-benefit unclear.

## Open Questions
1. **Does the reasoning/tooling trade-off hold across domains?**—Does a different application (microservices, data ETL, ML training pipelines) flip the result toward tools?
2. **What is xHigh doing?**—Can the paper decompose reasoning effort into interpretable components (search depth, branching, chain-of-thought tokens) to understand the mechanism?
3. **Can reasoning be made cheaper?**—Do alternatives to raw reasoning (hierarchical decomposition, external knowledge bases, test-driven generation) achieve similar gains at lower cost?
4. **How do agents fail on criteria that the rubric missed?**—Does first-try "perfect" generation (89% at xHigh) still have latent bugs, security issues, or performance problems undetected by the rubric?
5. **Does mixing reasoning and tools help?**—What if agents use high reasoning early, then selective tool use on flagged high-risk sections (deployment, auth, data handling)?
6. **How sensitive are results to prompt phrasing?**—Can adversarial prompts or poor instructions erode the reasoning-effort gains?

## Key Takeaway
First-try reliability in agentic code generation is dominated by reasoning compute, not tool access—invest in smarter models or higher reasoning budgets, not more external tools.
