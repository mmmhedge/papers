# Are Performance-Optimization Benchmarks Reliably Measuring Coding Agents?

**Authors:** Zhi Chen, Zhensu Sun, Yuling Shi, David Lo, Lingxiao Jiang
**Published:** 2026-07-01 | **arXiv:** [2607.01211v1](http://arxiv.org/abs/2607.01211v1)

## Summary
This paper audits three major benchmarks used to evaluate coding agents (GSO, SWE-Perf, SWE-fficiency) and finds that their leaderboard rankings are unreliable due to runtime instability across machines, problematic scoring rules, and saturation from public submissions. The work matters because these benchmarks are widely cited as evidence of progress in AI-driven code optimization, but the measurements are noisier and less discriminative than assumed.

## Theoretical Background
Coding agents are AI systems (typically language models augmented with tools) that attempt to solve software engineering tasks autonomously. Performance-optimization benchmarks evaluate these agents by having them apply patches to real code repositories and measuring whether the patched code runs faster than baselines. The field relies on standardized benchmarks to track progress, similar to how [[Attention Is All You Need]] and other foundational works established the importance of reproducible evaluation. Runtime performance is measured in seconds, and "validity" is determined by whether a patch satisfies the benchmark's specified rules (e.g., producing runtime improvements above a certain threshold).

## Core Contributions
1. **Cross-machine replay analysis**: Replayed official reference patches across four Google Cloud machine types and found that most reference patches do *not* satisfy the original benchmark rules consistently across machines—only 39/102 GSO tasks, 11/140 SWE-Perf tasks, and 411/498 SWE-fficiency tasks remain valid across all replays.

2. **Scoring rule sensitivity**: Demonstrated that benchmark rankings are highly sensitive to scoring rules; among eight shared submissions between GSO and SWE-fficiency, official rankings disagree on 9 of 28 pairwise comparisons. SWE-fficiency's scoring rule disproportionately weights the worst tasks (58.5%–82.8% weight on bottom-ten tasks).

3. **Saturation analysis**: Found that 85.3% of replay-valid tasks are already matched or beaten by at least one public submission, indicating the benchmarks have become saturated and offer limited signal for discriminating between agents.

## Connections to Other Work
This work is a critical audit in the spirit of benchmark critique papers across ML—similar to how papers have questioned leaderboard reliability in [[NLP]] evaluation and computer vision. It relates to broader conversations about [[agents]] evaluation methodology and the gap between benchmark performance and real-world utility. The findings echo concerns about reproducibility and environmental sensitivity in empirical ML research.

## Practical Applications
- **Benchmark redesign**: Developers of coding benchmarks can use these findings to redesign tasks with more stable performance signals and fairer scoring rules.
- **Agent evaluation strategy**: Teams building or deploying coding agents can identify which tasks in existing benchmarks have reliable performance signals and weight their evaluation accordingly.
- **Leaderboard transparency**: Benchmark maintainers can publish per-task validity rates and replay stability metrics alongside leaderboard scores to give stakeholders more honest performance estimates.

## Product & Startup Ideas
1. **Certified benchmark platform**: Build a benchmark infrastructure that automatically replays all submissions across multiple hardware configurations and certifies score stability before posting to leaderboards.

2. **Coding-agent evaluation suite**: Develop a premium evaluation service that runs coding agents on curated, stability-audited subsets of benchmarks and provides confidence-bound performance estimates.

3. **Benchmark health-check tool**: Create an open-source auditing tool that repository maintainers can run on their own benchmarks to detect instability, scoring-rule bias, and saturation—and suggest remedies.

4. **ML-assisted benchmark design**: Build a system that automatically identifies problematic tasks (unstable, saturated, mis-scored) and suggests alternative tasks or scoring rules using meta-learning over benchmark properties.

5. **Performance-optimization as a service**: Use insights about which tasks have reliable signals to offer a managed service where teams can submit code and get credible optimization recommendations backed by agent ensembles.

## Technical Limitations
- **Limited scope**: The audit covers three specific benchmarks on Google Cloud machines; results may not generalize to other cloud providers or on-premises hardware.
- **Snapshot in time**: The study captures public submissions at one moment; leaderboard dynamics change as new submissions arrive, potentially altering saturation conclusions.
- **No causal analysis of instability**: The paper documents that reference patches fail across machines but does not deeply analyze *why* (e.g., OS differences, thermal effects, I/O scheduling).
- **Scoring alternatives not proposed**: While the paper critiques existing scoring rules, it doesn't propose or validate alternative rules that would be fairer.
- **Assumes correctness of tools**: Relies on the benchmark infrastructure's implementation of validity rules; bugs in the official checker could confound results.

## Open Questions
- Can a subset of tasks be identified that are both stable across hardware *and* not saturated by public submissions, forming a more reliable "core" benchmark?
- Are there statistical methods (e.g., confidence intervals on runtime deltas) that could make performance-optimization benchmarks more robust to instability without redesign?
- How do coding agents generalize to optimization problems outside the public benchmarks? Do leaderboard rankings correlate with real-world code-optimization impact?
- Should benchmarks standardize on a single hardware target, or publish multiple leaderboards per machine type to acknowledge environmental sensitivity?
- Can automated meta-learning approaches discover scoring rules that are simultaneously fair, discriminative, and stable?

## Key Takeaway
Leaderboard rankings on major coding-optimization benchmarks significantly overstate agent progress and are unreliable due to runtime instability, biased scoring, and task saturation—requiring urgent redesign to restore their credibility as evaluation tools.
