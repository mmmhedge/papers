# Reinforcement Learning without Ground-Truth Solutions can Improve LLMs

**Authors:** Yingyu Lin, Qiyue Gao, Nikki Lijing Kuang, Xunpeng Huang, Kun Zhou et al.
**Published:** 2026-06-25 | **arXiv:** [2606.27369v1](http://arxiv.org/abs/2606.27369v1)

## Summary
This paper introduces RiVER, a reinforcement learning framework that trains LLMs on score-based optimization tasks (like competitive programming) without requiring ground-truth solutions. By using deterministic execution feedback and novel reward calibration techniques, RiVER improves both score-optimizing and exact-solution benchmarks, demonstrating that RL without ground truth can generalize to tasks requiring precise correctness.

## Theoretical Background
The work builds on [[RLHF]] and verifiable reward RL (RLVR) paradigms, which traditionally require binary correctness labels or ground-truth answers. The paper identifies theoretical challenges when extending group-relative RL to continuous, unbounded reward signals: (1) **scale dominance**—raw score magnitudes vary across problem instances, creating inconsistent policy gradients; and (2) **frequency dominance**—common suboptimal solutions can overwhelm rare high-quality outputs in expectation updates. The framework draws on importance weighting and reward shaping principles to address these issues through instance-wise normalization and selective emphasis.

## Core Contributions
1. **RiVER framework**: A reward calibration method combining instance-wise comparison, percentile-based ranking, and bounded feedback to stabilize RL training on continuous-valued scores.
2. **Transfer finding**: Demonstrating that score-based training (e.g., on heuristic contests) transfers to exact-solution benchmarks, despite never seeing ground truth during training.
3. **Empirical validation**: 8.9–9.4% improvement on ALE-Bench rating rank, plus 2.4–3.5% absolute improvement on LiveCodeBench/USACO exact-solution tasks.
4. **Diagnostic insight**: Showing that naive raw-score RL fails to transfer, isolating reward calibration as the critical component.

## Connections to Other Work
- **Related to** [[RLHF]] and process reward models, but sidesteps the need for human annotation by leveraging execution feedback.
- **Contrasts with** outcome-only RL baselines (e.g., raw score optimization), which overfit to the training distribution.
- **Extends** reward shaping literature (e.g., potential-based shaping) to the multi-instance competitive programming domain.
- **Relevant to** curriculum learning and domain adaptation—training on "easier" score-based tasks enables generalization to harder exact-solution tasks.

## Practical Applications
1. **Competitive programming training**: Scaling LLM preparation beyond platforms like LeetCode by incorporating heuristic contest data where exact solutions are rarely defined.
2. **AutoML and hyperparameter optimization**: Training agents on continuous performance scores (e.g., validation accuracy) without oracle optimal configurations.
3. **Scientific discovery**: Optimizing code for molecular simulations, physics engines, or financial models where "ground truth" is intractable but execution scores (e.g., computational efficiency, accuracy metrics) are available.
4. **Game AI**: Training agents on score-based feedback (points, resource management) that transfers to win-condition objectives.
5. **Software engineering**: Code optimization and refactoring where the quality metric is continuous (latency, memory, code complexity) rather than binary correctness.

## Product & Startup Ideas
1. **AI-powered coding bootcamp platform**: A SaaS tool that trains coding models on real competitive programming contests (AtCoder, Codeforces) and sells API access for real-time code optimization and debugging in IDEs.
2. **Autonomous scientific code generator**: A startup building tools for computational chemistry/biology that auto-generates optimized simulation code, trained on execution metrics (runtime, energy, accuracy) without ground-truth solutions.
3. **Enterprise code optimization engine**: Consulting service that fine-tunes LLMs on a company's codebase using performance metrics (latency, throughput, memory) as RL rewards, delivering continuous code efficiency improvements.
4. **Benchmark-as-a-service for LLM evaluation**: A platform aggregating open-ended coding benchmarks (ALE, LiveCodeBench) and serving as a training ground for models, with a marketplace for models ranked by transferability across hidden test sets.
5. **Hybrid tutoring platform**: Interactive platform where students solve code problems, and an RL-trained copilot learns from their submission scores to provide progressively better hints without knowing "the" answer.

## Technical Limitations
1. **Domain specificity**: Evaluation limited to coding tasks; generalization to other domains (translation, summarization) with continuous metrics unclear.
2. **Score design sensitivity**: RiVER's effectiveness depends on well-calibrated execution scores; poorly designed metrics could mislead training.
3. **Computational cost**: Training requires sampling multiple solutions per instance; scalability to very large models or datasets not demonstrated.
4. **Transfer assumptions**: The paper assumes score-based tasks are "similar enough" to exact-solution tasks; this may not hold for sufficiently distant domains.
5. **Baseline fairness**: Comparisons are primarily against raw-score RL; comparison to other recent reward shaping or process reward approaches would strengthen claims.
6. **Interpretability gap**: No analysis of *what* representations RiVER learns or *why* it transfers—mechanistic understanding is limited.

## Open Questions
1. How does RiVER perform on domains with sparse or noisy continuous feedback (e.g., partial test case scores)?
2. Can the framework extend to multi-objective optimization (e.g., code quality + speed + readability)?
3. What is the theoretical sample complexity of RiVER vs. ground-truth RL? Does it incur a cost?
4. How sensitive is performance to hyperparameters (percentile thresholds, calibration ranges)? Is there a principled way to set them?
5. Can RiVER be combined with process rewards or intermediate supervision to further improve convergence?
6. Does the framework work for closed-source LLMs where we cannot directly train, or is it limited to open-weight models?

## Key Takeaway
Score-based reinforcement learning with proper reward calibration can train general coding ability without ground-truth solutions and generalize to exact-solution benchmarks, making RL practical for domains where correctness is ill-defined.
