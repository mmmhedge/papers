# StructureClaw: Traceable LLM Agents and an Executable Benchmark for Structural Engineering Workflows

**Authors:** Sizhong Qin, Yi Gu, Yao Jiang, Ao Cai, Changjian Zhou et al.
**Published:** 2026-07-16 | **arXiv:** [2607.14896v1](http://arxiv.org/abs/2607.14896v1)

## Summary
StructureClaw is a workbench and benchmark for evaluating LLM agents on structural engineering tasks. Unlike typical LLM benchmarks that score isolated answers, this system traces the entire workflow—from interpreting requirements through modeling, validation, solving, and reporting—ensuring agents produce complete, executable, internally consistent engineering solutions. This matters because real engineering demands more than fluent text: it requires a chain of interdependent artifacts that hang together.

## Theoretical Background
LLM agents (language models guided by tools and memory to solve multi-step tasks) have been applied to code generation and scientific workflows, but evaluation remains shallow. Most benchmarks measure question-answering accuracy or script correctness in isolation—they don't verify that intermediate outputs are valid, that downstream steps can consume them, or that the full workflow produces executable results. Structural engineering presents a canonical case: a valid solution requires not just correct math but correct *ordering* of dependencies, consistent state across artifacts, and solutions that pass multiple validation gates.

The work builds on agent architectures with typed tools (functions agents can call with enforced input/output schemas) and artifact-centered state management—treating intermediate engineering outputs as first-class objects that must satisfy typed constraints and pass assertions.

## Core Contributions
1. **Artifact-centered evaluation framework**: Rather than grading final answers, StructureClaw verifies that all intermediate artifacts (interpreted specs, computable models, validation records, solver outputs, code-check reports) exist, are typed correctly, and pass domain-specific assertions in a single executable run.

2. **StructureClaw-Bench**: A curated set of 150 structural engineering scenarios with three difficulty tiers—standard workflows, interactive robustness (invalid inputs, edge cases), and multimodal reconstruction (from sketches/images)—where success requires end-to-end correctness.

3. **Empirical finding**: Full automatic workflow agents (with governed skills and local analysis backends) achieve 88.6% success vs. 56.8% for generic-skill baselines, but interactive and multimodal tests expose two critical gaps: safe handling of invalid numerical inputs and fixture-consistent model reconstruction.

4. **Implemented workbench**: A runnable system where LLM agents operate through typed tools, shared mutable artifact state, and local solvers (e.g., structural analysis engines), making workflows verifiable and debuggable.

## Connections to Other Work
- **LLM agent evaluation**: Relates to broader work on [[ReAct]] (reasoning + acting) and tool-use evaluation, but moves beyond script-grading to workflow-level correctness.
- **Code generation and verification**: Similar spirit to work on [[Program Synthesis]] and executable program verification, but applied to domain-specific engineering artifacts.
- **Multi-step reasoning**: Connects to [[Chain-of-Thought]] prompting and its extensions, showing that tracing intermediate states matters more than tracing reasoning steps alone.
- **Multimodal agents**: Adds to growing literature on vision-language models for domain tasks, identifying specific failure modes (fixture consistency) that text-only evaluation misses.

## Practical Applications
1. **Automated structural design review**: Feed sketches or requirements to StructureClaw agents; get validated engineering specs, FEA models, and compliance reports without manual handoffs.
2. **Construction document generation**: Generate code-compliant structural drawings, calculations, and certificates from narrative briefs, reducing design cycle time.
3. **Regulatory compliance automation**: Agents verify that designs meet local building codes by running assertions at each step; flag violations before they reach inspectors.
4. **Retrofit and adaptive reuse**: Agents interpret existing building geometry (from scans, CAD files) and propose valid structural modifications that preserve original fixtures.
5. **Engineering education**: Interactive tutoring: agents guide students through structural workflows, highlight logical gaps, and validate homework submissions against real code standards.

## Product & Startup Ideas
1. **StructureAI**: A SaaS platform for small AEC (architecture/engineering/construction) firms; architects upload design briefs and CAD, agents generate stamped structural reports and FEA analyses ready for permitting.
2. **BuildBot Compliance**: Real-time LLM agent that watches live construction and flags deviations from approved structural plans using site photos and sensor data.
3. **Retrofit-as-a-Service**: Agent-powered advisory for building owners; analyzes existing structures (via laser scans, thermal imaging) and proposes retrofit options with costs and compliance proofs.
4. **EngineerGPT for Contractors**: Tablet app for on-site teams; agent interprets verbal or sketched change orders, generates updated load calculations, and checks against original approvals—reduces RFI cycles.
5. **Structural Code Copilot**: IDE-like tool embedded in structural design software; as engineers sketch, an agent validates geometry, material choices, and code compliance in real time, suggesting fixes inline.

## Technical Limitations
1. **Domain specificity**: StructureClaw is calibrated for structural engineering; generalization to other engineering domains (HVAC, electrical, plumbing) is unclear and likely requires new artifact schemas and validation rules.
2. **Numerical robustness**: The benchmark identifies but does not fully solve safe handling of invalid or near-singular numerical inputs—agents still fail when inputs are out-of-range or malformed.
3. **Multimodal reconstruction**: Fixture-consistent model assembly from sketches/images remains hard; the paper shows this is a gap but offers no algorithmic fix, relying on better training data or prompting.
4. **Real-world variability**: The 150 scenarios are "controlled"—they don't capture the full messiness of real projects (ambiguous requirements, conflicting standards, partial/dirty CAD files, legacy code variations).
5. **Closed-loop feedback**: Agents don't iteratively refine when validation fails; they just fail. Real engineers debug; the system doesn't yet model that revision loop.
6. **Tool availability**: Assumes access to local structural solvers and code-checking engines; many firms use proprietary or expensive licensed software that StructureClaw cannot currently wrap.

## Open Questions
1. **Interactive refinement**: How should agents repair workflows when validation fails mid-pipeline? Should they backtrack, refactor artifacts, or request human clarification?
2. **Multimodal grounding**: Can agents reliably infer structural intent (load paths, boundary conditions) from hand sketches or construction photos, and does vision-language pretraining help?
3. **Code diversity**: Do agents trained on one building code (e.g., IBC, Eurocode) transfer to others, or do they overfit to specific clause wording and calculation rules?
4. **Scalability**: Can the artifact-centered evaluation scale to larger, more complex structures (tall buildings, bridges, industrial facilities) without explosive combinatorial explosion in test scenarios?
5. **Human-in-the-loop**: What's the optimal handoff between agent and engineer? Can agents reliably flag ambiguities and request human input without blocking the workflow?
6. **Explanation and traceability**: The paper emphasizes traceability; can agents explain *why* a design is valid, not just produce certificates—i.e., can they justify design choices in plain English to non-experts?

## Key Takeaway
Tracing entire engineering workflows through executable artifact chains reveals LLM agent failures that final-answer grading misses.
