# Deep4ge: DNN Training Trajectories for Fault Detection and Diagnosis

**Authors:** Sigma Jahan
**Published:** 2026-07-14 | **arXiv:** [2607.12868v1](http://arxiv.org/abs/2607.12868v1)

## Summary
This paper introduces Deep4ge, a public dataset of 14,227 deep learning training runs with documented faults, designed to enable automated detection and diagnosis of bugs in neural network code. The dataset captures per-epoch training metrics and behavioral features, addressing a gap in software engineering tools for debugging deep learning systems by providing researchers with real faults injected into TensorFlow/Keras programs.

## Theoretical Background
The work builds on two research traditions: (1) software fault detection and diagnosis, which uses program traces and runtime metrics to identify bugs, and (2) deep learning debugging, which observes how training dynamics (loss curves, gradient flow, weight distributions) change when code contains errors. Prior work has shown that subtle implementation faults—off-by-one errors, incorrect hyperparameters, wrong layer configurations—can be diagnosed by analyzing epoch-level training behavior rather than relying on final test accuracy alone. This paper operationalizes that insight by creating a systematic benchmark where faults are synthetically introduced via code transformations, allowing supervised learning models to learn patterns that distinguish correct from faulty training trajectories.

## Core Contributions
1. **Deep4ge Dataset**: 14,227 training runs (9,845 faulty, 4,382 correct) with comprehensive per-epoch instrumentation—a first public benchmark specifically for DNN fault detection/diagnosis.
2. **27 Source-Code Transformations**: Fault injection mechanisms spanning seven categories (e.g., weight initialization errors, gradient computation bugs, layer configuration mistakes), grounded in real Stack Overflow code.
3. **Rich Feature Engineering**: 26 epoch-level features capturing weights, gradients, activations, loss/accuracy trends, learning rates, and hardware metrics—enabling multiple downstream tasks.
4. **Three Task Formulations**: Binary fault detection (faulty vs. correct), multi-class diagnosis (which fault type), and early prediction (detect faults from incomplete training runs).
5. **Open Artifact Release**: Publicly available dataset and fault-injection framework for reproducibility and follow-up research.

## Connections to Other Work
This work sits at the intersection of [[software testing]] (automated fault detection), [[neural network interpretability]] (understanding training dynamics), and [[machine learning debugging]]. It complements mechanistic interpretability research by focusing on *observable* training traces rather than internal mechanism explanations. Similar to anomaly detection in time-series, the work treats training trajectories as sequences and applies supervised classification. The fault taxonomy echoes classical software engineering (off-by-one errors, configuration mistakes) but specialized for deep learning. Related datasets like [[MNIST]], [[ImageNet]], and [[TensorBoard]] traces exist, but Deep4ge is unique in systematically pairing correct and deliberately faulty runs with ground-truth fault labels.

## Practical Applications
1. **Automated Code Review Tools**: IDE plugins or CI/CD pipelines that flag likely bugs by comparing a model's training trajectory against the Deep4ge-trained detector before expensive full training completes.
2. **Early Training Abort**: Stop training runs early if trajectory anomalies match known fault signatures, saving GPU compute costs.
3. **Hyperparameter Validation**: Distinguish genuine hyperparameter choices from implementation errors by monitoring whether training behavior matches learned "healthy" patterns.
4. **Researcher Debugging Aid**: Interactive tools that highlight which fault category a suspicious training run most closely resembles, narrowing manual debugging scope.
5. **Model Card Metadata**: Augment model documentation with training health checks—e.g., "this model's training trajectory passed 95% of fault-detection tests."

## Product & Startup Ideas
1. **MLOps Monitoring Service**: SaaS platform that integrates with training pipelines (Weights & Biases, MLflow), ingests per-epoch metrics, and issues real-time alerts when trajectories deviate from healthy patterns; charge per training run or per-seat.
2. **AutoDebugger for ML Engineers**: Standalone tool that replays faulty training runs, compares them to Deep4ge signatures, and generates ranked hypotheses ("gradient clipping bug ranked #1"), accelerating manual debugging.
3. **Training Health Insurance**: Automated insurance product for cloud ML platforms—users pay a small premium per training run; platform uses Deep4ge classifier to flag high-risk runs and either abort or alert, reducing wasted compute spend.
4. **Educational Platform**: Interactive course on "debugging DNNs" with embedded Deep4ge cases—learners see faulty training curves and must diagnose the root cause before seeing the answer.
5. **Enterprise ML Governance**: Compliance tool for regulated industries (finance, healthcare) that certifies training runs passed fault-detection audits; generates audit logs and breach reports for model validation.

## Technical Limitations
1. **Scope Constraints**: Faults are synthetically injected via 27 source-code transformations; real-world bugs may exhibit different signatures or interact in ways the dataset doesn't capture.
2. **Architecture Bias**: Training runs derived from Stack Overflow examples and adapted to TensorFlow/Keras—may not generalize to custom architectures, PyTorch-dominant code, or large-scale distributed training.
3. **Early Prediction Trade-off**: Early fault detection (from partial runs) likely sacrifices precision for recall; no analysis of the Pareto frontier or optimal stopping points.
4. **Feature Engineering**: 26 hand-crafted features assume standard supervised learning setups; may not transfer to reinforcement learning, meta-learning, or federated training contexts.
5. **No Interacting Faults**: Dataset likely treats faults independently; multiple simultaneous bugs may produce emergent signatures not represented.
6. **Hardware Variability**: Recorded hardware metrics are snapshot-dependent; generalization across different GPUs, TPUs, or cluster configurations is untested.

## Open Questions
1. Can the Deep4ge classifier detect *naturally occurring* bugs in real GitHub or Kaggle competition code, or does synthetic injection create unrealistic patterns?
2. How does performance degrade when training dynamics shift due to data distribution changes, different initializers, or novel optimizers not seen during dataset generation?
3. Is there an optimal epoch at which to halt training for early fault prediction, and does it depend on fault category?
4. Can the learned features transfer to other programming frameworks (PyTorch, JAX) or only TensorFlow/Keras?
5. Do certain fault types consistently produce "easier" detection signals than others? Can a practitioner quickly assess confidence in a flagged fault?
6. How do results scale to very large models (billions of parameters, distributed training) where per-epoch logging overhead becomes non-trivial?

## Key Takeaway
Deep4ge provides a public benchmark dataset to train automated tools for detecting and diagnosing bugs in deep learning code by analyzing per-epoch training behavior.
