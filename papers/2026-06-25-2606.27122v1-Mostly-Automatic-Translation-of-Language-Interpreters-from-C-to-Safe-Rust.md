# Mostly Automatic Translation of Language Interpreters from C to Safe Rust

**Authors:** Bo Wang, Brandon Paulsen, Joey Dodds, Daniel Kroening, Umang Mathur et al.
**Published:** 2026-06-25 | **arXiv:** [2606.27122v1](http://arxiv.org/abs/2606.27122v1)

## Summary
Reboot is a mostly-automatic system that translates C interpreter programs to safe Rust, reducing manual intervention to 1–11 user interactions per codebase while maintaining 100% test suite pass rates. The work combines feature reduction (incremental decomposition by program features) with a multi-agent architecture that orchestrates unreliable LLM-based coding agents, eliminating memory vulnerabilities like buffer overflows and use-after-free bugs inherent to C implementations.

## Theoretical Background
This work builds on several established areas:
- **Program synthesis and code generation**: Uses LLM-based agents to generate code transformations, extending [[code-generation]] techniques to the domain of language translation.
- **Ownership and type theory**: Leverages Rust's ownership model and type system as a formal constraint that guarantees memory safety—the paper assumes familiarity with lifetime annotations and borrow checking.
- **Multi-agent orchestration**: Applies insights from [[multi-agent-systems]] to coordinate multiple fallible coding agents through validation loops, similar to debates or consensus mechanisms in agent systems.
- **Incremental validation**: Builds on software testing and continuous integration practices, where each intermediate milestone is validated before proceeding.

## Core Contributions
1. **Feature reduction methodology**: A novel decomposition strategy that breaks translation into a sequence of increasingly complete programs, each testable and validated independently. This differs from end-to-end translation by creating checkpoints that prevent error accumulation.
2. **Multi-agent feedback loop architecture**: An orchestration framework that handles unreliable LLM coding agents by iterating on failures, collecting feedback from test suites, and guiding the agents back on track—contrasted in ablation studies against single-agent approaches.
3. **Practical demonstration at scale**: Six real-world interpreter translations (6k–23k LoC), proving the approach works on actual production interpreters (mujs, etc.) rather than toy examples.
4. **Security validation**: Demonstrates that C memory bugs (heap buffer overflows, use-after-free) are eliminated in the translated Rust, with 62%–92% pass rates on held-out validation tests.

## Connections to Other Work
- **[[Program synthesis]]**: Related to neural program synthesis and sketch-based synthesis, but focused on the specific challenge of safe language translation rather than general program generation.
- **[[LLM agents and tool use]]**: Extends work on LLM-based coding agents (like Codex or GPT agents) by adding structured validation and failure recovery; similar to ideas in agentic AI loops.
- **[[Memory safety]]**: Complements formal verification approaches (e.g., theorem proving) by leveraging Rust's type system as a built-in safety guarantee rather than post-hoc verification.
- **[[Domain adaptation]]**: The feature-reduction strategy is a form of curriculum learning, easing the transition from C to Rust by incrementally increasing complexity.

## Practical Applications
1. **Legacy system modernization**: Organizations with large C codebases (interpreters, VMs, domain-specific languages) can semi-automatically migrate to memory-safe Rust, reducing security audit costs.
2. **Security-critical interpreter deployment**: JavaScript engines, Python interpreters, and domain-specific language runtimes can be translated to eliminate entire classes of memory bugs, improving production reliability.
3. **Supply chain risk reduction**: Translating widely-used interpreters (e.g., in embedded systems or cloud environments) reduces the attack surface for memory exploitation vulnerabilities.
4. **Compliance and certification**: Industries requiring memory-safe code (aerospace, medical devices, financial systems) can justify safer interpreter implementations.

## Product & Startup Ideas
1. **Reboot-as-a-Service**: A web platform where teams upload C interpreter source code, select a feature decomposition strategy, and receive a safe Rust translation with minimal manual review. Charge per lines-of-code or per translation project.
2. **Memory-Safety Audit Tool**: Market Reboot as a security assessment product—translate C libraries to Rust, run a diff-based vulnerability report highlighting bugs eliminated, sell to enterprises concerned about supply chain memory safety.

## Technical Limitations
1. **Multi-agent fallibility**: The system relies on LLM coding agents which remain unpredictable; no guarantees are provided on convergence time or whether all features can be successfully translated. Some programs may require more than 11 interventions.
2. **Feature decomposition manual specification**: The choice of which features to reduce and in what order appears to require domain expertise. The paper does not present automated strategies for decomposing arbitrary programs.
3. **Validation test coverage dependency**: Validation relies on provided test suites. The 62%–92% pass rates on held-out validation tests suggest the system can miss subtle bugs or edge cases not covered by tests.
4. **Performance not addressed**: No performance profiling is discussed. Safe Rust translations may have runtime overhead (e.g., bounds checking, borrowing constraints) compared to optimized C code.
5. **Incomplete feature support**: The paper does not clarify which C features (macros, inline assembly, function pointers, complex pointer arithmetic) are hardest to translate; some interpreters may require architectural changes.
6. **Scalability to very large codebases**: Tests max out at 23k LoC; unclear how the approach scales to monolithic 100k+ LoC interpreters.

## Open Questions
1. **Automated feature reduction**: Can feature decomposition be learned or inferred automatically from call graphs, module structure, or test dependencies, removing the need for manual specification?
2. **Convergence guarantees**: What properties of the source C code and LLM quality determine whether the system converges in k interventions? Can we predict or bound the number of human interactions needed?
3. **Performance characterization**: How do safe Rust translations compare to C originals in execution speed and memory usage? Are there systematic optimizations that close performance gaps?
4. **Partial safety**: Can the approach handle C code with mandatory unsafe blocks (e.g., FFI, hardware access), preserving safety guarantees for the non-unsafe portions?
5. **Cross-language generalization**: Does feature reduction + multi-agent orchestration generalize to other unsafe-to-safe language pairs (C++ to Rust, Go to Kotlin, etc.)?
6. **Validation test synthesis**: Can validation tests be automatically generated (e.g., via fuzzing or symbolic execution) when the original test suite is insufficient?

## Key Takeaway
Reboot demonstrates that **feature-reduction decomposition combined with multi-agent LLM orchestration can semi-automatically translate real C interpreters to memory-safe Rust with minimal human oversight**, achieving both correctness and security improvements on codebases up to 23k lines.
