# AxDafny: Agentic Verified Code Generation in Dafny

**Authors:** Benjamin Breen, Austin Letson, Borja Requena Pozo, Leopoldo Sarra
**Published:** 2026-06-30 | **arXiv:** [2606.32007v1](http://arxiv.org/abs/2606.32007v1)

## Summary
This paper presents AxDafny, a system where AI agents generate not just code but also the formal proofs needed to verify that code is correct. The key innovation is a repair loop: when verification fails, the agent automatically suggests fixes (new invariants, assertions, termination arguments) and tries again. The authors also release a benchmark of 250 programming problems in Dafny with formal specs, and show that their approach achieves 92.7% verification success—a significant improvement over prior work.

## Theoretical Background
Dafny is a programming language designed for writing provably correct code—programs come with formal specifications and machine-checkable proofs. Traditionally, humans write both the code and the proof artifacts (invariants, assertions). Recent work has explored using language models like GPT to generate code, but almost all prior attempts stop at executable output; they don't generate the formal proof machinery. This paper builds on two trends: (1) agentic AI systems that iteratively refine outputs based on feedback, and (2) the growing use of large language models for code generation. The key insight is treating verification failure as actionable feedback—similar to how an agent might use a compiler error message to fix code.

## Core Contributions
1. **AxDafny framework**: A verifier-guided repair loop where an LLM generates code + proof artifacts, receives verification feedback, and automatically repairs by suggesting new invariants, assertions, and termination arguments—without human intervention.
2. **LiveCodeBench-Pro-Dafny (LCB-Pro-Dafny)**: A new benchmark of 250 competition-style problems translated into Dafny with formal specifications and a verifier-based evaluation harness (not just pass/fail on test cases).
3. **Empirical insights**: Demonstration that verification success and runtime test performance measure different code properties—you can pass tests but fail formal verification, and vice versa. AxDafny achieves 92.7% on DafnyBench, 6.5 percentage points above the prior state-of-the-art.

## Connections to Other Work
This extends the paradigm of [[RLHF]] (reinforcement learning from human feedback) to formal verification: instead of human judges, the Dafny verifier itself provides the signal. It relates to recent work on [[agents]] that use tool feedback loops (e.g., code generation systems that read compiler errors). The paper also touches on the longstanding challenge of [[alignment]]—ensuring AI-generated code doesn't just pass tests but satisfies formal specifications, which is a stronger notion of correctness. Prior work on proof-hint baselines and neural-guided theorem proving are direct predecessors, but AxDafny is the first to close the loop end-to-end at this scale.

## Practical Applications
1. **Safety-critical systems**: Autonomous vehicles, medical devices, and financial software where formal verification is mandatory—AxDafny could dramatically reduce the human effort to generate certified code.
2. **Smart contract development**: Blockchain code must be formally verified to prevent exploits; this system could accelerate secure contract writing.
3. **Compiler and OS kernel development**: Projects like seL4 and CakeML already use formal methods; AxDafny could assist engineers in generating verified components faster.
4. **Regulated software**: Banking, aviation, and healthcare systems with formal verification requirements could use this to reduce certification costs.

## Product & Startup Ideas
1. **"VerifiedAI"**: A cloud IDE for writing formally verified code in Dafny/Coq/Lean, where agentic assistance auto-generates proofs and integrates AxDafny-style repair loops—sell to enterprises needing certified software.
2. **Smart contract security platform**: A Solidity-to-Dafny transpiler + AxDafny backend that automatically generates verified contract code, marketed to DAOs and DeFi protocols as "proof-as-a-service."
3. **Formal verification SaaS for regulated industries**: A GitHub integration that runs AxDafny-style verification on pull requests in healthcare/fintech codebases, alerting teams when code lacks formal proof.
4. **AI-assisted theorem proving assistant**: Extend AxDafny to Isabelle/Coq/Lean and license it to academic researchers and industry labs doing proof-heavy work.
5. **Compliance-code generator**: For industries like medical devices, auto-generate reference implementations with formal proofs baked in, reducing FDA/regulatory review time.

## Technical Limitations
1. **Scalability to large proofs**: The paper doesn't address how AxDafny scales to complex, multi-method systems where proofs have deep mutual dependencies—the repair loop may get stuck in local optima.
2. **Specification quality dependency**: AxDafny relies on developers providing correct formal specifications; garbage specs → garbage proofs. No mechanism is shown for auto-synthesizing specs.
3. **Limited to Dafny**: Generalization to other proof assistants (Isabelle, Coq, Lean 4) is unclear—each has different syntax and proof culture.
4. **Termination arguments**: The paper claims AxDafny generates termination arguments, but termination proofs are notoriously hard; it's unclear how well the agent handles complex ranking functions.
5. **LLM hallucination under pressure**: When stuck, the model may hallucinate spurious invariants that temporarily satisfy the verifier but are semantically wrong—no ground-truth check exists.
6. **Computational cost**: Iterative verification + LLM calls can be expensive; no analysis of wall-clock time or token usage compared to human proof writing.

## Open Questions
1. **Can AxDafny handle multi-threaded or distributed algorithms?** Concurrent invariants are notoriously hard; does the agent learn to reason about them?
2. **How does performance degrade on problems requiring novel proof techniques?** The benchmark may overweight problems solvable with standard patterns.
3. **Can the repair loop be accelerated with smaller, fine-tuned models?** Does GPT-4o-level capability matter, or could smaller models work with better in-context learning?
4. **How does AxDafny interact with automated tactic solvers?** Could integration with SMT solvers (Z3, etc.) replace some repair steps?
5. **What's the relationship between test coverage and formal verification?** The paper notes they measure different things—can you predict one from the other?
6. **Can the system learn proof strategies across problems?** Does solving problem N help with problem N+1, or does each problem start cold?

## Key Takeaway
AxDafny demonstrates that LLMs can iteratively generate both code and formal proofs by treating verification feedback as a repair signal, achieving state-of-the-art results and suggesting that agentic loops over formal tools are a viable path toward AI-assisted certified software development.
