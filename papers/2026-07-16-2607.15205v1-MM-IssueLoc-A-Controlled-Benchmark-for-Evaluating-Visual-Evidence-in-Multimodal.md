# MM-IssueLoc: A Controlled Benchmark for Evaluating Visual Evidence in Multimodal Repository-Level Issue Localization

**Authors:** Shaoxiong Zhan, Shi Hu, Boyu Feng, Hai Lin, Andrew Gong et al.
**Published:** 2026-07-16 | **arXiv:** [2607.15205v1](http://arxiv.org/abs/2607.15205v1)

## Summary
This paper introduces MM-IssueLoc, a benchmark for evaluating how well AI systems can locate bugs in code repositories when given both text descriptions and visual evidence like screenshots or error dialogs. Current benchmarks treat bug localization as text-only, so this work fills a gap by creating a controlled dataset with 652 real issues across 23 programming languages and measuring whether visual information actually helps systems find the right code location.

## Theoretical Background
Repository-level issue localization is a code understanding task where systems must identify which files or functions contain bugs based on issue reports. Prior work treated this as a text retrieval problem — matching issue descriptions to code using language models or semantic search. Multimodal learning combines text and image inputs, typically through embedding models or vision transformers that process both modalities. However, existing software engineering benchmarks (like SWE-bench) focus on end-to-end patch generation, which conflates the localization step with code repair, making it unclear whether visual inputs actually aided the system or were simply ignored.

## Core Contributions
1. **Controlled benchmark design**: MM-IssueLoc isolates bug localization from patch synthesis, enabling clear measurement of visual evidence contribution. Previous multimodal SE benchmarks evaluated repair end-to-end, obscuring localization performance.

2. **Explicit visual annotation**: 652 real-world issues annotated with 7 image categories (screenshots, error dialogs, UI states, logs, etc.) and 4 relevance levels, enabling fine-grained analysis of which visual types matter.

3. **Diagnostic methodology**: VCE (Visual-to-Contextual-Evidence) conversion transforms images into structured text, allowing direct comparison of image-augmented vs. text-only retrieval on the same issues.

4. **Cross-benchmark analysis**: Shows that high performance on text-dominant benchmarks (e.g., SWE-bench) does not transfer to multimodal localization, revealing a domain gap.

5. **Baseline systems**: Evaluates LLM-based agents and retrieval-based systems; introduces MM-IssueLoc-VL-Emb, a controlled multimodal retriever for reproducible evaluation.

## Practical Applications
1. **Bug triage automation**: Software teams can route incoming issues to the correct developers faster by automatically pinpointing affected code locations from screenshots and error dialogs.

2. **CI/CD visual error detection**: Integrate MM-IssueLoc-style systems into continuous integration pipelines to automatically detect and locate UI regression bugs from rendered screenshots.

3. **Mobile/web app testing**: Automatically map visual bugs (broken layouts, rendering issues) back to source code locations without manual inspection.

4. **Internal tool bug reporting**: Employees report bugs with screenshots; system auto-suggests which code file/function to check, reducing developer context-switching.

5. **Open-source issue routing**: Route community-submitted issues (often with screenshots) to the right repository location, accelerating triage.

## Product & Startup Ideas
1. **Visual Bug Locator SaaS**: A GitHub/GitLab integration that automatically pins issue screenshots to exact code locations, reducing developer time on triage by 50%.

5. **Enterprise Internal Tools Support Bot**: Chatbot that accepts user-uploaded screenshots of broken internal apps/dashboards and returns pinpointed code locations to the platform team.

## Technical Limitations
1. **Image-to-text conversion fidelity**: VCE-based diagnostics rely on converting images to text; lossy transformations may discard visual nuances (layout, color, animation) that humans use for bug diagnosis.

2. **Limited baseline depth**: Current best agent (38.96% file Acc@5) suggests existing LLM architectures struggle with multimodal reasoning at repository scale; unclear if this is a model capacity or benchmark hardness issue.

3. **Language and domain bias**: 23 languages covered, but benchmark may skew toward common languages/projects; performance on low-resource languages or niche domains unknown.

4. **Image relevance annotation variability**: 4-level relevance scale may not capture nuanced cases where images provide partial or indirect hints; inter-annotator agreement not reported.

5. **Generalization risk**: Test issues are from real repos but may not reflect the full diversity of bug types (e.g., concurrency bugs, security issues) or visual artifact types.

6. **Multimodal architecture choices**: Benchmark does not prescribe how to fuse vision and text; different embedding strategies may perform very differently, limiting reproducibility.

## Open Questions
1. Does performance improve monotonically with more training data on multimodal localization, or is there a fundamental architectural bottleneck?

2. Can retrieval-based systems outperform LLM agents on this task with better vision encoders (e.g., newer vision transformers), or is agent-based reasoning necessary?

3. How much do existing LLMs *use* visual input vs. solving via text-only cues? Can you measure or steer this via prompting or fine-tuning?

4. What visual artifact types (error dialogs, UI screenshots, logs, etc.) most reliably help localization? Should systems prioritize certain image categories?

5. How does performance scale to very large repositories (thousands of files) or complex multilingual codebases?

6. Can weak supervision (e.g., from GitHub issue-PR linking) train multimodal localization models without expensive annotation?

## Key Takeaway
Visual evidence in bug reports is largely ignored by current systems — this benchmark makes that gap measurable and pushable.
