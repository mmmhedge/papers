# Beyond Success Rate: Cost-Aware Evaluation of Offensive and Defensive Security Agents

**Authors:** Paul Kassianik, Blaine Nelson, Yaron Singer
**Published:** 2026-07-16 | **arXiv:** [2607.15263v1](http://arxiv.org/abs/2607.15263v1)

## Summary
This paper argues that security agent evaluations should measure cost-efficiency, not just success rate. The authors benchmark language-model agents on both offensive hacking tasks (CTF challenges) and defensive cybersecurity tasks (log analysis), showing that offensive and defensive agents scale differently with computational budget—and that real-world operational constraints make raw reasoning power less important than smart tool selection and data navigation.

## Theoretical Background
Language-model agents use language models as reasoning engines that can call external tools (code execution, database queries, APIs) to complete tasks. Prior security benchmarks like Cybench and BOTS v1 (Splunk's security challenge) measure success rates under unlimited or generous compute budgets, but operational security work (incident response, penetration testing) operates under strict cost constraints—each LLM inference call, tool invocation, or telemetry query has measurable expense. This work applies an economic lens to agent evaluation, drawing from concepts in computational efficiency and resource-constrained decision-making.

## Core Contributions
1. **Cost-stratified benchmarking framework**: Evaluates agents at fixed inference budgets (dollars/tokens spent) rather than just peak success, revealing distinct performance curves for different agent types.
2. **Red-team vs. blue-team scaling contrast**: Shows offensive CTF tasks benefit from additional reasoning compute and that scaled open-weight models can match proprietary systems economically. Defensive SOC (security operations center) investigations do *not* follow this pattern—success depends more on disciplined tool use and telemetry hygiene than raw compute.
3. **Operational cost decomposition**: Breaks performance down by inference spend (LLM reasoning) vs. tool spend (external queries/enrichment), showing where budget is actually consumed.
4. **SOC-native evaluation methodology**: Argues benchmarks should measure economic efficiency and fit for real operational workflows, not just task completion.

## Connections to Other Work
This work bridges [[agents|agent-based systems]] and practical security operations, critiquing the "capability maximization" paradigm common in LLM evaluation (similar to debates in [[alignment]] about whether frontier capability is the right metric). It extends prior work on [[reinforcement-learning]] agent efficiency and resource-constrained planning into a novel domain. Related to cost-aware inference in [[large-language-models]] (e.g., speculative decoding, early exit strategies) but applies those ideas to agent behavior and tool selection rather than just token generation.

## Practical Applications
1. **Security operations budgeting**: Organizations can now estimate which LLM-based SOC assistant fits their infrastructure cost and response-time constraints.
2. **Incident response automation**: Red teams and blue teams can optimize their agent deployments by understanding where compute dollars matter (offensive reasoning scales linearly; defensive success plateaus).
3. **Telemetry system design**: Results guide security teams on which data sources to prioritize and which enrichment queries are worth the cost.
4. **Penetration testing ROI**: Consultancies can forecast whether LLM-assisted exploitation tools justify the inference costs vs. human hours.
5. **Procurement and tool selection**: CISOs can use cost-aware benchmarks to compare agents (proprietary vs. open) at their actual operational budget level.

## Product & Startup Ideas
1. **CostWatch for Security Agents**: A monitoring dashboard that tracks inference spend vs. detection quality for deployed SOC agents in real time, with optimization recommendations.
2. **OpenSecAgent marketplace**: A platform selling fine-tuned, cost-optimized open-weight security agents (offensive and defensive) calibrated for different budget tiers.
3. **Incident Response ROI calculator**: B2B SaaS tool that lets enterprises simulate their SOC's agent setup (which LLM, which tools, which telemetry sources) and predict mean-time-to-resolution and cost-per-incident.
4. **Defensive Tool-Use Optimizer**: A system that learns to predict which telemetry queries actually help agents solve cases, pruning expensive dead-end enrichments before they're called.
5. **Red-Team as a Service (cost-audited)**: Managed penetration testing using cost-efficient LLM agents, with transparent billing that shows inference vs. tool spend—positioning cost-awareness as a compliance and budget-transparency feature.

## Technical Limitations
1. **Small benchmark scope**: Evaluation is limited to Cybench and Splunk BOTS v1; results may not generalize to other security domains (cloud intrusion, malware analysis, network defense).
2. **Fixed tool set**: Agents can only call pre-defined tools; real security work often requires novel tool composition or custom integrations that aren't modeled.
3. **No adaptive budgeting**: Agents don't learn to optimize their own spending mid-task; cost allocation is exogenously fixed.
4. **Telemetry realism**: BOTS v1 logs are historical/synthetic; real-world SOC telemetry has noisier patterns, missing data, and adversarial obfuscation.
5. **Human-in-the-loop gaps**: Security operations are rarely fully automated; the paper doesn't measure cost or friction of human oversight, escalation, or validation that would occur in production.
6. **Model generalization**: Results are tied to specific LLM snapshots; improved reasoning or tool-use in future models may shift the red-team/blue-team scaling curves.

## Open Questions
1. Can agents learn cost-aware policies (e.g., reinforce-learning agents that optimize for success per dollar) rather than having cost imposed post-hoc?
2. How do results change with multi-agent teams (coordinated red and blue teams) or adversarial dynamics (red agent evading blue agent detection)?
3. What is the role of human judgment in SOC workflows—does adding agent capability *replace* human costs or just shift them to validation/escalation?
4. Can telemetry design itself be optimized given a security agent's weaknesses (e.g., which logs should an SOC collect to maximize agent effectiveness per unit cost)?
5. How do these findings apply to other safety-critical domains (healthcare diagnosis, financial crime) where cost-accuracy trade-offs are also critical?

## Key Takeaway
Language-model security agents scale differently for hacking (reasoning helps) vs. incident response (tool discipline matters more)—cost-aware evaluation matters for real deployment.
