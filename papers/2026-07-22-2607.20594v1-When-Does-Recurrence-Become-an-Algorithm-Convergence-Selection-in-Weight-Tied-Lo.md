# When Does Recurrence Become an Algorithm? Convergence Selection in Weight-Tied Looped Transformers

**Authors:** Tong Zhang, Junhao Hu, Yun Peng, Tao Xie
**Published:** 2026-07-22 | **arXiv:** [2607.20594v1](http://arxiv.org/abs/2607.20594v1)

## Summary
This paper investigates when recurrent transformers (the same transformer block applied repeatedly) actually learn to solve algorithmic problems, rather than just memorizing patterns. The authors discover a "budget law" that predicts how fast the model solves a problem per loop based on training data size and loop count, and show that weight tying (reusing the same block) fundamentally shifts which algorithm the model learns—from parallel to serial computation. These findings are invisible to standard analysis methods, so the authors introduce new measurement techniques to detect what algorithms transformers actually implement.

## Theoretical Background
The paper builds on mechanistic interpretability — the study of understanding what computational procedures neural networks actually perform, rather than just their input-output behavior. It also draws on complexity theory, particularly NC1-completeness, which characterizes which problems can be solved by very shallow parallel circuits. The work uses weight-tied recurrent transformers, where a single transformer block (the basic building block of models like GPT) is applied T times in sequence. The key mathematical tool is convergence-time scaling analysis, which tracks how quickly model outputs stabilize during training. Prior work has shown that transformers can implement algorithms like sorting and prefix sums, but little was known about *which* algorithms recurrent architectures learn or *why*.

## Core Contributions
1. **The Budget Law**: SGD selects a computation speed (v positions solved per loop) that exactly matches the minimum demanded by the training contract (data size / loop count). This relationship holds across different training regimes with remarkable precision (exponent 0.98 ± 0.04).

2. **Architecture-driven algorithm selection**: Weight tying doesn't just compress a model—it *fundamentally changes which algorithm is learned*. The same problem that a deep untied transformer solves in parallel, a weight-tied version solves serially. Adding positional information doesn't override this architectural preference.

3. **Complexity bottlenecks are unexpected**: NC1-complete problems (theoretically hard to parallelize) are learned perfectly, while group order (the size of the symmetry operator) becomes the actual bottleneck. This contradicts complexity theory's predictions.

4. **Portability of mechanisms**: The learned algorithm (and its speed) transfers across different training budgets via warm-starting, but cannot be imposed through curriculum or input reordering—algorithms emerge from free training dynamics, not external structure.

5. **New measurement instrument**: Standard metrics (loss, accuracy) saturate and hide what's happening. The authors introduce convergence-time scaling τ(n,i) and validate it causally via damage cones, showing it predicts out-of-distribution generalization where traditional tail metrics fail.

## Practical Applications
1. **Transformer model verification**: Before deploying transformers for critical algorithmic tasks (program synthesis, reasoning), verify they learn the intended algorithm using convergence-time scaling, not just test accuracy.

2. **Curriculum learning design**: The failure of input-schedule curricula but success of warm-starting suggests that budget constraints (data/compute) should drive curriculum design, not problem difficulty ordering.

3. **Architecture selection for latency-critical systems**: If you have a fixed inference budget (T loops available at test time), the budget law predicts exactly what speed you can achieve, enabling better hardware-algorithm codesign.

4. **Debugging learning failures**: When a recurrent model fails on a hard problem (e.g., large group operators), damage cones can pinpoint which heads caused the failure, enabling targeted fixes.

5. **Training efficiency optimization**: The realization that SGD automatically matches training speed to the data budget suggests that over-allocating loops (training with T >> necessary) wastes compute—metrics like τ(n,i) help find the optimal T.

## Product & Startup Ideas
1. **AlgorithmDetective (B2B SaaS)**: A diagnostic tool for ML teams that automatically analyzes transformer checkpoints to reveal which algorithm was learned, what speed it operates at, and whether it will generalize. Uses convergence-time scaling as the core measurement. Targets companies deploying transformers for reasoning/synthesis tasks.

2. **ArchitectureAdvisor (Research platform)**: An interactive tool that, given a target algorithm and data budget, recommends whether to use deep-untied or shallow-tied transformers and predicts the exact speed/accuracy tradeoff via the budget law. Sells to academic labs and large ML shops.

3. **LoopScheduler (Inference optimization)**: A runtime system that measures τ(n,i) on small validation inputs at deployment time, then automatically adjusts T at inference to match the learned algorithm's speed, optimizing latency-accuracy tradeoff per input.

4. **CurriculumCraft (Training automation)**: A training framework that abandons input-schedule curricula in favor of budget-aware curricula (varying data size and loop count per phase), proven here to transfer learned algorithms better. Reduces tuning overhead.

5. **CircuitForensics (Interpretability consulting)**: A service that, for organizations deploying high-stakes transformers (finance, robotics), audits what algorithms were learned, whether they're robust, and provides remediation plans (warm-starting with different budgets, operator-first training, etc.).

## Technical Limitations
1. **Limited scope**: The paper only tests on group word problems (A5, S5, etc.), all relatively small and structured. Generalization to natural language, vision, or messy real-world tasks is unknown.

2. **Budget law scalability**: The linear relationship (v ~ n_train/T_train) was validated up to modest model sizes and T values. It's unclear whether the exponent remains 0.98 at GPT-scale or if there are regime changes.

3. **Damage cones are correlational**: While causality is tested via ablation, damage cones might not fully capture non-linear interactions between heads. The method still requires human interpretation.

4. **Warmstart assumption**: The portability result assumes access to a pre-trained checkpoint. Cold-start learning under different budgets is not addressed.

5. **Architecture coverage**: Only weight-tied recurrent transformers are tested. Untied models, RNNs, and hybrid architectures are not compared deeply. The degree to which findings apply broadly is unclear.

6. **Computational cost of measurement**: Convergence-time scaling requires tracking many intermediate checkpoints and computing damage cones; the overhead at scale is not discussed.

## Open Questions
1. Do the budget law and convergence-time scaling hold for real algorithmic tasks in NLP (e.g., numerical reasoning, sorting, path-finding in language)?

2. Can warm-starting techniques be used to *transfer* algorithms across tasks (e.g., train on A5, warm-start to solve S5 faster)?

3. What happens at the boundary of the budget law—does it break for very large T or very small n? Are there phase transitions?

4. Can an operator-first curriculum be automated? Given an unknown target algorithm, how do you design the curriculum?

5. How does the budget law interact with batch size, learning rate, and optimizer choice? Is it universal or SGD-specific?

6. Can convergence-time scaling predict which algorithms are *learnable* (i.e., identify a hard problem before wasting compute)?

## Key Takeaway
Weight-tied transformers learn algorithms at speeds predicted by data budget, not problem complexity—revealing what transformers actually compute requires new measurement tools beyond accuracy.
