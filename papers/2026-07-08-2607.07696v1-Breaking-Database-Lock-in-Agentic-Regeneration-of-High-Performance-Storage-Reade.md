# Breaking Database Lock-in: Agentic Regeneration of High Performance Storage Readers for Database Bypass

**Authors:** Victor Giannakouris, Immanuel Trummer
**Published:** 2026-07-08 | **arXiv:** [2607.07696v1](http://arxiv.org/abs/2607.07696v1)

## Summary
This paper presents Jailbreak, a system that uses large language models to automatically generate code for reading database storage files directly, bypassing traditional database drivers like JDBC and ODBC. By having LLMs analyze database source code and documentation, the system synthesizes custom columnar data readers that produce Apache Arrow buffers consumable by modern analytics engines, achieving up to 27x speedups on analytical workloads while maintaining correctness.

## Theoretical Background
Modern analytical databases face a critical bottleneck: data access flows through database drivers and query execution layers designed for transactional (OLTP) workloads, not bulk analytics (OLAP). Meanwhile, database storage formats — while complex — are fully documented in source code and schema specifications. This paper builds on two key insights: (1) that LLMs can ingest and reason about technical specifications at scale, enabling code synthesis for specialized tasks without human experts, and (2) that columnar in-memory formats like Apache Arrow have become standardized interchange layers across diverse analytics engines (DuckDB, Spark, cuDF). The work sits at the intersection of LLM-assisted code generation and database systems optimization.

## Core Contributions
1. **LLM-driven storage format decoding**: Demonstrates that LLMs can reliably synthesize database-specific storage readers by analyzing source code and documentation, eliminating the need for hand-written parsing logic.
2. **Database bypass methodology**: Shows a concrete path to circumvent database driver lock-in by reading storage files directly, materializng data as Arrow buffers for consumption by any downstream query engine.
3. **Empirical validation at scale**: Provides TPC-H benchmark validation on PostgreSQL and MySQL, demonstrating both correctness (against JDBC/ODBC baselines) and significant performance gains (up to 27x).
4. **Generalizability claim**: Argues the approach is format-agnostic — applicable to any database whose file format is documentable, not just PostgreSQL/MySQL.

## Connections to Other Work
This work relates to:
- **[[LLM code synthesis]]**: Extends prior work on using LLMs for program generation to a novel domain (database internals).
- **[[Database optimization]]**: Complements traditional query optimization and pushdown predicates by attacking the data access bottleneck itself.
- **[[Apache Arrow]]**: Leverages the columnar interchange standard as a unifying abstraction, making the approach compatible with modern analytics stacks.
- **[[Agent-based systems]]**: Frames the LLM as an agentic tool that autonomously understands and regenerates specialized components, aligning with broader trends in agentic AI.
- **Data lakehouse architectures**: Similar in spirit to direct file access strategies used by systems like Delta Lake and Iceberg, but here applied to legacy databases.

## Practical Applications
1. **Analytics on frozen replicas**: Organizations running read-only analytical queries on database replicas can now bypass the replica's query engine entirely, improving throughput for snapshot-based workloads.
2. **Legacy database analytics integration**: Data engineers can extract data from locked-down legacy systems (Oracle, SQL Server) into modern analytics stacks (Spark, DuckDB) without licensing expensive middleware or writing custom parsers.
3. **Cloud data warehouse migrations**: Accelerate bulk data ingestion during migration projects by synthesizing readers for proprietary on-prem database formats.
4. **Offline analytics pipelines**: ETL jobs processing historical snapshots can now materialize data far faster, reducing batch job duration and infrastructure cost.
5. **Multi-engine analytics**: Teams running heterogeneous query engines can standardize on Arrow buffers, generated uniformly via LLM-synthesized readers across different databases.

## Product & Startup Ideas
1. **Database Bridge SaaS**: A cloud service that ingests database credentials and metadata, automatically generates and hosts database-specific Arrow readers, and exposes them as HTTP endpoints. Customers subscribe per-database system.
2. **LLM-powered ETL tool**: A low-code ETL platform where users define source databases (any format), and the system automatically synthesizes readers, optimizes data movement, and materializes to cloud data warehouses. Differentiator: zero-configuration data format understanding.
3. **Legacy data unlocking service**: Consultancy/SaaS combo targeting enterprise customers with locked-in legacy databases (proprietary formats, old versions). Offer accelerated migration to Snowflake/BigQuery by synthesizing readers + orchestrating pipelines.
4. **Database format specification library**: Build a public repository of LLM-synthesized, tested, and validated readers for common database versions. Monetize via premium support, version management, and integration with major analytics platforms.
5. **Real-time replica acceleration**: Plug-in for Kafka/Flink that intercepts writes to database replicas, synthesizes Arrow readers on-the-fly, and allows streaming consumers to read from storage directly for ultra-low-latency analytics.

## Technical Limitations
1. **File format stability assumption**: Relies on storage formats being stable and fully documented. Undocumented internals, binary format changes between versions, or proprietary obfuscation will break the approach.
2. **LLM correctness and hallucination**: The generated code is only as good as the LLM's understanding of the storage spec. Complex formats with edge cases, compression schemes, or transaction semantics may be misunderstood, leading to silent data corruption or subtle bugs.
3. **Schema and type mapping**: Converting database types to Arrow types is non-trivial for exotic types (spatial, JSON, custom types). The paper doesn't deeply address type coercion and schema evolution.
4. **Concurrency and consistency**: Direct file reading bypasses the database's transaction isolation mechanisms. The approach is explicitly targeting read-only snapshots; concurrent writes invalidate the assumption.
5. **Scalability to massive files**: Materializing entire tables into in-memory columnar buffers hits memory limits. The paper doesn't discuss spilling, partitioning, or streaming strategies for tables larger than RAM.
6. **Validation scope**: TPC-H validation is narrow. Real-world databases have messy data, custom encodings, and schema migrations that TPC-H doesn't stress.

## Open Questions
1. **How does performance degrade with storage format complexity?** Can the approach handle advanced features like compression (zstd, LZ4), encryption, or custom type encodings?
2. **What is the failure rate of LLM-synthesized readers on out-of-distribution database versions or configurations?** How would versioning and fallback mechanisms work?
3. **Can the system handle schema evolution and data migration between versions?** What if the storage format changed mid-table?
4. **How does end-to-end latency compare for mixed workloads (online + batch)?** Can this approach be used for real-time serving, or is it strictly offline?
5. **What are the security implications?** If LLMs can understand storage formats, can adversaries too? Does this weaken database security assumptions?
6. **How generalizable is the approach to non-tabular formats?** Can it work for graph databases, document stores, or time-series databases?
7. **Can LLM-synthesized readers be formally verified or tested systematically?** What does a correctness guarantee look like?

## Key Takeaway
LLMs can automatically synthesize specialized database storage readers by analyzing source code and documentation, enabling direct columnar data access that bypasses database drivers and unlocks significant performance gains for analytical workloads.
