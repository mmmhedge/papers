# Supervised Fine-Tuning vs. In-Context Learning: An Equilibrium Analysis of LLM Personalization under Congestion

**Authors:** Fengzhuo Zhang, Zhuoran Yang, Dirk Bergemann
**Published:** 2026-07-15 | **arXiv:** [2607.14371v1](http://arxiv.org/abs/2607.14371v1)

## Summary
This paper analyzes when users should invest in expensive model fine-tuning versus cheaper in-context learning for personalizing large language models, accounting for shared computational resources. The authors develop an equilibrium framework showing that congestion from other users' choices fundamentally reshapes which personalization strategy is optimal, and prove that platforms benefit by offering both methods despite increased load.

## Theoretical Background
The paper bridges machine learning and economic theory. It builds on prior work on [[fine-tuning]] versus [[prompt engineering]] trade-offs in LLMs, extending classical resource-allocation problems to settings where multiple users compete for shared computational capacity. The framework draws on equilibrium analysis from game theory — specifically, how individual incentives aggregate into system-wide outcomes — combined with statistical learning theory. Key concepts include signal-to-noise ratios (how much signal your personalization data contains relative to noise), pretraining coverage (how well the base model already handles your task), and congestion externalities (how one user's resource consumption harms others).

## Core Contributions
1. **Equilibrium characterization**: Derives conditions under which ICL or SFT dominates based on data quality, task difficulty, and pretraining quality — and shows congestion can reverse these rankings.
2. **Non-monotonicity results**: Proves counterintuitive findings — improving pretraining reduces congestion, but broader pretraining coverage or harder tasks sometimes increase total resource demand.
3. **Platform design implications**: Shows that offering both personalization methods is always profit-maximizing for the platform, even when it increases total computational load. This directly explains observed platform behavior.
4. **Empirical validation**: Demonstrates predictions hold on GPT-2 linear regression tasks and validates against real platform offerings from 21 major AI companies (2021–2025).

## Connections to Other Work
Relates to [[RLHF]] and [[instruction-tuning]] literature on model customization, but with a novel focus on resource constraints and strategic behavior. Connects to congestion-pricing problems in [[multi-agent-systems]] and cloud computing. The equilibrium analysis parallels work on mechanism design for shared resources. Extends [[in-context learning]] research by explicitly modeling the cost-benefit trade-off against fine-tuning in a congested setting.

## Practical Applications
- **Platform resource allocation**: LLM-serving platforms (OpenAI, Anthropic, cloud providers) can use this framework to set pricing, compute quotas, and which personalization methods to expose to users.
- **User decision support**: Practitioners choosing between SFT and ICL can apply the theoretical conditions to their data quality and task characteristics.
- **Capacity planning**: Platforms can predict total compute demand under different personalization mixes to right-size infrastructure investments.
- **API pricing strategy**: Informs whether to bundle or separately price fine-tuning and in-context learning access.

## Product & Startup Ideas
1. **LLM personalization advisor**: A tool that analyzes a user's data, task, and current platform congestion to recommend whether to fine-tune or use in-context learning — potentially embedded in platform UIs.
2. **Smart fine-tuning broker**: A service that batches users' fine-tuning jobs across platforms to exploit price differences and congestion variation, automatically routing to least-congested providers.
3. **Congestion-aware prompt marketplace**: A platform where users share high-quality in-context prompts and pricing dynamically adjusts based on current server load, incentivizing sharing during off-peak hours.
4. **LLM resource futures market**: Allow users to pre-purchase personalization capacity (SFT or ICL slots) as futures contracts, hedging against future congestion — analogous to cloud capacity markets.
5. **Hybrid personalization optimizer**: Software that dynamically splits a user's personalization budget between SFT and ICL based on real-time platform load, task difficulty, and data quality signals.

## Technical Limitations
- **Linear regression focus**: Theory assumes relatively simple statistical settings; generalization to complex, non-linear LLM adaptation tasks is unclear.
- **Homogeneous users**: Model assumes users are statistically similar; real platforms have heterogeneous data quality, task difficulty, and privacy constraints.
- **Static equilibrium**: Framework assumes users reach stable strategies; does not model dynamic learning or strategic timing (e.g., users delaying fine-tuning to avoid congestion peaks).
- **Pretraining as exogenous**: Takes pretraining quality as given; doesn't model how platform investments in pretraining affect equilibrium.
- **No privacy trade-offs**: Ignores that fine-tuning may offer privacy benefits over in-context learning, which could reverse recommendations in sensitive domains.
- **Capacity constraints as soft**: Assumes congestion via shared latency/throughput; doesn't model hard capacity limits or admission control policies.

## Open Questions
- How do results change when users have heterogeneous data quality or task characteristics?
- Can the platform design optimal congestion pricing (compute costs that vary with load) to steer users toward socially efficient equilibria?
- What happens when users learn platform load patterns and strategically time fine-tuning requests?
- Does the theory extend to more complex tasks beyond linear regression, where fine-tuning gains are non-linear in data?
- How should platforms handle users with conflicting privacy/efficiency preferences (wanting fine-tuning for privacy but ICL for speed)?
- Can this framework be extended to multi-task learning, where a user personalizes for multiple tasks simultaneously?

## Key Takeaway
Shared computational constraints create surprising reversals in when to fine-tune versus use prompts—and platforms profit by offering both.
