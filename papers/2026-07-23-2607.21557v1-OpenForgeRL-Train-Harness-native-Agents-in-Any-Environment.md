# OpenForgeRL: Train Harness-native Agents in Any Environment

**Authors:** Xiao Yu, Baolin Peng, Ruize Xu, Hao Zou, Qianhui Wu et al.
**Published:** 2026-07-23 | **arXiv:** [2607.21557v1](http://arxiv.org/abs/2607.21557v1)

## Summary
OpenForgeRL is an open-source framework that enables end-to-end training of AI agents that use complex inference harnesses (like Claude Code or Codex) in real-world environments. The key innovation is decoupling training from the harness itself—using a lightweight proxy to record harness behavior as training data, while a Kubernetes orchestrator runs distributed rollouts—making it practical to apply reinforcement learning to sophisticated multi-turn agents that were previously hard to train with standard RL tools.

## Theoretical Background
Modern AI agents increasingly rely on "harnesses"—elaborate inference systems that coordinate multi-turn reasoning, tool use, and external system access. Traditional reinforcement learning stacks (like veRL) and supervised fine-tuning pipelines assume simple, stateless inference but struggle to express the stateful, multi-process nature of these harnesses. The paper builds on established RL training paradigms and the practical reality that deployed agents use these complex intermediate systems rather than raw language models.

## Core Contributions
1. **Lightweight proxy architecture**: Records harness model calls and interactions without modifying the harness itself, enabling standard RL codebases to train on top of it.
2. **Kubernetes-based distributed rollout orchestration**: Each training rollout runs in a remote container, enabling scale and isolation while supporting diverse environments (tool use, GUI agents, browser agents, computer-use agents).
3. **End-to-end training validation**: Demonstrates that agents trained with OpenForgeRL match or exceed larger open-source baselines on multiple benchmarks (ClawEval, QwenClawBench, OSWorld, Mind2Web, WebVoyager).
4. **Empirical analysis of harness effects**: Quantifies how choice of harness and RL affect agent behavior (reliability, self-verification, multi-step planning), revealing that error recovery remains a weakness.

## Connections to Other Work
This work sits at the intersection of [[reinforcement-learning]] for language models (related to [[RLHF]] and similar techniques), agent design and tool use (extending work on [[ReAct]] and code-based reasoning), and practical systems engineering for scaling agent training. It addresses a real gap: most RL infrastructure assumes simple forward passes, but deployed agents use stateful harnesses with side effects. The paper also connects to broader research on agent reliability and multi-step reasoning under uncertainty.

## Practical Applications
1. **Commercial AI agents**: Companies can now train custom agents on their own harnesses and environments (internal tools, APIs, databases) using only hundreds of examples.
2. **GUI/browser automation**: Enterprises automating customer service, data entry, or web scraping can train agents on real interfaces without expensive annotation.
3. **Computer-use agents**: Organizations deploying agents to control desktop environments (Windows, macOS, Linux) can fine-tune them on real deployment harnesses.
4. **Research at scale**: Academic labs studying agent behavior can efficiently run controlled experiments across different harness designs and RL algorithms without infrastructure overhead.

## Product & Startup Ideas
1. **Agent Customization-as-a-Service**: A managed platform where companies upload their proprietary harness and environment, run OpenForgeRL in the cloud, and receive fine-tuned agents for internal workflows (CRM, ERP, compliance tasks).
2. **Harness Marketplace & Benchmarking**: A repository of optimized harness implementations with OpenForgeRL training recipes, allowing teams to compare which harness performs best for their task and download pre-trained agents.
3. **Error Recovery Specialist Tools**: A product focused on the identified weakness—training agents specifically on error handling, rollback, and recovery scenarios using OpenForgeRL's infrastructure.
4. **Multi-Modal Agent Studio**: An IDE/IDE plugin where users design GUI-based automation workflows, train agents on them via OpenForgeRL, and deploy directly to their browser/desktop.
5. **Enterprise Agent Ops**: Continuous training and A/B testing platform for agents, tracking which harness/RL config performs best in production and auto-retraining on new tasks.

## Technical Limitations
1. **Harness brittleness**: The proxy records and trains on harness behavior, but harness changes or version updates may break training pipelines or invalidate checkpoints.
2. **Error recovery gap**: The paper acknowledges agents are weak at error recovery and multi-step planning under adversarial conditions—core for production reliability.
3. **Sample efficiency**: Requires "hundreds to a few thousand tasks" per domain, which may be expensive for niche or safety-critical environments.
4. **Kubernetes dependency**: The orchestrator assumes Kubernetes availability; deployment in edge or restricted environments may be difficult.
5. **Generalization unclear**: No analysis of transfer learning—can agents trained on one harness/environment transfer to another? How does domain shift affect performance?
6. **Training stability**: No discussion of RL training instability, reward hacking, or convergence diagnostics specific to harness-based agents.

## Open Questions
1. **Can agents trained on one harness transfer to another?** What is the generalization penalty if you train on Codex but deploy on Claude Code?
2. **What is the minimum task budget?** Below what data scale does training fail or become sample-inefficient?
3. **How does RL credit assignment work in harness environments?** Harnesses introduce latency and stochasticity—do existing RL algorithms handle this well?
4. **Can you train error recovery directly?** The paper identifies error recovery as weak—what RL reward structure or data curriculum would improve it?
5. **How do you handle harness side effects and state?** If an agent's action modifies persistent backend state, how do you safely parallelize rollouts?
6. **Scaling analysis**: What are the compute/latency bottlenecks as you scale to millions of tasks? Does the proxy become a bottleneck?

## Key Takeaway
OpenForgeRL decouples agent training from inference harnesses, enabling practical end-to-end RL on real-world agent systems at scale without modifying the harness itself.
