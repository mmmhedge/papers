# Do AI Agents Know When a Task Is Simple? Toward Complexity-Aware Reasoning and Execution

**Authors:** Junjie Yin, Xinyu Feng
**Published:** 2026-07-14 | **arXiv:** [2607.13034v1](http://arxiv.org/abs/2607.13034v1)

## Summary
This paper identifies a critical inefficiency in LLM-based agents: they often over-read context and re-examine already-seen information, turning simple tasks into expensive operations. The authors propose E3, a three-step framework (Estimate, Execute, Expand) that teaches agents to first judge task complexity, execute minimally, and only expand scope when needed—achieving 85% cost reduction and 91% fewer tokens while maintaining 100% success on code-editing benchmarks.

## Theoretical Background
LLM agents operate by reasoning over context and taking actions in external environments (code repositories, file systems, etc.). Prior work has focused on making agents more capable through better prompting, retrieval strategies, or reasoning methods (like chain-of-thought), but largely ignores **execution efficiency**—the idea that agents should budget their cognitive effort proportionally to task difficulty. This paper draws on concepts from cognitive science (cognitive load, working memory constraints) and applies them to agent behavior. The authors formalize **minimum-sufficient execution** (the smallest set of information and steps needed to reliably complete a task) and introduce the **Agent Cognitive Redundancy Ratio (ACRR)**—a metric measuring how much extra context an agent inspects beyond what was necessary.

## Core Contributions
1. **Formalization of task-aware execution**: Defines minimum-sufficient execution and ACRR as measurable concepts, moving beyond intuition to quantifiable metrics.
2. **E3 framework**: A practical three-phase algorithm where agents estimate task difficulty first, commit to a minimal execution path, and only expand scope (re-reading files, checking dependencies) when verification fails.
3. **MSE-Bench**: A deterministic benchmark of 121 code-edit tasks in a controlled simulator that isolates execution efficiency from other confounds.
4. **Empirical validation**: Demonstrates 85% cost reduction and 91% token savings on the benchmark while maintaining perfect success, plus real-world validation on gpt-4o editing actual open-source code.
5. **Positioned framework**: Introduces "engineering-grounded AI (EGAI)" as a research direction—agents whose effort is anchored to the actual engineering requirements of the task.

## Connections to Other Work
This work relates to several threads:
- **Agent reasoning frameworks** (e.g., ReAct, chain-of-thought): E3 complements these by adding a planning phase before reasoning.
- **Adaptive retrieval** and context-management approaches: E3 outperforms strong adaptive baselines, suggesting task-aware **estimation** is more efficient than reactive adaptation.
- **Token/cost optimization** in LLMs: Connects to broader work on inference efficiency, but from an agent-workflow perspective rather than model-level optimization.
- **Mechanistic interpretability**: Understanding why agents over-read relates to how LLMs allocate attention across context.
- **Alignment and agent oversight**: Agents that understand task scope are potentially more auditable and predictable.

## Practical Applications
1. **Code refactoring and migration tools**: Reduce costs of automated code reviews, bug fixes, and dependency updates by 85%+ while maintaining reliability.
2. **DevOps automation**: LLM agents managing infrastructure, deployments, or configuration changes can now estimate effort upfront and avoid unnecessary log/file inspection.
3. **Data engineering workflows**: Agents preparing data pipelines can pre-estimate scope, reducing token costs in budget-constrained environments.
4. **Customer support/ticketing**: LLM agents triaging and resolving support tickets can estimate complexity upfront, routing simple issues to minimal-resource paths.
5. **Testing and QA automation**: Agents that understand whether a test case is simple or complex can allocate inspection effort accordingly, speeding up CI/CD pipelines.

## Technical Limitations
1. **Benchmark limitations**: MSE-Bench is deterministic and runs in a controlled simulator; real-world code repositories have unpredictable edge cases, ambiguous requirements, and dynamic dependencies that may not trigger E3's expansion logic reliably.
2. **Estimation accuracy**: E3 relies on accurate initial complexity judgment; if the agent systematically misjudges task difficulty, it will commit to insufficient scope and fail. No principled method is provided for improving estimation accuracy.
3. **Verification requirement**: E3 only expands when verification fails—assumes there is a cheap, reliable verification oracle (e.g., pytest passing). Many real tasks lack clear test suites or have slow feedback loops.
4. **Generalization across domains**: Tested on code editing; unclear if task-aware execution transfers to other domains (customer support, data annotation, scientific reasoning) where "scope" is less concrete.
5. **Provider rate limits**: Real-world experiments hit rate limits; E3 doesn't solve the underlying cost-per-request issue, only reduces total requests. Very high-volume scenarios may still exceed budget.
6. **LLM capability dependency**: Success depends on the underlying model's ability to judge complexity accurately; weaker models may not benefit from E3's framework.

## Open Questions
1. **Can we learn complexity estimation from task outcomes?** Could E3 use reinforcement learning or inverse reinforcement learning to improve initial estimates based on past successes and failures?
2. **How does E3 scale to collaborative multi-agent systems?** If multiple agents must coordinate on a large codebase, does task-aware execution become harder or easier?
3. **What is the optimal expansion strategy?** E3 expands on verification failure, but should it also expand based on uncertainty thresholds or confidence scores from the initial estimate?
4. **How do different LLM architectures affect estimation quality?** Do larger models estimate task complexity more accurately? Could a small "complexity classifier" head be added to LLMs?
5. **Real-world generalization**: Can MSE-Bench results transfer to production agent systems (GitHub Copilot, Devin, etc.)? Are the gains real or specific to controlled conditions?
6. **Integration with [[RLHF]] and preference learning**: Can user feedback on over-reading be used to fine-tune complexity estimation?

## Key Takeaway
AI agents waste tokens re-reading old context; estimating task difficulty upfront cuts costs 85% without sacrificing accuracy.
