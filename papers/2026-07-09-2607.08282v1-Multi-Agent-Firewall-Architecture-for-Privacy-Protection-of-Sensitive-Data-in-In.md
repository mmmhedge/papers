# Multi-Agent Firewall Architecture for Privacy Protection of Sensitive Data in Interactions with Language Models

**Authors:** Hugo García Cuesta, Pablo Mateo Torrejón, Alfonso Sánchez-Macián
**Published:** 2026-07-09 | **arXiv:** [2607.08282v1](http://arxiv.org/abs/2607.08282v1)

## Summary
This paper presents a privacy-protecting firewall system that sits between users and language models, intercepting and analyzing all interactions to prevent sensitive data leakage. The system combines rule-based detection with AI-driven semantic analysis across browser and programmatic interfaces, achieving 94.93% accuracy at blocking risky data sharing—solving a critical gap where LLM integration into workflows creates uncontrolled data exposure risks.

## Theoretical Background
The work builds on two foundational areas. First, data loss prevention (DLP) systems — security tools that identify and block transmission of sensitive information like passwords, credit cards, or proprietary code. Second, multi-agent systems — architectures where multiple specialized computational agents (often powered by LLMs themselves) coordinate to solve complex tasks. The paper applies agent-based orchestration to the security domain: instead of a monolithic detector, the system uses distributed agents (deterministic rule checkers, semantic analyzers, code detectors) that work together in a pipeline. This leverages the flexibility of [[prompt-based reasoning]] for understanding context while maintaining the speed and reliability of traditional pattern matching.

## Core Contributions
1. **Hybrid detection pipeline**: Combines deterministic (fast, rule-based) and LLM-driven (semantic, context-aware) detection in a single framework, reducing false positives vs. purely deterministic DLP.
2. **Full traffic interception architecture**: Handles HTTP(S), WebSocket, and browser-native LLM API calls simultaneously via extension + proxy layers—not just one interface.
3. **Code-specific leakage detection**: Proprietary module for identifying proprietary source code patterns, addressing a major enterprise security concern distinct from generic PII detection.
4. **Pluggable agent design**: Framework for adding new security agents (e.g., prompt injection evasion) without restructuring the core system.
5. **Empirical validation**: F1 scores up to 94.93% on realistic mixed workloads, with analysis of computational cost vs. detection depth tradeoffs.

## Connections to Other Work
This relates to [[RLHF]] and alignment research in that both aim to control LLM behavior, but here the focus is external control (preventing bad outputs) rather than training-time alignment. It complements [[prompt injection]] defense literature by treating injection as one of many attack vectors in a unified framework. The multi-agent orchestration parallels work on [[AutoGPT]] and agent-based systems, but applies the paradigm to security rather than task automation. It differs from traditional DLP (which works post-hoc on stored/sent data) by intercepting at interaction time, before data enters the LLM's training pipeline or logs.

## Practical Applications
1. **Enterprise LLM adoption**: Companies integrating ChatGPT or Claude into internal tools (coding assistants, document analysis) need this to prevent engineers accidentally pasting source code or credentials.
2. **Healthcare workflows**: Clinical notes, patient IDs, and medical records flowing through medical LLMs for summarization or coding; the firewall prevents regulatory violations (HIPAA).
3. **Financial services**: Blocking trading desk analysts from sharing proprietary models, transaction data, or customer PII when using LLM-powered research tools.
4. **Legal/compliance teams**: Preventing disclosure of privileged attorney-client communications or work product when querying LLMs for document review.
5. **Government/defense contractors**: Stopping accidental exfiltration of classified or controlled unclassified information (CUI) through civilian LLM services.

## Product & Startup Ideas
1. **Enterprise LLM Security Suite**: SaaS dashboard that deploys the firewall across an organization's employee base, with centralized logging, incident response, and integration with existing DLP systems (Symantec, Forcepoint). Pricing per user or per API call blocked.
2. **Industry-specific firewall templates**: Pre-configured agents for healthcare (HIPAA compliance), finance (PCI-DSS), legal (attorney-client privilege), sold as turnkey deployments to regulated sectors.
3. **LLM API gateway**: Hosted proxy service that sits between enterprises and LLM providers (OpenAI, Anthropic, etc.), scanning all requests without client-side deployment—easier adoption, cloud-native.
4. **Prompt injection detection layer**: Spin out the injection evasion agent as a standalone API for app developers embedding LLMs; rapid detection of adversarial prompts in real time.
5. **Compliance reporting platform**: Combine the firewall with automated audit logging and policy reports (showing which data was blocked, why, by whom) for SOC 2 / ISO 27001 certification and incident investigation.

## Technical Limitations
1. **False positive rate**: Even at 94.93% F1, the system will block some legitimate queries (e.g., discussion of a customer's *public* code on GitHub). High latency configurations that call LLMs repeatedly for context may frustrate users.
2. **Semantic analysis cost**: LLM-driven agents are computationally expensive; the paper mentions cost/latency tradeoffs but doesn't deeply analyze when rule-only mode is sufficient, making real-world ROI unclear.
3. **Evasion robustness**: Adversarial users can obfuscate sensitive data (encoding, misspelling, fragmentation); the paper doesn't address sophisticated evasion.
4. **Training data bias**: Semantic agents trained on public data may miss domain-specific sensitive patterns (e.g., internal identifier formats, proprietary jargon).
5. **Interception gaps**: Browser extensions can be disabled; programmatic integrations may bypass the proxy entirely if not centrally enforced.
6. **No temporal analysis**: System treats each query independently; an attacker could leak data across multiple benign-looking queries.

## Open Questions
1. How does performance degrade on rare, domain-specific sensitive data (e.g., internal project codenames, proprietary algorithm names)?
2. What is the false positive rate in practice across diverse enterprise workflows, and how much productivity loss results?
3. Can adversarial prompts (e.g., "translate this to pig latin: [sensitive data]") evade the semantic detectors, and how would you detect them?
4. How should the system handle encrypted or obfuscated data? Can it detect when someone deliberately hides sensitive info to bypass the firewall?
5. What temporal or probabilistic models could detect coordinated multi-query leakage attacks?
6. How do results scale to custom fine-tuned LLMs or open-source models (Llama, Mistral) vs. proprietary APIs?
7. Can the agent architecture be extended to detect exfiltration via indirect channels (e.g., embedding secrets in generated code that the user then copies)?

## Key Takeaway
Multi-agent security pipelines can balance speed and semantic understanding to protect organizations from accidental data leakage when integrating LLMs into workflows, achieving enterprise-grade detection without fully reimplementing LLM training or access controls.
