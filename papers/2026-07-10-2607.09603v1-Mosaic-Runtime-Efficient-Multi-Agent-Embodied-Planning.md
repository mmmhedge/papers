# Mosaic: Runtime-Efficient Multi-Agent Embodied Planning

**Authors:** Kunjal Panchal, Saayan Mitra, Sunav Choudhary, Victor Bursztyn, Somdeb Sarkhel et al.
**Published:** 2026-07-10 | **arXiv:** [2607.09603v1](http://arxiv.org/abs/2607.09603v1)

## Summary
This paper tackles a critical bottleneck in LLM-based multi-agent planning: the high latency caused by failed actions and poor coordination. Mosaic proposes a framework combining lightweight semantic memory (storing objects in relative coordinates) with Integer Linear Programming for constraint-based coordination, achieving 27-32% faster execution and higher success rates on embodied AI tasks. This matters because it makes multi-agent LLM planning practical for real-time applications.

## Theoretical Background
Multi-agent embodied planning involves autonomous agents (robots or simulated entities) that must coordinate to complete tasks in partially observable environments. Prior work uses LLMs to generate high-level plans, but execution fails frequently due to two problems: (1) **state tracking under partial observability** — agents can only sense their local environment, so maintaining a global world model is fragile, and (2) **coordination inefficiency** — without explicit constraints, agents generate redundant or conflicting actions. The paper builds on constraint-based optimization techniques ([[Integer Linear Programming]]) and agent-centric representations from spatial reasoning in robotics. The key insight is that failures cascade (one failed action blocks subsequent ones), making efficiency upstream critical.

## Core Contributions
1. **Agent-centric semantic memory**: Stores tracked objects in relative (ego-centric) coordinates rather than global ones, enabling lightweight geometric reasoning and transformation between agent perspectives without maintaining a fragile global map.
2. **ILP-based action allocation**: At each planning step, formulates coordination as an integer linear program that assigns actions while enforcing physical feasibility (no collisions, resource constraints) and inter-agent dependencies, eliminating redundant/conflicting actions before execution.
3. **Empirical demonstration**: Shows that these two changes together reduce LLM calls by 30-33%, execution steps by 25-31%, and improve success rates by 4-10 points on standard benchmarks (AI2-THOR, search-and-rescue).

## Connections to Other Work
Mosaic bridges **embodied AI** (grounding language in physical worlds) with **multi-agent coordination**. It relates to:
- [[STRIPS/Classical Planning]] — using constraint satisfaction rather than pure sampling
- **Hierarchical planning** — LLMs handle high-level intent; ILP handles low-level feasibility
- **Decentralized POMDP (Dec-POMDP)** theory — partial observability is central, but Mosaic sidesteps it via relative coordinates
- **Visual grounding** in vision-language models — semantic memory is lightweight and vision-compatible
Unlike end-to-end RL approaches ([[Reinforcement Learning]]), Mosaic is model-agnostic and doesn't require reward engineering.

## Practical Applications
1. **Warehouse automation**: Multi-robot picking and packing systems where coordination and latency directly impact throughput.
2. **Search-and-rescue operations**: Drones or ground robots coordinating under communication constraints and partial visibility.
3. **Collaborative household robots**: Multiple robotic arms working together on assembly or tidying tasks in homes.
4. **Manufacturing floor coordination**: Autonomous agents (AMRs, arms) scheduling and deconflicting actions in real-time.
5. **Autonomous vehicle fleets**: Coordinating urban delivery or rideshare agents to avoid redundant routes and collisions.

## Product & Startup Ideas
1. **Multi-Robot Orchestration Middleware**: A cloud service that wraps LLM-based planning with Mosaic's coordination layer; robotics companies integrate via API, paying per planning call.
2. **Real-Time Warehouse Optimizer**: SaaS for existing warehouse automation platforms (Symbotic, Covariant) that plugs in Mosaic's constraint solver to reduce latency and coordination failures.
3. **Sim-to-Real Planning Engine**: A simulation platform (built on AI2-THOR or MuJoCo) where roboticists design and test multi-agent policies with guaranteed low-latency execution, then deploy to hardware.
4. **On-Device Multi-Agent Controller**: Embedded inference stack (Mosaic + small LLM) for edge robots (drones, construction equipment) to operate without cloud latency.
5. **Autonomous Fleet Safety Verification**: A tool that takes multi-agent plans and formally verifies feasibility via ILP before execution, targeting insurance/regulatory compliance in autonomous vehicles.

## Technical Limitations
1. **Scalability with agent count**: ILP can become intractable with many agents and high-dimensional action spaces; the paper doesn't report performance degradation curves.
2. **Partial observability boundaries**: Agent-centric coordinates only work if agents can reliably sense nearby objects; performance degrades if sensing is noisy or range-limited in ways not tested.
3. **LLM dependency**: Mosaic still calls an LLM for planning; if the LLM generates nonsensical goals, the framework cannot recover—no learning from execution history.
4. **Geometric assumptions**: Relative coordinates assume stable, trackable objects; dynamic or occluded environments may break assumptions.
5. **Benchmark scope**: AI2-THOR and search-and-rescue are relatively constrained; generalization to open-world or adversarial settings unclear.

## Open Questions
1. How does Mosaic scale to 10+ agents? Does ILP solve time grow prohibitively?
2. Can the framework incorporate learned value functions to guide ILP objective selection, rather than hard constraints alone?
3. How does performance degrade under realistic sensor noise and occlusion?
4. Does agent-centric memory generalize to 3D navigation tasks or non-tabletop manipulation?
5. Can Mosaic leverage execution failures as feedback to refine semantic memory or planning, or is it purely reactive?

## Key Takeaway
Efficient multi-agent embodied planning requires solving coordination constraints offline (via ILP) before LLM execution, not just better state tracking.
