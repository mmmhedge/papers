# Shared Selective Persistent Memory for Agentic LLM Systems

**Authors:** Sanjana Pedada, Aditya Dhavala, Neelraj Patil
**Published:** 2026-07-10 | **arXiv:** [2607.09493v1](http://arxiv.org/abs/2607.09493v1)

## Summary
This paper tackles a practical problem in LLM-based agent systems: when agents perform multi-turn tasks (like generating code or dashboards), they traditionally lose all learned context between sessions, forcing them to re-specify the same configurations, schemas, and constraints repeatedly. The authors propose "shared selective persistent memory"—a system that intelligently saves only reusable context (task specs, data schemas, tool configs, output rules) while discarding session-specific reasoning, and crucially, allows this memory to be shared across users with access controls. In a deployed platform, this achieves 96% task completion versus 79% without memory, while cutting token costs by 97x and task time by 14x through a "zero-token refresh" mechanism that decouples data from generated code.

## Theoretical Background
Agentic LLM systems (systems where language models call tools and decide what to do next) face a fundamental tension: **context window limits** mean you can't naively store entire conversation histories, but **starting from scratch** forces repeated specification of domain knowledge. Prior work has explored prompt engineering, in-context learning, and retrieval-augmented generation (RAG), but these don't address the specific problem of *persistent, reusable* configuration across sessions. The paper builds on ideas from **memory-augmented neural networks** and **knowledge graphs**, but applies them to a practical system design problem—what to remember, and how to share it safely. The zero-token refresh mechanism is inspired by **program synthesis** (separating the program structure from runtime data) and **lazy evaluation** (deferring data binding until execution).

## Core Contributions
1. **Selective memory architecture**: Formalization of four memory categories (task specs, schemas, tool configs, constraints) with explicit rules for what to retain and discard—moving beyond crude "save everything" or "save nothing" approaches.
2. **Shared, role-based memory**: Workspaces can be transferred across users with access control, enabling organizational memory reuse without redundant specification—a novel collaborative angle for agentic systems.
3. **Zero-token refresh mechanism**: Decoupling generated programs from runtime data such that recurring updates don't require LLM re-invocation; proves this works across 12/12 trials on public datasets.
4. **Empirical demonstration of harm from naive persistence**: Shows that full-history retention *actively degrades* task completion (71% vs. 96%), providing evidence that the selective approach is not just incremental but fundamentally better.
5. **Production validation**: Results from a deployed collaborative workspace platform, not just simulation, with enterprise scenarios and public dataset replication.

## Connections to Other Work
This relates to but differs from [[Retrieval-Augmented Generation (RAG)]] (which retrieves relevant context but doesn't persist learned task structure), **prompt engineering** (which optimizes single prompts but not multi-session learning), and **few-shot in-context learning** (which uses examples but doesn't build persistent organizational memory). It shares concerns with [[agent memory systems]] and [[knowledge graphs]] but focuses on the practical question of what to store. The collaborative reuse angle is adjacent to work on **multi-agent coordination** and **organizational learning**, though most agentic LLM work treats sessions as independent. The zero-token refresh idea echoes **program synthesis** techniques that separate code generation from data binding.

## Practical Applications
1. **Collaborative data analytics platforms**: Teams building dashboards and reports from multiple data sources (CSV, SQL, REST APIs) reuse schema definitions and output constraints across projects without re-specification.
2. **Enterprise software development**: Multi-user workspaces where developers and data engineers share tool configurations, library imports, and API schemas across codebases, reducing boilerplate and errors.
3. **BI/analytics governance**: Centralized memory of approved data transformations, metrics definitions, and security constraints propagates automatically to all generated reports, ensuring consistency.
4. **Consulting/professional services delivery**: Reusable templates for client engagements (CRM schema, standard report layouts, access patterns) scale delivery without re-engineering for each client.
5. **Internal tools platforms**: Companies use shared persistent memory to maintain evolving tool integrations (Slack, Jira, internal APIs) across all LLM-generated automation scripts.

## Product & Startup Ideas
1. **Collaborative Code Generation Workspace**: A SaaS platform for teams (data, analytics, engineering) where LLM agents generate, version, and maintain code artifacts; shared memory cuts onboarding time and template setup by 80%.
2. **Enterprise Data Lineage + Agent Memory**: Product that combines data catalogs (like Collibra, Alation) with agentic LLM workspace memory; agents auto-generate documentation and transforms while learning organizational schemas and governance rules.
3. **Vertical SaaS for Regulated Industries**: Healthcare/finance-specific SaaS where shared selective memory encodes compliance rules, data schemas, and audit constraints; agents generate compliant reports/analyses without repeated specification.
4. **Open-Source Agentic Framework**: Framework (like LangChain, AutoGen) with built-in selective memory and workspace-sharing primitives; monetize via hosted platform, enterprise plugins, or compliance/audit tooling.
5. **Internal Tools Automation for Enterprises**: Managed service where companies onboard their internal APIs, databases, and tool integrations once; shared memory enables their LLM agents (and non-technical users) to auto-generate automation scripts without re-engineering.

## Technical Limitations
1. **Memory design is somewhat manual**: The four-category taxonomy (task specs, schemas, configs, constraints) may not generalize to all domain types; no learned or adaptive categorization—requires human judgment about what to persist.
2. **Assumes stable, well-structured schemas**: Works well for tabular, REST, and SQL data but may struggle with unstructured text, evolving schemas, or highly heterogeneous data sources.
3. **Role-based access control is basic**: The paper doesn't detail fine-grained permissions (e.g., field-level access), masking strategies for sensitive data, or audit trails; real enterprise governance may require more sophistication.
4. **Zero-token refresh is program-data separation**: Only works if the generated program can be cleanly separated from the data it consumes; complex stateful or interactive workflows may not decompose this way.
5. **Evaluation is limited to three enterprise scenarios + public datasets**: No comparison to competing memory architectures (e.g., learned embedding-based memory, RAG with curation), and enterprise scenarios aren't fully disclosed (may not replicate to different domains/verticals).
6. **Scalability of shared memory not addressed**: No discussion of memory size, lookup latency, or how performance scales when many users/workspaces share memory; potential bottleneck at scale.

## Open Questions
1. **Can selective memory be learned rather than hand-engineered?** Could you train a classifier to predict which context categories to retain, or adaptively learn what to remember per task type?
2. **How does selective memory interact with LLM fine-tuning or specialized models?** If you fine-tune agents on the persistent memory, does it boost performance further, or does it create overfitting risk?
3. **What happens with adversarial or out-of-distribution tasks?** If a user asks an agent to do something novel that doesn't fit the learned schema, does selective memory help or hurt?
4. **Can you quantify the "staleness" of persisted memory?** How often do task specs, schemas, or tool configs become outdated, and how should the system detect and refresh stale memory?
5. **How does this compare to retrieval-augmented generation (RAG) with learned reranking?** Could you use RAG to dynamically fetch relevant context instead of pre-curating four categories?
6. **Does shared memory create security/privacy liability?** If a workspace contains sensitive customer data schemas or API credentials, what are the risks of sharing across users, and how should masking/anonymization work?

## Key Takeaway
Selectively persisting reusable configuration and schema across agentic LLM sessions—while sharing it across users with access control—beats both naive full-history retention and stateless approaches, and can reduce token costs 97x and task time 14x in practice.
