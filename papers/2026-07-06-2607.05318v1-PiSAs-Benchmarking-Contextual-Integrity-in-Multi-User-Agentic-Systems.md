# PiSAs: Benchmarking Contextual Integrity in Multi-User Agentic Systems

**Authors:** Shubham Gupta, Nazanin Mohammadi Sepahvand, Abhinav Kumar, Cem Subakan, Spandana Gella et al.
**Published:** 2026-07-06 | **arXiv:** [2607.05318v1](http://arxiv.org/abs/2607.05318v1)

## Summary
This paper introduces PiSAs, a benchmark for measuring privacy leaks in multi-user LLM-based agent systems where multiple users share infrastructure. The core problem: as LLM agents move from personal assistants to organizational tools, sensitive information can leak not just in final outputs but also across inter-agent messages and shared memory. Existing privacy benchmarks miss this because they focus on single-user scenarios; PiSAs fills that gap by testing whether agents correctly identify what information is appropriate for each task and which users should access it.

## Theoretical Background
The paper builds on **contextual integrity (CI)** — a privacy framework that says information sharing is appropriate when it follows the norms and expectations of a given social context. Traditional CI-based benchmarks evaluate whether systems leak private data *to external recipients* (e.g., outputs visible to the public). However, they don't address *internal privacy* — spillage between agents in a shared system.

The work also implicitly engages with [[alignment]] concerns: as agents become more autonomous and interconnected, ensuring they respect privacy boundaries becomes a control problem. The underlying challenge is that LLMs must learn context-dependent rules about who can see what, which requires both architectural design and reliable model judgment.

## Core Contributions
1. **Dual CI annotation scheme**: Each piece of information is labeled with (a) whether it's task-appropriate and (b) which users can legitimately access it. This enables fine-grained privacy assessment.
2. **System-agnostic benchmark**: Works across different agent topologies (hierarchical, peer-to-peer, etc.) and memory designs, making it broadly applicable.
3. **Cross-component measurement**: Explicitly measures spillage across outputs, inter-agent messages, and shared memory — interfaces that prior work ignored.
4. **Empirical finding**: Even state-of-the-art models fail to reliably filter inappropriate content or enforce access control, showing the problem is fundamentally a model judgment problem, not just system design.

## Connections to Other Work
This relates closely to:
- [[RLHF]] and broader [[alignment]] work, which addresses how to make models follow human-intended constraints — here, privacy constraints.
- [[Multi-agent systems]] literature on coordination and communication, but with a privacy lens.
- Privacy benchmarks like those in differential privacy and federated learning, though those focus on statistical guarantees rather than behavioral privacy.
- Information flow analysis in security, adapted for the LLM era.

The key novelty is applying CI to *intra-system* leakage in organizational multi-agent settings, not just inter-system boundaries.

## Practical Applications
1. **Enterprise AI assistants**: Companies deploying shared LLM-based agent systems (e.g., for HR, customer support, finance) need to ensure agents don't leak employee salary data across departments or customer info to unauthorized teams.
2. **Healthcare networks**: Multi-agent systems in hospitals where agents handle patient records must prevent cross-department information spillage (e.g., mental health records leaking to billing agents).
3. **Regulatory compliance**: Orgs handling GDPR, HIPAA, or SOX-regulated data need automated privacy checks during agent design and deployment.
4. **Secure collaboration platforms**: Tools enabling multiple teams to work with shared LLM infrastructure while enforcing role-based access policies.

## Product & Startup Ideas
1. **Privacy Compliance Layer for Agent Platforms**: A wrapper/middleware that sits between agent orchestration systems (like LangChain, AutoGen) and LLMs, automatically injecting CI-aware prompts and filtering inter-agent messages. Market: enterprises building internal AI applications.
2. **Agent Privacy Audit Service**: SaaS tool that benchmarks your multi-agent system against PiSAs, identifies spillage risk hotspots, and recommends architectural fixes. Model: recurring audit + consulting.
3. **Privacy-First Agent Framework**: Open-source or commercial agent framework baked with privacy-by-design principles (role-based memory, cryptographic access control, CI annotations). Target: startups and enterprises building agent systems.
4. **Data Governance for AI Teams**: Platform combining data lineage, access control, and automated CI-checking for organizations deploying shared LLM infrastructure. Sell to data governance/compliance teams.
5. **Privacy-Preserving Prompt Engineering**: Toolkit that helps teams design agent prompts that reliably enforce CI constraints without explicit rule-writing — train smaller models to learn CI patterns. Target: prompt engineering teams.

## Technical Limitations
1. **Model dependency**: Results show failures are due to LLM judgment errors, but the paper doesn't deeply investigate *why* models fail or provide solutions beyond system redesign. It diagnoses the problem but leaves treatment to future work.
2. **Annotation cost**: Dual CI annotations (task-appropriateness + user access rights) require domain expertise and scale poorly; no clear path to automatic labeling is proposed.
3. **Limited scope of topologies**: While "system-agnostic," the benchmark likely focuses on a discrete set of agent architectures. Highly bespoke or adversarial setups might not be well-covered.
4. **Static user roles**: Assumes access rights are predefined; doesn't handle dynamic, context-dependent permissions or threshold-based access (e.g., "X can see salary if they're an HR director AND reviewing the employee").
5. **No privacy-utility tradeoff analysis**: Strengthening CI compliance may degrade agent utility (agents become overly cautious). The paper doesn't quantify or explore this tension.

## Open Questions
1. **Can privacy be improved without sacrificing utility?** What architectural or training strategies (e.g., retrieval-augmented generation with access control, fine-tuning on privacy examples) actually reduce spillage while keeping agents effective?
2. **How do different CI definitions vary across cultures, industries, and jurisdictions?** The paper assumes a shared CI model; real deployments face heterogeneity.
3. **What's the minimum model capability to reliably enforce CI?** Do smaller, privacy-fine-tuned models work as well as giant models? Could agents delegate privacy checks to specialized models?
4. **How do adversarial agents exploit shared infrastructure?** If one agent is compromised, how quickly can it exfiltrate data from shared memory? Can cryptographic techniques help?
5. **Can CI be statically analyzed from system design alone?** If yes, could you catch spillage risks *before* runtime, reducing reliance on LLM judgment?

## Key Takeaway
LLM agents in multi-user settings leak sensitive information not just in outputs but across inter-agent messages and memory, and the core bottleneck is that even state-of-the-art models fail to make reliable privacy judgments — suggesting we need new architectural or training approaches beyond the ones tested here.
