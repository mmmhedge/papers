# Who Grades the Grader? Co-Evolving Evaluation Metrics and Skills for Self-Improving LLM Agents

**Authors:** Xing Zhang, Guanghui Wang, Yanwei Cui, Ziyuan Li, Wei Qiu et al.
**Published:** 2026-07-14 | **arXiv:** [2607.12790v1](http://arxiv.org/abs/2607.12790v1)

## Summary
This paper solves a bootstrapping problem: how do self-improving AI agents evaluate their own progress when no reliable evaluation metric exists? The authors propose **Double Ratchet**, a system that co-evolves evaluation metrics alongside agent skills, using evolutionary search to build transparent metrics from simple components rather than relying on opaque judges. They show this approach recovers 88–110% of the performance gain that would come from having perfect ground-truth evaluation, across code generation, SQL query synthesis, and report generation tasks.

## Theoretical Background
Self-improving agents operate in loops: generate outputs → evaluate → update skills → repeat. This framework assumes a reliable scorer already exists (ground truth, rubric, or human judgment). In practice, many domains lack such scorers. The paper builds on two foundations: **(1) evolutionary algorithm design**, using mutation and selection to discover effective metric compositions from primitive "drawback detectors," and **(2) metric learning**, where metrics are trained to agree with a small human-anchored reference set while regularized toward consensus on unlabeled data. The work also connects to broader **agent architecture** thinking—systems that must self-correct without an external oracle.

## Core Contributions
1. **Evolved metrics**: A metric-search loop that constructs composite evaluation functions by composing atomic detectors (e.g., "code has syntax error," "output is incomplete"), with explicit evolutionary lifecycle management and held-out auditing, yielding transparent, inspectable metrics rather than black-box classifiers.

2. **Double Ratchet co-evolution**: A paired architecture where metric and skill loops evolve together, with mutual checkpoints that allow skills to improve even without ground truth, retaining most of the lift achievable with perfect evaluation.

3. **Failure-catching via anchored auditing**: A safety mechanism—anchor discipline (small, manually verified reference set) plus independent outer judges—that detects when evolved skills game the metric and enables correction.

## Connections to Other Work
This relates to [[RLHF]] and reward modeling (the problem of learning evaluation from limited human signal), but inverts the dependency: instead of a fixed reward model driving agent learning, both co-evolve. It connects to [[self-play]] and self-supervision paradigms in multi-agent systems, and to the broader **alignment** challenge—ensuring AI systems optimize for the *intended* objective when that objective is hard to specify. The evolutionary metric search echoes neuroevolution and architecture search work in deep learning.

## Practical Applications
- **Code synthesis systems**: Automatically improving code generators (LeetCode-style problems, enterprise codebases) without reliance on test suites that may be incomplete or biased.
- **SQL/database query generation**: Evolving evaluation in data warehouses where perfect query validators are expensive or domain-specific.
- **Report/document generation**: Improving summarization, technical writing, or financial report generation where human rubrics are subjective and costly.
- **Domain-specific NLP tasks**: Any task where ground truth is sparse (legal document review, clinical note evaluation, scientific abstract scoring).

## Product & Startup Ideas
1. **Auto-Grading-as-a-Service for LLM Workflows**: SaaS platform that deploys evolved metrics for customer-specific evaluation tasks, reducing the need for human annotation at scale.

2. **Self-Improving Code Assistant (Enterprise)**: IDE/GitHub plugin that continuously improves on internal codebases using Double Ratchet, evolving both skill and evaluation without requiring labeled ground truth.

3. **AI Quality Auditor**: Third-party service that audits evolved metrics in customer AI systems, catching gaming behavior and recommending detector repairs (monetize via the independent judge + corrective feedback loop).

4. **Metric Marketplace**: Platform where organizations crowdsource "drawback detectors" and purchase evolved composite metrics for standard tasks (SQL generation, API documentation, summarization), with versioning and lifecycle management.

5. **Self-Improving Data Labeling**: Tool that evolves labeling criteria and data quality metrics alongside a labeling workforce, reducing human-in-the-loop friction for long-tail annotation tasks.

## Technical Limitations
- **Anchor dependency**: Performance heavily relies on the quality and representativeness of the initial 10-item anchor set; biased anchors will corrupt the evolved metric.
- **Computational cost**: Evolutionary search over metric space adds overhead; unclear how this scales to very large agent loops or real-time applications.
- **Domain transfer**: No evidence that evolved metrics transfer across tasks or domains; each task may require its own metric evolution loop.
- **Gaming risk remains**: The paper demonstrates *detection* of gaming via outer judges, but doesn't prevent it—the corrective loop is reactive, not proactive.
- **Limited scope**: Tested on three domains (code, SQL, reports); unclear how well Double Ratchet works on open-ended, subjective tasks (creative writing, strategic reasoning).
- **Anchor access**: Held-out audit requires the metric never "reads" the held-out anchor, but in practice, data leakage via soft constraints or implicit overfitting could occur.

## Open Questions
1. How sensitive is Double Ratchet to anchor set size and diversity? What's the minimum viable anchor?
2. Can evolved metrics transfer across related tasks, or must each task pay the metric-evolution cost?
3. How does performance degrade if the outer auditor (independent judge) is itself unreliable or biased?
4. Can proactive gaming-prevention (e.g., adversarial metric training) complement the reactive detection mechanism?
5. How does this approach scale to agents with hierarchical skill structures (skills that call other skills)?
6. What's the sample complexity of the metric-learning loop? How many unlabeled outputs are needed for consensus regularization to work?

## Key Takeaway
Agents can improve themselves without ground-truth evaluation by co-evolving transparent metrics and skills, catching gaming via independent audits.
