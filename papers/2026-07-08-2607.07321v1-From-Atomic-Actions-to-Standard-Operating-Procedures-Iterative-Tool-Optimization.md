# From Atomic Actions to Standard Operating Procedures: Iterative Tool Optimization for Self-Evolving LLM Agents

**Authors:** Haipeng Ding, Yuexiang Xie, Zhewei Wei, Yaliang Li, Bolin Ding
**Published:** 2026-07-08 | **arXiv:** [2607.07321v1](http://arxiv.org/abs/2607.07321v1)

## Summary
This paper addresses a fundamental inefficiency in LLM agents: they typically work with low-level atomic tools (like "read file" or "search query"), forcing them to repeatedly reconstruct complex multi-step workflows from scratch. The authors propose EvoSOP, a framework that lets agents automatically synthesize these atomic actions into reusable Standard Operating Procedures (SOPs)—higher-level tools that bundle multi-step logic. The result is agents that learn faster, succeed more often, and require fewer interaction steps to complete tasks.

## Theoretical Background
LLM agents operate by decomposing complex goals into reasoning steps and tool calls. Traditionally, they rely on tool libraries composed of fine-grained operations (file I/O, API calls, search). The paper builds on the observation that real workflows follow recurring patterns—similar to how humans develop procedures. The framework extends prior work on agent scaffolding and tool use by introducing a lifecycle for tool synthesis: extracting procedures from successful execution traces, merging duplicates, evaluating utility, and pruning low-value tools. This mirrors concepts from program synthesis and abstraction learning, but applied iteratively to the agent's own toolset.

## Core Contributions
1. **SOP abstraction**: A formal mechanism to synthesize multi-step atomic action sequences into reusable higher-order tools that agents can call directly.
2. **EvoSOP framework**: A four-phase lifecycle (construction, merging, evaluation, pruning) that systematically optimizes the toolset over time without manual curation.
3. **Empirical validation**: Demonstration that iterative tool optimization increases task success rates and reduces reasoning overhead compared to static baseline toolsets.
4. **Analysis of learned patterns**: Evidence that self-evolution produces reliable, interpretable tool-use strategies rather than brittle workarounds.

## Connections to Other Work
This work sits at the intersection of agent design, tool use, and program synthesis. It extends [[In-Context Learning]] and [[Chain-of-Thought]] reasoning by allowing agents to build their own abstractions rather than relying on fixed scaffolding. It relates to [[RLHF]]-style agent self-improvement, but through tool creation rather than preference tuning. The iterative optimization loop resembles curriculum learning and experience replay in [[reinforcement-learning]], adapted here for discrete tool composition. It also connects to research on hierarchical task decomposition and compositional generalization in agents.

## Practical Applications
1. **Autonomous data pipelines**: Agents that build and optimize their own ETL workflows by synthesizing frequently-used data transformation sequences.
2. **Retrieval-augmented generation at scale**: Agents that learn specialized retrieval and reasoning procedures tailored to domain-specific documents, reducing latency.
3. **Software engineering assistants**: Code agents that discover and reuse common refactoring, testing, and deployment patterns across projects.
4. **Customer service automation**: Bots that synthesize multi-turn dialogue patterns (e.g., "verify-customer → fetch-order → resolve-issue") into callable procedures, improving response consistency.
5. **Scientific research automation**: Agents that learn to execute standard experimental protocols (literature search → hypothesis formation → simulation setup) as reusable SOPs.

## Product & Startup Ideas
1. **Self-Optimizing Agent Platform**: A no-code/low-code platform where businesses upload domain workflows and an agent automatically synthesizes and optimizes its toolset, then exposes those learned procedures as APIs for other systems to consume.
2. **Agent Marketplace**: A service where teams can share vetted SOPs discovered by their agents (e.g., "efficient customer data lookup"), creating a curated library of reusable procedures across industries.
3. **Workflow Mining for Enterprises**: Software that integrates with existing RPA and BPM systems, analyzes execution logs, and auto-generates higher-level procedures that compress repetitive micro-interactions.
4. **AI-Powered Process Consulting**: A tool that profiles how agents solve tasks, extracts their learned procedures, and surfaces insights to business process teams (e.g., "you're doing these 5 steps in sequence 80% of the time—automate them").
5. **Agent Performance Analytics**: A dashboard that monitors tool-set evolution, flags emerging procedural patterns, and suggests consolidation or deletion—essentially "linting" for agent toolkits.

## Technical Limitations
1. **Trajectory dependency**: SOP extraction quality depends on having diverse, successful execution traces; agents with high failure rates may synthesize faulty procedures.
2. **Semantic drift**: Merging and pruning rely on heuristics; procedures that are semantically similar but syntactically different may not be recognized as duplicates, leading to redundancy.
3. **Generalization**: Procedures learned on one task distribution may not transfer to out-of-distribution goals; no clear mechanism for detecting when an SOP is misapplied.
4. **Computational cost**: The construction-merging-evaluation-pruning cycle adds overhead; the paper doesn't fully quantify the cost of optimization relative to savings.
5. **Tool interdependency**: No explicit handling of procedures that depend on dynamic state or context; assumes procedures are largely context-independent.
6. **Scalability to massive toolsets**: Unclear how the framework scales when agents have hundreds or thousands of candidate SOPs and complex dependency graphs.

## Open Questions
1. How does SOP quality degrade as the toolset grows? Is there a natural saturation point beyond which new procedures hurt rather than help?
2. Can agents use meta-learning to transfer learned procedures across different domains or task families?
3. What is the theoretical characterization of "good" SOPs? Can you predict which candidate procedures will accelerate learning before deploying them?
4. How sensitive is the framework to the merging and pruning thresholds? Are there principled ways to tune these hyperparameters?
5. Can agents represent and reason about conflicting or conditional procedures (e.g., "use SOP-A if X, else use SOP-B")?
6. How do learned SOPs interact with in-context learning? Do agents prefer to use tools or to learn equivalent patterns in their parameters?

## Key Takeaway
Agents that can synthesize and iteratively optimize their own tools—rather than relying on static atomic actions—achieve faster, more reliable task completion by building reusable abstractions of recurring workflows.
